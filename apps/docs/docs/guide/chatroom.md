# 群聊、模板与调度

群聊是 TeamAgentX 的核心工作区。一个群聊可以包含多个助手、群规则、工作目录、触发模式、定时任务和外部平台连接。你可以把它理解成一个带 AI 成员的项目群。

![群聊工作区](/images/user-guide/chat-room-clean.png)

## 群聊列表

在“消息”页左侧可以看到群聊列表。

| 操作 | 说明 |
| --- | --- |
| 创建群聊 | 新建一个项目群或主题群 |
| 导入群组模板 | 从别人导出的模板恢复群配置 |
| 刷新 | 重新加载群聊列表 |
| 复制群聊 | 复制群配置，适合基于旧群创建新项目群 |
| 导出群组模板 | 把当前群的结构打包，便于复用 |
| 删除群聊 | 删除群聊及其消息，此操作不可恢复 |

列表中置顶群聊会优先显示。快速对话群聊通常使用助手头像，普通群聊使用群头像。

## 创建群聊

点击“创建群聊”后填写基本信息。

![创建群组](/images/user-guide/group-create-dialog.png)

| 字段 | 说明 |
| --- | --- |
| 群组名称 | 建议用项目或任务名称，例如“官网改版”“后端排查” |
| 头像 | 用于区分群聊 |
| 群组描述 | 写清这个群的目标和用途 |
| 工作目录 | 群内助手共同使用的运行目录；桌面端能力更完整 |
| 选择助手 | 勾选要加入群里的助手 |

工作目录非常重要。正式项目建议指向项目根目录，不要指向用户主目录或过大的父目录。

### 创建群聊的完整步骤

1. 进入左侧“消息”页。
2. 点击“创建群聊”。
3. 填写群组名称，例如“官网文档编写”“移动端问题排查”。
4. 选择头像，方便以后在群列表里识别。
5. 填写群组描述，说明这个群的用途。
6. 选择工作目录。如果只是普通聊天可以先留空，如果要让助手处理项目文件，应选择项目根目录。
7. 勾选要加入的助手。
8. 点击创建。
9. 进入群后发送一条测试消息。

测试消息可以写：

```text
@总控 请介绍一下你在这个群里的职责，并告诉我下一步应该怎么使用这个群。
```

如果助手回复正常，说明群聊创建成功。如果没有回复，先检查助手是否真的在群里、模型是否启用、消息里是否写了正确的 `@助手名`。

## 群聊顶部工具栏

进入群聊后，顶部会显示群名、成员数量和常用操作。

| 按钮 | 用途 |
| --- | --- |
| 文件夹 | 打开群工作目录或相关目录入口 |
| 添加助手 | 给当前群添加更多助手 |
| 停止所有任务 | 有任务失控或不想继续时停止执行 |
| 任务看板 | 查看待处理、执行中、完成、失败/取消、待恢复任务 |
| 更多 | 打开群规则、截图聊天记录、定时任务、清空消息 |
| 群设置 | 修改名称、头像、描述、工作目录、默认助手和触发模式 |

不同模式下按钮可能略有差异。例如快速对话群聊通常不能再添加新助手。

## 群设置

点击群名或右上角设置按钮可以打开群设置。

![群设置](/images/user-guide/room-settings-panel.png)

群设置包含：

| 设置 | 说明 |
| --- | --- |
| 群组名称 | 修改群聊显示名称 |
| 头像 | 选择群头像 |
| 群组描述 | 写清群用途，会帮助用户判断是否进入正确群 |
| 工作目录 | 决定群内助手读写文件和运行命令的位置 |
| 默认接收助手 | 用户发送未 `@` 的消息时，自动交给该助手 |
| 助手触发模式 | 控制助手之间是否能通过 `@` 自动触发 |
| 群规则 | 注入给群内所有助手的统一行为要求 |
| 外部平台机器人 | 查看或管理当前群绑定的机器人 |
| 清空消息 | 删除当前群消息 |
| 删除对话 | 删除整个群聊 |

## 群规则

群规则会注入到群内所有助手上下文中，适合写共同约束。

推荐模板：

```text
- 所有回复使用中文。
- 先理解任务，再说明计划，最后总结结果。
- 保持改动最小，不做用户没有要求的功能。
- 修改代码必须说明涉及文件和验证方式。
- 遇到权限不足、命令失败、需求不清时，先说明阻塞原因。
- 不提交密钥，不输出或保存敏感凭证。
```

群规则适合写“所有助手都要遵守”的内容。单个助手的角色要求仍然应该写在助手提示词里。

### 群规则怎么写更稳

群规则不要写成空泛口号，最好写成可执行要求。

| 不够清楚 | 更推荐 |
| --- | --- |
| 好好回答 | 回复使用中文，先给结论，再给步骤 |
| 注意安全 | 不输出密钥，不提交敏感文件，不执行删除数据操作 |
| 帮我做项目 | 修改前先确认工作目录，改完说明文件和验证方式 |
| 多助手合作 | 总控先拆解任务，再安排具体助手执行 |

一个项目群可以这样写：

```text
这是“官网文档”项目群。
- 用户主要目标是维护产品文档。
- 文档内容要面向普通用户，避免技术黑话。
- 涉及代码或配置变更时，必须先说明影响范围。
- 没有用户明确要求时，不要提交代码、不发布、不删除文件。
- 所有最终输出使用中文 Markdown。
```

## 助手触发模式

群设置里的“助手触发模式”决定消息如何触发助手。

| 模式 | 行为 | 适合场景 |
| --- | --- | --- |
| 协调模式 | 未 `@` 的消息先给系统协调助手，由它判断派发给谁 | 新手推荐，适合多数项目群 |
| 自由协作 | 助手回复中的 `@其他助手` 会继续触发其他助手 | 熟悉多助手协作后使用，适合自动接力 |
| 手动模式 | 助手回复里的 `@` 只作为提及，不触发执行 | 想完全由用户控制，或避免助手互相触发 |

如果你发现助手之间互相叫来叫去，优先切到“手动模式”。如果你不知道该找哪个助手，优先使用“协调模式”。

### 三种触发模式举例

协调模式：

```text
用户：帮我把这个需求拆成开发任务。
系统协调助手：判断应该由总控或产品助手处理。
```

自由协作：

```text
用户：@总控 请安排一次完整开发流程。
总控：@工程师 请实现；@测试助手 请验证。
系统：工程师和测试助手会继续被触发。
```

手动模式：

```text
用户：@总控 请安排一次完整开发流程。
总控：建议你找 @工程师 和 @测试助手。
系统：不会自动继续触发，用户需要自己再发送 @工程师。
```

新手推荐先用协调模式或手动模式。自由协作很强，但如果提示词和群规则没写好，助手可能互相连续触发，导致消息过多。

## 默认接收助手

默认接收助手用于处理“没有明确 `@` 的用户消息”。

适合设置为默认的助手：

- 总控助手。
- 项目助理。
- 当前群最常用的助手。

不建议设置为默认的助手：

- 只做专项工作的助手，例如“删除数据助手”“发布助手”。
- 经常运行命令或改文件的助手。
- 已停用或不稳定的助手。

## 成员管理

点击成员数量或“添加助手”可以查看和管理群成员。

常见操作：

- 查看普通助手和系统助手。
- 添加一个或多个活跃助手。
- 移除不再参与当前群的助手。
- 进入助手详情，查看上下文、任务队列和执行历史。
- 设置“注入群历史”，决定该助手执行时是否看到群内历史消息。

“注入群历史”适合需要理解上下文的助手；如果只想让助手处理新消息，可以关闭。

### 注入群历史怎么选

| 助手类型 | 建议 |
| --- | --- |
| 总控助手 | 开启，方便理解整个讨论 |
| 文档助手 | 开启，方便整理上下文 |
| 工程师助手 | 通常开启，但任务很独立时可以关闭 |
| 高风险操作助手 | 谨慎开启，避免被旧消息误导 |
| 只处理当前消息的工具型助手 | 可以关闭 |

如果助手总是引用很早之前的内容，或者被旧结论带偏，可以关闭注入群历史，或清空上下文后再试。

## 发消息和触发助手

推荐显式 `@助手名`：

```text
@工程师 请检查当前项目如何启动，先只阅读文件，不要修改。
```

多助手协作可以这样写：

```text
@总控 请先拆解这个任务，再分别安排工程师实现、测试助手验证、文档助手整理说明。
```

如果群设置了默认接收助手，可以不写 `@`，但重要任务仍建议明确指定助手。

### 消息写法模板

普通任务：

```text
@助手名 请完成【目标】。范围是【文件/页面/群聊/项目】。限制是【不要做什么】。最后输出【交付物】。
```

代码任务：

```text
@工程师 请在当前工作目录检查登录页报错原因。先只分析，不要修改文件。请输出原因、涉及文件和建议修复方案。
```

文档任务：

```text
@文档助手 请把本群关于模型配置的讨论整理成小白教程。要求包含步骤、截图位置说明、常见错误和排查方法，输出 Markdown。
```

多助手任务：

```text
@总控 请拆解“新增用户手册”任务，先安排文档助手写初稿，再安排测试助手检查是否遗漏普通用户会遇到的问题。
```

如果任务比较危险，例如删除数据、发布内容、提交代码、修改数据库，建议明确写“先给计划，等我确认后再执行”。

## 任务看板和执行详情

任务看板用于观察群内执行状态。

| 状态 | 含义 |
| --- | --- |
| 等待执行 | 任务已进入队列，还没轮到执行 |
| 正在执行 | 助手正在处理 |
| 已完成 | 助手执行结束 |
| 失败/取消 | 执行报错或被用户停止 |
| 待恢复 | 被中断后可恢复的任务 |

在助手消息里也可以点击“查看执行详情”，查看工具调用、耗时、Token、任务清单和历史结果。

### 什么情况下要看任务看板

- 助手很久没有回复。
- 你同时触发了多个助手，想知道谁在排队。
- 想停止一个正在执行的任务。
- 想查看失败原因。
- 想确认定时任务有没有生成执行记录。

如果任务卡住，不要连续重复发送同一条消息。先看任务看板，确认它是等待、执行中、失败还是被取消。

## 定时任务

定时任务可以在指定时间自动向群里发送任务消息，并触发指定助手。

![定时任务面板](/images/user-guide/cron-tasks-panel.png)

点击“创建任务”后进入表单。

![创建定时任务](/images/user-guide/cron-task-create-dialog.png)

| 字段 | 说明 |
| --- | --- |
| 任务名称 | 用于识别任务，例如“每日项目总结” |
| 描述 | 说明任务目的，可选 |
| 调度类型 | Cron 表达式、固定间隔或一次性执行 |
| Cron 表达式 | 适合每天/每周/每月固定时间 |
| 间隔分钟数 | 适合每隔 N 分钟执行一次 |
| 执行时间 | 一次性任务的具体时间 |
| 执行内容 | 到点时发送给助手的任务文本 |
| 触发助手 | 选择一个、多个或全部助手 |
| 最大重试次数 | 执行失败后的重试上限 |
| 是否启用 | 创建后是否立即开始调度 |

### 调度类型怎么选

| 类型 | 例子 | 适合场景 |
| --- | --- | --- |
| Cron 表达式 | 每天 9 点、每周五 18 点 | 固定日程 |
| 固定间隔 | 每 30 分钟、每 6 小时 | 巡检、监控 |
| 一次性执行 | 明天 10 点执行一次 | 提醒、单次任务 |

注意：选择多个助手或“全部助手”时，系统会拆成多条消息逐个触发，避免一条消息同时 `@` 太多助手导致混乱。

### 定时任务例子

| 场景 | 推荐配置 |
| --- | --- |
| 每日站会总结 | 每天 9 点，触发总控，总结昨日进展和今日待办 |
| 每周发布说明 | 每周五 18 点，触发文档助手，整理本周变更 |
| 定期服务巡检 | 每 30 分钟触发运维助手，检查服务状态 |
| 一次性提醒 | 指定时间触发项目助理，提醒检查上线清单 |

不要在“执行内容”里手写 `@助手名`。应该使用“触发助手”字段选择目标助手，让系统自动处理提及。

### Cron 表达式快速理解

Cron 适合固定日程。常见写法如下：

| 想要的时间 | 示例表达式 | 说明 |
| --- | --- | --- |
| 每天 9 点 | `0 9 * * *` | 每天上午 9:00 |
| 每天 18 点 | `0 18 * * *` | 每天下午 6:00 |
| 每周一 9 点 | `0 9 * * 1` | 周一上午 9:00 |
| 每周五 18 点 | `0 18 * * 5` | 周五下午 6:00 |
| 每月 1 日 9 点 | `0 9 1 * *` | 每月第一天上午 9:00 |

如果不熟悉 Cron，优先使用界面提供的固定间隔或一次性执行。只有“每天/每周/每月固定时间”这类任务，再使用 Cron 表达式。

### 定时任务执行内容怎么写

执行内容就是到点后系统发给助手的任务。建议写得像正常聊天任务一样清楚。

```text
请整理过去 24 小时本群的重要进展，输出：
1. 已完成事项
2. 当前阻塞
3. 今天建议优先处理的 3 件事
不要编造没有在群聊中出现的信息。
```

如果触发多个助手，执行内容要适合所有被选中的助手。否则建议拆成多个定时任务，每个任务只触发一个明确角色。

## 导出群组模板

群组模板用于把一个群的配置打包复用。它适合团队共享“标准工作群”的结构。

导出通常包含：

- 群名称、描述、头像等可复用元信息。
- 群规则。
- 助手配置快照。
- 技能引用或技能内容。
- 定时任务定义。
- 触发模式等群设置。

导出通常不会包含：

- 用户私密聊天消息。
- 本地绝对工作目录中的敏感内容。
- API Key 等模型密钥。
- 系统内部不可迁移的运行状态。

操作方式：

1. 在群聊列表中打开目标群的菜单。
2. 点击“导出群组模板”。
3. 浏览器会下载模板包。
4. 把模板包发给需要复用的人，或保存为团队标准模板。

### 什么时候应该导出模板

适合导出模板的情况：

- 你已经调好一个标准项目群，想给其他项目复用。
- 团队希望统一助手组合、群规则和定时任务。
- 你要把配置迁移到另一台电脑。
- 你想备份当前群的结构，但不需要备份聊天记录。

不适合导出模板的情况：

- 群里包含不应共享的助手提示词。
- 群规则里写了内部密钥、客户名称或敏感流程。
- 你只是想保存聊天记录，这时应该使用截图或复制消息，而不是模板。

导出前建议检查：

- 群规则是否适合分享。
- 助手描述和提示词是否包含敏感信息。
- 定时任务是否会在导入后误触发。
- 模板接收方是否具备对应模型和技能。

## 导入群组模板

点击群聊列表顶部“导入群组模板”。

![导入群组模板](/images/user-guide/group-template-import.png)

一般流程：

1. 选择别人导出的模板包。
2. 系统预览模板内容和兼容性。
3. 如果模板里需要本地能力映射，例如模型、图片能力、技能目录，按提示选择本地可用配置。
4. 确认导入后，系统创建一个新的群聊副本。

导入后建议检查：

- 群规则是否符合你的团队。
- 助手是否都已成功创建。
- 模型供应商是否已映射到本机可用模型。
- 技能是否已正确安装。
- 定时任务是否应该立即启用。
- 工作目录是否需要改成当前项目目录。

### 导入模板后的第一件事

导入成功后，不建议马上执行复杂任务。先做一次小测试：

```text
@总控 请检查这个群的助手配置、群规则和定时任务是否适合当前项目。先只给检查报告，不要修改。
```

然后再逐项确认：

- 群名称是否需要改成当前项目名称。
- 工作目录是否指向当前项目。
- 默认接收助手是否正确。
- 触发模式是否符合你的使用习惯。
- 定时任务是否需要暂停或调整时间。
- 助手绑定的模型是否可用。

## 截图聊天记录

“截图聊天记录”会把当前聊天内容导出为图片，适合：

- 给团队同步执行结果。
- 保存一次重要讨论。
- 把助手输出贴到工单或文档。

导出前建议先隐藏或清理含密钥、Token、客户信息的消息，避免截图泄漏敏感内容。

## 清空消息和清空上下文

这两个操作不同：

| 操作 | 影响 |
| --- | --- |
| 清空消息 | 删除当前群可见聊天记录 |
| 清空上下文 | 清理某个助手在当前群里的执行上下文，让它忘掉之前对话 |

如果只是界面太乱，用清空消息。如果助手记住了错误信息或持续误解任务，用清空上下文。

## 复制群聊、导出模板和截图的区别

| 功能 | 主要用途 | 是否适合分享给别人 |
| --- | --- | --- |
| 复制群聊 | 在本机基于旧群快速创建新群 | 通常不用于跨机器分享 |
| 导出群组模板 | 复用群配置、助手组合、群规则和定时任务 | 适合，但要先检查敏感信息 |
| 截图聊天记录 | 保存或分享某次聊天内容 | 适合分享结果，不适合复用配置 |

如果你想“再建一个类似的群”，用复制群聊。如果你想“发给别人也能导入”，用导出群组模板。如果你想“把这次讨论结果发到工单”，用截图聊天记录。

## 常见问题

### 助手没有响应

检查是否明确 `@助手名`、助手是否在群里、模型是否启用、当前是否有任务排队，以及 Socket 是否连接正常。

### 助手用错目录

检查群设置里的“工作目录”。正式项目不要留空，也不要指向过大的父目录。

### 定时任务没有触发

检查任务是否启用、调度表达式是否正确、服务器是否运行、触发助手是否仍在群里。

### 模板导入后助手不可用

通常是本机缺少对应模型或技能。先到“模型”页配置模型，再到“技能”页检查技能导入状态。

### 群聊里出现敏感内容

不要导出截图或模板前直接分享。先删除相关消息，必要时更换已经泄漏的外部平台 Token 或 API Key。
