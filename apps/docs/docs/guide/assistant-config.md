# 助手配置

助手是 TeamAgentX 里真正工作的 AI 成员。一个助手通常包含名称、头像、模型供应商、提示词、技能、工作目录、语音设置和启用状态。

![助手管理](/images/user-guide/assistant-management.png)

## 助手页能做什么

| 功能 | 说明 |
| --- | --- |
| 搜索助手 | 按助手名称快速过滤 |
| 创建助手 | 新建一个可加入群聊或快速对话的助手 |
| 创建分类 | 给助手分组，例如“开发”“写作”“系统” |
| 拖拽排序 | 调整助手在分类中的顺序，也可拖到其他分类 |
| 编辑助手 | 修改名称、模型、提示词、描述、分类等 |
| 复制助手 | 以已有助手为模板创建新助手 |
| 启用/停用 | 临时让助手不可被调度，不删除配置 |
| 删除助手 | 删除普通助手；系统助手不可删除 |
| 安装 Skills | 给某个助手安装共享技能 |
| 快速对话 | 为单个助手创建一对一临时会话 |

系统分类和系统助手有保护限制，通常不能拖拽、删除或随意修改。这是为了保证群助手、技能管理、定时任务等内置能力稳定可用。

## 创建助手

点击“创建助手”会打开创建表单。

![创建助手](/images/user-guide/assistant-create-dialog.png)

常用字段如下：

| 字段 | 说明 |
| --- | --- |
| 名称 | 群聊里 `@名称` 使用的名字，应简短清晰 |
| 头像 | 用于区分助手身份 |
| 模型供应商 | 选择模型页中已配置且兼容的模型，或使用本地 Agent 配置 |
| 思考模式 | 部分工具支持不同推理强度或思考模式 |
| Codex/Claude 模型 | 使用本地 Codex/Claude 时可指定模型，留空则用本地默认 |
| 图片生成能力 | 开启后可调用受控图片工具，需要选择图片模型 |
| 描述 | 给用户看的用途说明 |
| 分类 | 让助手出现在指定分组里 |
| 系统提示词 | 定义助手角色、边界、工作方法和输出格式 |

### 创建助手的完整步骤

1. 进入左侧“助手”页。
2. 点击“创建助手”。
3. 填写名称，建议使用清晰角色名。
4. 选择头像，方便在群聊里辨认。
5. 选择模型供应商。没有可选项时，先去“模型”页配置文本模型。
6. 填写描述，让其他用户知道这个助手适合做什么。
7. 选择分类，例如开发、文档、运营。
8. 编写系统提示词。
9. 如果这个助手需要图片能力，开启图片生成并选择图片模型。
10. 点击保存。
11. 创建后可以先点“快速对话”测试，再加入群聊。

新手建议先创建一个“总控助手”和一个专项助手。总控负责拆任务，专项助手负责具体执行。

### 名称怎么取

推荐使用“角色型名称”，而不是抽象代号。

| 好名称 | 不推荐 |
| --- | --- |
| 工程师 | Agent1 |
| 测试助手 | 助手A |
| 文档助手 | 小助手 |
| 总控 | AI |

因为群聊触发依赖 `@助手名`，名称越清楚，用户越不容易误触发。

## 模型供应商怎么选

助手有两类常见运行方式：

| 方式 | 适合场景 |
| --- | --- |
| 绑定模型供应商 | 使用你在“模型”页配置的 API 模型，适合普通聊天、写作、轻量任务 |
| 使用本地 Agent 配置 | 使用本机 Claude Code、Codex 等本地工具配置，适合需要工具链、终端、代码上下文的任务 |

如果界面提示“暂无兼容供应商”，请先去“模型”页创建并启用模型，或确认本地 Agent 已登录。

### 普通模型和本地 Agent 怎么选

如果你不确定，先选“绑定模型供应商”。它更像普通聊天机器人，配置简单，适合大多数写作、问答、总结、轻量分析。

本地 Agent 更适合需要读写本地项目、运行命令、调用开发工具的场景。它通常依赖本机已经安装并登录的 Claude Code、Codex 或其他 Agent 工具。

| 场景 | 推荐 |
| --- | --- |
| 问答、写作、总结、翻译 | 绑定模型供应商 |
| 代码修改、终端命令、项目排查 | 本地 Agent 配置 |
| 新手第一次体验 | 绑定模型供应商 |
| 熟悉本地开发工具后 | 按助手职责选择 |

## 提示词怎么写

提示词决定助手的角色和工作习惯。普通用户可以从这 5 段开始写：

```text
你是一个前端工程师助手，负责 React/TypeScript 项目的功能开发和问题排查。

工作要求：
- 先理解需求，再说明计划。
- 保持改动最小，不做用户没要求的功能。
- 修改后说明改了哪些文件、如何验证。
- 如果需要用户决策，先提出问题，不要擅自扩大范围。
- 所有回复使用中文。
```

一个好提示词应该包含：

- 角色：这个助手是谁。
- 任务范围：它擅长做什么，不做什么。
- 工作流程：先分析、再执行、最后总结。
- 质量要求：是否要测试、是否要保守改动。
- 输出风格：中文、表格、简洁说明、代码块等。

不要在一个助手里塞太多互相冲突的角色。比如“既当产品经理又当工程师又当测试又当文案”通常会降低稳定性。更好的做法是拆成多个助手，在群里协作。

### 可直接复制的提示词模板

总控助手：

```text
你是一个团队总控助手，负责理解用户目标、拆解任务、判断应该由哪个助手处理，并在必要时提醒用户确认风险。

工作要求：
- 先复述你理解的目标。
- 把复杂任务拆成清晰步骤。
- 能直接处理的轻量问题可以直接回答。
- 需要专业助手时，说明建议找哪个助手以及原因。
- 不擅自执行高风险操作，例如删除数据、提交代码、发布内容。
- 回复使用中文，结构清晰。
```

文档助手：

```text
你是一个文档助手，负责把复杂信息整理成普通用户能看懂的说明。

工作要求：
- 面向没有技术背景的用户写作。
- 多使用步骤、示例和注意事项。
- 避免只写概念，要告诉用户在哪里点、怎么填、做完如何判断成功。
- 输出 Markdown。
- 遇到不确定的产品行为，明确标注需要确认，不要编造。
```

工程师助手：

```text
你是一个工程师助手，负责理解代码项目、排查问题和实现小范围修改。

工作要求：
- 修改前先确认工作目录和任务范围。
- 保持改动最小，不做未要求的重构。
- 修改后说明涉及文件、验证命令和风险。
- 遇到会破坏数据、删除文件、改数据库结构的操作，先提醒用户确认。
- 回复使用中文。
```

测试助手：

```text
你是一个测试与质量检查助手，负责发现风险、补充验证步骤和检查遗漏。

工作要求：
- 优先指出可能的 bug、边界条件和回归风险。
- 给出可执行的测试步骤。
- 如果没有发现问题，也要说明检查范围和剩余风险。
- 不夸大结论，不把未验证内容说成已验证。
- 回复使用中文。
```

## 编辑、复制、启用和删除

助手卡片菜单支持常用维护操作：

| 操作 | 说明 |
| --- | --- |
| 编辑 | 修改当前助手配置 |
| 复制 | 创建一个副本，适合基于旧助手微调新角色 |
| 停用 | 助手保留在列表中，但不会参与执行 |
| 启用 | 恢复可用状态 |
| 删除 | 删除助手配置；系统助手不允许删除 |

复制助手时，名称通常会自动追加“副本”。复制后建议立刻修改名称和提示词，避免群聊里两个助手职责太接近。

## 助手详情页

点击助手卡片会进入详情页。详情页通常包含：

| 页签 | 用途 |
| --- | --- |
| 配置 | 查看/编辑提示词、类型、模型、分类、工作目录和启用状态 |
| Skills | 查看已安装技能，安装新的技能 |
| 历史 | 查看快速对话会话 |
| 语音 | 配置助手消息的朗读方式 |

详情页顶部还能进行快速对话、安装 Skill、编辑、启用/停用等操作。

## 工作目录怎么理解

工作目录是助手处理本地文件和运行命令时所在的位置。

| 场景 | 推荐工作目录 |
| --- | --- |
| 让工程师助手修改某个项目 | 项目根目录 |
| 让文档助手整理某个资料文件夹 | 资料文件夹 |
| 临时聊天，不需要读写文件 | 可以留空或使用系统创建的会话目录 |
| 多个项目同时使用 | 每个群聊分别设置自己的项目目录 |

不要把工作目录随便设成用户主目录、磁盘根目录或包含大量无关文件的父目录。目录越准确，助手越不容易误读或误改文件。

## 快速对话

快速对话适合“只找一个助手临时聊一下”。

使用方式：

1. 进入“助手”页。
2. 找到目标助手。
3. 点击“快速对话”。
4. 可选择工作目录；留空时系统会创建独立会话目录。
5. 进入会话后直接发消息，不需要 `@助手名`。

适合快速对话的场景：

- 测试某个助手提示词是否好用。
- 临时让文档助手润色一段文字。
- 让工程师助手快速解释一段代码。

不适合快速对话的场景：

- 多助手协作。
- 需要长期积累项目上下文。
- 需要围绕固定项目群规则持续工作。

## 图片能力

如果希望助手生成图片，需要同时满足：

1. “模型”页已经添加并启用图片模型。
2. 助手配置中开启“图片生成能力”。
3. 助手配置中选择具体图片模型。
4. 用户任务明确要求生成图片。

开启后，助手会通过受控工具生成图片，而不是随意访问外部图片服务。

图片任务建议写清楚画面、风格、比例和用途。例如：

```text
请生成一张产品功能介绍配图，画面是三个 AI 助手围绕一个项目看板协作，风格简洁明亮，适合放在中文官网文档里。
```

## 语音设置

非系统助手可以在详情页配置语音。常见选项包括：

- 是否开启语音播报。
- 手动播放或自动播报。
- 使用浏览器本地语音或远程语音模型。
- 选择模型、音色或内置语音风格。
- 试听当前声音。

如果远程语音模型不可选，请先到“模型”页添加语音类型模型。

## 分类和排序

分类用于管理大量助手。

推荐分类方式：

| 分类 | 适合放什么 |
| --- | --- |
| 开发 | 工程师、测试、代码审查、Git 助手 |
| 文档 | 文档助手、翻译助手、发布说明助手 |
| 运营 | 内容、社群、用户反馈、竞品分析助手 |
| 系统 | 系统内置助手，通常不手动调整 |

拖拽助手可以调整排序或移动分类。系统分类不允许放入普通助手，系统助手也通常不能被普通拖拽操作改变。

## 助手创建后的检查清单

创建助手后建议检查：

- 名称是否方便在群聊里 `@`。
- 模型供应商是否可用。
- 提示词是否写清了角色、范围和输出方式。
- 是否需要安装技能。
- 是否需要开启图片或语音能力。
- 是否已经加入目标群聊。
- 如果要操作文件，工作目录是否正确。
- 如果暂时不用，是否应该停用而不是删除。

## 推荐助手组合

| 组合 | 用途 |
| --- | --- |
| 总控 + 工程师 + 测试 | 常规软件开发任务 |
| 产品 + 设计 + 文档 | 需求分析、界面方案、说明书输出 |
| 资料员 + 写手 + 审稿 | 长文、小说、内容生产 |
| 运维 + 监控 + 总结 | 定时巡检和报告生成 |

刚开始建议少建助手，等流程稳定后再拆分更细角色。
