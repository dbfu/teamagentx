## 二、竞品基准研究

### 竞品参考

| 产品 | 解决方式 | 可提取的模式 |
|---|---|---|
| Notion | 提供 ZIP 导入来承载多文件内容与目录结构；导入前提示限制和兼容性；批量文件优先使用单文件打包。来源：[Notion Help - Import data into Notion](https://www.notion.com/help/import-data-into-notion) | 多文件内容用单个压缩包承载；导入前预检；用户只处理一个文件 |
| 同类模板/迁移产品 [推演] | 导出通常是“单击下载模板文件”，导入通常是“选择文件后直接恢复或创建副本” | 导出零配置、导入最少决策、冲突时系统兜底 |

### 用户心智模型
用户会默认认为“导出”得到的是一个可以备份、分享、再导入的单文件，而不是一份仅供查看的 JSON。[推演]

### 行业惯例
- 多文件资源应由单个压缩包承载，而不是要求用户管理散落文件。
- 导入前可以做兼容性预检，但不应要求用户逐项做复杂映射。
- 目录结构、附件、脚本等资源应与模板主体一起迁移，否则“模板导入导出”会被用户理解为不完整能力。

### 已知反模式
- 只导出结构化 JSON，不包含技能目录、脚本、模板文件，导致导入后能力残缺。
- 导出时暴露过多开关，造成导出结果不一致，增加回溯和支持成本。
- 导入时要求用户手动映射模型或逐条确认资源，核心流程决策点过多。
- 沿用“模板包”命名，让用户误以为这是更复杂的打包概念，而不是群组模板本身。

### 认知复杂度上限
成熟产品在此类迁移场景下通常为 2-3 步、1-2 个决策点：
- 导出：点击导出 -> 下载单文件
- 导入：选择文件 -> 预检并确认导入

### 基准结论
- ✅ 本需求与行业惯例对齐：单文件导入导出、预检但少决策、资源完整迁移
- ⚠️ 当前实现偏离惯例：仅支持 JSON，且导出可选项过多
- ❌ 风险：如果继续沿用 JSON 主体，包含文件夹脚本与模板资源的技能将无法稳定 1:1 迁移

## 五、5W2H 全景分析

**What** — 做什么
> 优化现有群组模板导入导出能力：统一命名为“群组模板”，导出改为一键生成包含完整资源的 ZIP 文件，导入改为最少决策路径，并把模型映射收敛为默认模型策略。

**Why** — 为什么做
> 当前 JSON 导出无法自然承载技能目录、脚本、模板文件等资源，且导出/导入步骤与文案偏复杂，不符合用户对“模板迁移”和“备份恢复”的直觉。

**Who** — 谁来用
> 主要用户: 团队管理员 / 群组创建者 [推演]
> 次要用户: 负责分发模板的协作负责人 [推演]
> 受影响方: 被导入模板创建出的新群组成员、维护导入导出逻辑的开发者

**When** — 什么时候用
> 触发时机: 用户想备份群组、复制协作配置、跨环境迁移群组模板时
> 使用频率: 低频但关键操作
> 时间约束: 需要在一次操作中完成，不能依赖多次手工整理文件

**Where** — 在哪里用
> 使用环境: Web 管理界面 + 后端模板导入导出接口
> 入口位置: 会话列表/群组相关的模板导入与导出弹窗

**How** — 怎么做
> 核心操作路径: 导出时点击“导出群组模板”直接下载 ZIP；导入时选择 ZIP，系统预检兼容性后直接导入或自动重命名副本
> 技术实现方向: 基于现有模板快照能力，把 JSON 载荷升级为 ZIP 载体，并保留现有预检与导入计划能力

**How Much** — 做到什么程度
> 规模/量级: 单次模板需覆盖 1 个群组、其下全部助手/分类/技能/定时任务，以及技能关联文件资源
> 质量标准: 导出文件可被本系统稳定再导入；用户主流程不超过 3 步；默认路径无需额外配置
> 验收底线: 模板名称、格式、导出内容、导入行为与用户提出的 4 条规则保持一致

## 七、核心痛点 & 业务价值

### 场景 1: 一键导出完整群组模板

**现在的痛点**
> 当前导出产物是 JSON，用户无法确认其中是否完整携带技能脚本、模板目录和关联资源；如果技能包含文件夹内容，JSON 方案会让“可完整迁移”这件事不可信。

**实现后的业务价值**
> 导出后直接得到 1 个 ZIP 文件，降低分享、备份和迁移成本；同时避免因资源缺失导致导入后模板不可用。

**不实现的负面影响**
> 用户会继续把导出理解为“不完整备份”，模板复用信任度下降；一旦导入后技能缺文件，排查成本会显著上升。

**价值可信度**
- ⚠️ 基于合理推断（行业经验/类似案例） [推演]

### 场景 2: 选择模板文件后直接导入

**现在的痛点**
> 现有流程需要理解“模板包”概念、预期 JSON 文件格式、以及导入时潜在的名称与模型处理逻辑；对低频用户来说，认知负担偏高。

**实现后的业务价值**
> 用户只需选择 ZIP 文件并确认导入，系统自动处理默认模型和同名兜底，减少学习成本和操作错误。

**不实现的负面影响**
> 导入流程仍会停留在“技术导向”的表达方式，导致低频使用者不敢用或导入后结果不可预期。

**价值可信度**
- ⚠️ 基于合理推断（行业经验/类似案例） [推演]

## 八、标准用户故事 & 验收标准

### Story 1
**User Story**: 作为团队管理员，我想要一键导出群组模板，以便把当前群组完整备份并分发给其他团队复用。

**Acceptance Criteria**:
- [ ] AC1: When the user clicks the export action, the system shall start export immediately without requiring the user to configure whether skills or cron tasks are included.
- [ ] AC2: When export succeeds, the system shall download exactly one `.zip` file to the user device.
- [ ] AC3: When the exported group contains skills with nested files or folders, the exported template shall preserve those file paths inside the ZIP.
- [ ] AC4: When export UI and success feedback are shown, the system shall use the term `群组模板` and shall not show `群组模板包`.
- [ ] AC5: When the export file is generated, the payload shall exclude message history, local absolute paths, and secret credentials.

**Out of scope for this story**: 自定义选择只导出部分技能或部分定时任务

### Story 2
**User Story**: 作为团队管理员，我想要导入一个群组模板文件，以便在不理解底层资源结构的情况下快速创建新群组。

**Acceptance Criteria**:
- [ ] AC1: When the user selects a valid template ZIP file, the system shall parse the template and run compatibility preview before import.
- [ ] AC2: When no existing group has the same target name, the system shall import the template directly without asking the user to choose a duplicate strategy.
- [ ] AC3: When an imported template references text, image, or audio model bindings, the system shall resolve them by the current default model strategy instead of requiring manual mapping in the import flow.
- [ ] AC4: When the target group name conflicts with an existing group, the system shall generate an available new name and continue the import flow.
- [ ] AC5: When the template file format is invalid or missing required manifest or snapshot content, the system shall block import and show an explicit error message.

**Out of scope for this story**: 导入时提供逐模型、逐资源的高级映射编辑能力

### Decision Log

| Decision | Alternatives Considered | Why This Choice |
|---|---|---|
| 用户可见命名统一为 `群组模板` | 保留 `群组模板包` | 用户表达已经明确要求去掉“包”，且“模板”足以表达用途 |
| 导出格式切换为 ZIP 单文件 | 继续使用 JSON；JSON + base64 文件嵌入 | ZIP 更适合承载目录结构和多文件资源，且用户只处理一个文件 |
| 导出默认全量 | 保留“是否包含技能/定时任务”开关 | 与“一键导出”目标一致，减少导出结果差异 |
| 导入模型使用默认模型策略 | 导入时人工映射；严格按原模型 ID 恢复 | 与“最简操作”目标一致，降低低频使用门槛 |
| 同名时系统自动兜底命名 | 强制用户手选 duplicateAction | 无同名直接导入、有同名自动处理，更符合最少决策原则 |

### Must (必须做)

| M4 | 导入模型映射默认落系统默认模型策略 | 用户显式表达“导入模型映射也是直接使用默认模型” |

| M6 | 导入导出主流程遵循最简操作，导出一键完成，导入只保留必要预检与确认 | 用户显式表达“遵循用户最简操作” |

### Should (应该做)

| # | Requirement | Why Not Must |

| S3 | 明确区分“文件无效”“缺少必填结构”“资源缺失”的错误信息 | 提升恢复效率，但不影响主功能成立 |

| C2 | 导入完成后展示资源恢复详情页 | 锦上添花，不是当前最简操作目标的一部分 |

### Won't (本期不做)

| # | Requirement | Decision Reason |
|---|---|---|
| W1 | 导出时允许逐项勾选技能、定时任务或资源目录 | 与“一键导出”“默认全部导出”冲突 |

| W4 | 导出/导入消息历史 | 当前需求未提及，且涉及额外数据体积与隐私边界 |

## 十、功能详细需求定义

### 功能 1: 一键导出群组模板 ZIP

**功能描述**
> 系统将当前群组导出为单个 ZIP 格式的群组模板文件，默认包含该群组的全部技能、全部定时任务及其模板资源。

**输入**
| 字段 | 类型 | 必填 | 取值范围/格式 | 说明 |
|---|---|---|---|---|
| chatRoomId | string | 是 | 有效群组 ID | 被导出的群组 |
| exportAction | string | 是 | 固定为当前导出动作 | 用户点击导出触发 |

**处理逻辑**
1. 系统读取目标群组、助手、分类、技能使用关系和定时任务。
2. 系统收集技能资源文件，并保留原有相对路径结构。
3. 系统生成模板元数据与群组快照。
4. 系统将元数据、快照和技能资源打包为单个 ZIP 文件。
5. 系统向用户返回可下载文件，并在界面中使用“群组模板”命名。

**输出**
| 情况 | 输出内容 | 格式 |
|---|---|---|
| 成功 | 单个群组模板下载文件 | `.zip` |
| 失败 | 明确错误信息 | 文本错误 |

**边界情况**
| 场景 | 系统行为 |
|---|---|
| 目标群组不存在 | 阻止导出并提示群组不存在 |
| 群组没有定时任务 | 仍允许导出，ZIP 中对应内容为空集合 |
| 群组没有技能资源文件 | 仍允许导出，ZIP 中保留结构化快照 |
| 技能资源读取失败 | 阻止导出并返回可识别错误 |
| 并发请求 | 每次请求独立生成导出文件，不互相污染 |
| 权限不足 | 阻止导出并提示无权限 [推演] |

**与其他功能的依赖关系**
- 依赖: 群组快照导出能力、技能收集能力
- 被依赖: 模板再导入能力

### 功能 2: 群组模板 ZIP 预检与导入

**功能描述**
> 系统接收单个群组模板 ZIP 文件，完成结构校验和兼容性预检后，按默认模型策略创建新的群组副本。

**输入**
| 字段 | 类型 | 必填 | 取值范围/格式 | 说明 |
|---|---|---|---|---|
| templateFile | file | 是 | `.zip` | 用户选择的群组模板文件 |
| desiredGroupName | string | 否 | 非空文本 | 用户可修改目标群组名 |

**处理逻辑**
1. 系统解析 ZIP，并校验模板主清单与快照是否存在。
2. 系统读取模板摘要、技能资源和兼容性描述。
3. 系统执行兼容性预检，并按默认模型策略解析模型绑定。
4. 若目标名称无冲突，则直接导入；若有冲突，则系统生成可用名称继续导入。
5. 系统创建群组、分类、助手、技能资源和定时任务副本，并返回最终群组信息。

**输出**
| 情况 | 输出内容 | 格式 |
|---|---|---|
| 成功 | 新群组 ID、最终群组名称、导入结果摘要 | JSON/结构化响应 |
| 失败 | 阻断原因 | 文本错误 |

**边界情况**
| 场景 | 系统行为 |
|---|---|
| 文件为空 | 阻止导入并提示需选择模板文件 |
| 文件不是 ZIP | 阻止导入并提示文件格式无效 |
| ZIP 缺少 manifest 或 snapshot | 阻止导入并提示模板结构无效 |
| ZIP 内存在无法识别的附加文件 | 忽略非必要文件，不影响主流程 [推演] |
| 存在名称冲突 | 自动生成可用群组名并继续 |
| 默认模型不可用 | 阻止导入并提示当前环境缺少可用默认模型 [推演] |
| 定时任务为空 | 允许导入，创建无定时任务的新群组 |

**与其他功能的依赖关系**
- 依赖: ZIP 解析能力、模板预检能力、群组导入计划能力
- 被依赖: 导入成功后的群组访问与协作流程

### 功能 3: 最简操作导入导出体验

**功能描述**
> 系统应把导入导出交互压缩到最少步骤，避免用户在默认路径上做非必要选择。

**输入**
| 字段 | 类型 | 必填 | 取值范围/格式 | 说明 |
|---|---|---|---|---|
| actionType | string | 是 | `export` / `import` | 用户动作类型 |
| templateFile | file | 否 | `.zip` | 导入时使用 |

**处理逻辑**
1. 导出弹窗不展示“是否包含技能/定时任务”的配置开关。
2. 导出操作触发后直接执行，不要求用户选择导出范围。
3. 导入流程只保留文件选择、必要预检和确认导入。
4. 系统自动处理默认模型和同名冲突，不要求用户手工指定策略。

**输出**
| 情况 | 输出内容 | 格式 |
|---|---|---|
| 成功 | 成功提示与关键结果 | 界面反馈 |
| 失败 | 可理解错误提示 | 界面反馈 |

**边界情况**
| 场景 | 系统行为 |
|---|---|
| 用户取消文件选择 | 停留在当前状态，不报错 |
| 预检发现阻断问题 | 显示阻断原因并禁止继续导入 |
| 导入成功但自动改名 | 成功提示中明确展示最终群组名 |
| 网络超时 | 提示操作失败并允许用户重新发起 |

**与其他功能的依赖关系**
- 依赖: 导出 ZIP、导入预检与导入执行
- 被依赖: 最终用户体验与验收判断

## 十一、非功能需求

### Performance (性能)
- Response time: 常规群组模板导出/预检/导入在可接受等待时间内完成；首版以现有系统默认标准为准 [推演]
- Throughput: 低频管理操作，无需为高并发批量导入导出优化；遵循系统默认标准
- Data volume: 需支持包含多文件技能资源的单个模板 ZIP；当资源体积较大时，系统需给出明确失败或限制提示 [推演]
- Degradation behavior: 当模板过大或资源读取失败时，应中止操作并返回明确错误，不得生成半有效模板

### Security (安全)
- Authentication: 仅已登录且有权访问目标群组的用户可导出模板 [推演]
- Authorization: 导入导出遵循现有群组访问控制
- Data sensitivity: 模板不得包含密钥、凭证、绝对路径、消息历史等敏感数据
- Attack surface: 需防范恶意 ZIP、异常路径、超大文件带来的内存与路径安全风险 [推演]
- Audit trail: 导入导出失败原因应可在服务端日志中定位 [推演]

### Compatibility (兼容性)
- Browser/platform targets: 兼容当前 Web 端支持范围，遵循系统默认标准
- API versioning: 本次按新功能处理，可直接定义 ZIP 群组模板的接口契约
- Data format compatibility: 新标准格式为 ZIP；历史 JSON 模板不在支持范围内
- Third-party integration constraints: 上传处理需遵循 `@fastify/multipart` 的文件体积与内存使用约束；ZIP 处理需遵循 `fflate` 的浏览器/Node 运行边界

### Usability (易用性)
- Learnability: 新用户无需培训即可完成“导出一个模板”“导入一个模板”的默认流程
- Error recovery: 文件无效、结构缺失、模型不可用时，需给出明确可理解错误
- Accessibility: 无特殊要求，遵循系统默认标准
- Mobile/responsive: 无特殊要求，遵循系统默认标准

### Maintainability (可维护性)
- Code coverage expectation: 需补充导出 ZIP、导入 ZIP、旧格式异常、自动重命名、默认模型映射等测试 [推演]
- Documentation requirements: 用户文案与接口说明需同步更新“群组模板”命名
- Observability: 应记录导入导出开始、失败、成功及关键原因
- Deployment: 无特殊要求，遵循系统默认标准

### Scalability & Extensibility (可扩展性)
- Growth assumptions: 后续可能出现更多模板资源类型，但本期只需保证 ZIP 能容纳现有技能目录结构 [推演]
- Extension points: 模板主清单可保留版本字段，以支持未来格式升级
- Configuration vs code: 导出范围不对用户开放配置；实现层可保留内部扩展能力，但不暴露到当前交互
- Multi-tenancy: 无特殊要求，遵循系统默认标准
