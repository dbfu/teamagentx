## 二、文档目标

本文件不直接规定某个具体实现，而是定义 TeamAgentX 在聊天场景中的语音播放交互原则，作为后续代码审查、回归测试和行为验收的统一基线。

适用范围：
- Web 聊天中的手动播放
- Web 聊天中的自动播报
- 播放中的暂停/停止
- 房间切换时的播放处理
- 预热缓存、远端音频准备完成与真实播放之间的关系

不在本文范围：
- STT 录音交互
- 移动端原生播放器细节
- 音频格式、音质、供应商能力差异

## 三、核心交互原则

### P1. 显式用户意图优先

只有明确的用户操作，或明确的上下文切换，才能打断当前播放。

允许打断当前播放的事件：
- 用户点击当前消息的暂停/停止
- 用户切换房间
- 用户关闭页面或组件卸载

默认不允许打断当前播放的事件：
- 新消息到达
- 自动播报队列新增任务
- 预热缓存完成
- 远端音频刚好返回

### P2. 非用户事件不得抢占当前播放

系统内部事件可以准备下一条待播内容，但不能抢占已经开始的播放。

约束：
- 自动播报不能中断已经在播的消息
- 手动播放请求若不是用户明确要求“改播另一条”，也不能隐式停掉当前播放
- 缓存命中与否只能影响“是否需要等待”，不能影响“是否允许中断”

### P3. 暂停/停止是最终意图

一旦用户对当前消息执行停止，该次播放会话必须进入不可自动恢复状态。

约束：
- 即使远端音频稍后才准备完成，也不能自动开始播放
- 即使预热 Promise 稍后 resolve，也不能自动续播
- 若要再次播放，必须由用户重新显式触发

### P4. 单房间单活跃播放

同一房间同一时刻只能存在一个真实的语音播放会话。

说明：
- “真实播放会话”指实际驱动浏览器音频输出的那条链路
- 预热请求、缓存读取、音频准备，不算真实播放

### P5. 自动播报必须串行且不丢消息

自动播报可以排队，但不能并播，也不能因为当前正在播放就把后续消息静默丢掉。

约束：
- 当前播放未结束时，新到达的可播消息应进入待播队列
- 当前播放结束后，应继续消费队列中的下一条
- 队列中的消息除非被用户明确停止或上下文失效，否则不能无故消失

### P6. 房间边界强于播放连续性

切换房间时，当前房间的语音播放和待播队列必须一起失效。

约束：
- 切房间后不得继续播放旧房间语音
- 切房间后旧房间待播消息不得漏播到新房间
- 已播放记录可保留，但活跃播放状态必须清空

### P7. UI 状态必须与真实播放状态一致

前端展示出来的“正在播放”必须能准确映射到真实音频输出状态。

约束：
- `playingVoiceMessageId` 只能指向当前真实正在播的消息
- 播放结束、失败、取消后，UI 状态必须及时清理
- 用户点击停止后，UI 不得继续显示播放中

### P8. 用户主动取消不应视为错误

用户主动停止是一条正常交互路径，不应冒充异常。

约束：
- 用户停止后不弹“播放失败”
- 内部可以使用 `cancelled` 语义区分主动取消和真实失败
- 真实失败才允许提示错误

### P9. 预热与真实播放必须职责分离

预热只能负责“提前准备音频”，不能直接变成“自动开始播放”。

约束：
- 预热完成本身不是播放信号
- 缓存命中只能优化延迟，不能改写交互控制权
- 真实播放必须仍由手动播放或自动播报会话来驱动

## 四、用户心智模型

用户对聊天语音播放的默认预期应是：

1. 当前正在播，就应该完整播完，除非我亲自停掉它。
2. 我点了暂停，就应该彻底安静下来，不会过一会儿自己又响。
3. 新消息可以排队，但不能抢我当前正在听的内容。
4. 我切到别的房间，上一间房的声音就应该结束。

如果实现违反这些直觉，用户会把问题理解为“播放器不可控”，而不是“偶发延迟”。

## 五、验收检查清单

审查语音播放代码时，至少逐项回答以下问题：

- [ ] 当前播放是否只会被用户操作或房间切换中断？
- [ ] 新消息到达时，是否只入队/预热，而不会抢占当前播放？
- [ ] 用户停止后，等待中的远端音频或缓存完成是否会被正确作废？
- [ ] 同一房间是否保证任一时刻只有一个真实音频输出？
- [ ] 当前播放期间到达的新自动播报消息，是否会进入队列而不是被丢弃？
- [ ] 切换房间后，旧房间的播放和待播任务是否全部失效？
- [ ] 播放中 UI 标记是否与真实音频状态一致？
- [ ] 用户主动停止时，是否不会被错误提示打扰？
- [ ] 预热逻辑是否仅做准备，不直接触发自动播放？

## 八、结论

本文件的核心立场只有一句话：

**语音播放控制权必须掌握在用户和明确的会话边界手里，而不能落在缓存完成、请求返回或其他内部异步事件手里。**
