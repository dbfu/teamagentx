# 助手触发系统规格说明

[English](11-agent-trigger-system_EN.md) | 中文

本文档说明 TeamAgentX 中助手的当前触发规则、群聊模式及提示词设计建议，供创建助手和群规则时参考。

---

## 1. 核心概念

### 1.1 触发的本质

每条消息进入群聊后，系统会先扫描消息内容中的可触发 `@助手名称`，再结合群聊模式决定谁进入任务队列执行。

- 有效 `@助手名称` 会直接触发对应助手
- 快速对话群在用户消息未 @ 其他助手时，会直接触发 `quickChatAgentId`
- 普通群在协调模式下，未被直接 `@` 的消息会先交给内置的 `群调度助手`
- 普通群在非协调模式下，用户未 @ 时才可能落到默认助手

### 1.2 消息类型

| 类型 | 说明 |
|------|------|
| 人类消息 | 用户在聊天框发送的消息 |
| 助手消息 | 助手生成后由系统写入群聊的消息，`isHuman = false` |

---

## 2. 群聊触发模式（agentTriggerMode）

群聊有三种触发模式，决定了**用户消息和助手消息如何流转**。

### 2.1 coordinator 模式（默认）

```
agentTriggerMode = 'coordinator'
```

这是当前新建普通群聊的默认模式。

- 用户消息未 `@` 助手时，不直接触发默认助手，而是先触发内置 `群调度助手`
- 普通助手发出的消息也会先交给 `群调度助手` 判断是否需要继续派发
- 只有用户消息中的有效 `@助手`，或 `群调度助手` 发出的有效 `@助手`，才会直接触发目标助手
- 普通助手自己写出的 `@助手` 默认只作为展示文字，不构成直接派发

适合“让系统自动判断下一步该找谁”的多人协作群。

### 2.2 auto 模式（自由协作）

```
agentTriggerMode = 'auto'
```

助手消息中的 `@xxx` 或 ` @xxx` 可以直接触发其他助手。多助手辩论、固定接力、显式工作流编排等场景适合此模式。

### 2.3 manual 模式

```
agentTriggerMode = 'manual'
```

助手消息中的可触发 `@xxx` **不触发**任何助手，仅作为展示文字。只有用户发消息才能触发助手。适合单助手对话、不需要助手互相协作的场景。

---

## 3. 完整触发规则表

以下规则按优先级从高到低执行：

| 消息来源 | 消息内容 | 群类型 / 模式 | 触发结果 |
|---------|---------|-------------|---------|
| 用户 | 无 @ | 快速对话群 | 触发快速对话助手（跳过历史注入） |
| 用户 | 有一个可触发 `@xxx` | 快速对话群 | 触发被 @ 的助手（快速对话助手不触发） |
| 用户 | 无 @ | 普通群，`coordinator` 模式 | 触发内置 `群调度助手` |
| 用户 | 无 @ | 普通群，`auto/manual` 模式，有默认助手 | 触发默认助手 |
| 用户 | 无 @ | 普通群，`auto/manual` 模式，无默认助手 | 无响应 |
| 用户 | 有一个可触发 `@xxx` | 普通群任意模式 | 触发被 @ 的助手 |
| 助手（非群调度助手） | 任意内容 | `coordinator` 模式 | 先触发 `群调度助手` 判断是否继续派发 |
| 助手（群调度助手） | 有一个可触发 `@xxx` | `coordinator` 模式 | 触发被 @ 的助手 |
| 助手 | 有一个可触发 `@xxx` | `auto` 模式 | 触发被 @ 的助手（自己除外） |
| 助手 | 无 @ | `auto` 模式 | **无响应**，流程到此中断 |
| 助手 | 任意内容 | `manual` 模式 | **无响应**，消息仅展示 |

---

## 4. 助手触发助手

### 4.1 消息内容中直接 @（推荐）

助手在回复内容中写 `@目标助手名称`，或在正文中写空格加 `@目标助手名称`，系统会先解析有效 mention，再按群模式决定是否触发。

```
# 主持人提示词示例
宣布辩题后，写 "@正方一辩 请开篇立论"
```

**@ 解析规则**：
- `@` 位于行首，或 `@` 前一个字符是空格时触发
- 标点后直接 `@`、文字后直接 `@` 不触发
- 助手名后跟空格、行尾或标点符号（`!?.,:;！？。，；：`）
- 同一条消息中同一助手被多次 @，只触发一次
- 单条消息最多触发一个助手；出现多个不同助手 @ 时，前端会拦截用户消息，后端只处理第一个有效 @
- 长名称优先匹配（避免"一辩"误匹配"正方一辩"）

**模式差异**：
- `auto`：普通助手写出的有效 `@` 会直接触发目标助手
- `manual`：普通助手写出的有效 `@` 不会触发任何助手
- `coordinator`：只有内置 `群调度助手` 写出的有效 `@` 会直接触发目标助手；普通助手写出的 `@` 只会作为输入交给 `群调度助手` 判断

---

## 5. 默认助手

每个普通群可设置一个默认助手（`defaultAgentId`）。在 `auto` 或 `manual` 模式下，用户发消息**且不包含任何 @** 时，默认助手自动触发。

- 默认助手必须是该群的成员
- 快速对话群不走此逻辑，改用 `quickChatAgentId`
- `coordinator` 模式下不使用默认助手；切换到该模式时会清空 `defaultAgentId`

---

## 6. 历史上下文注入

助手被触发时，系统通常会将群历史注入上下文（除非 `injectGroupHistory = false`）：

| 配置项 | 默认值 | 说明 |
|-------|--------|------|
| `AGENT_HISTORY_THRESHOLD` | 20 | 超过此条数触发摘要压缩 |
| `AGENT_MEMORY_RECENT_MESSAGES` | 10 | 保留最近 N 条原文注入 |
| `AGENT_MEMORY_COMPACT_MESSAGES` | 40 | 触发摘要的消息数上限 |

补充说明：
- 快速对话群触发 `quickChatAgentId` 时，当前实现会跳过历史注入
- 普通群中的业务助手默认仍按 `injectGroupHistory` 配置注入历史

**重要影响**：助手能看到触发它的消息是谁发的、内容是什么。这会让 LLM 有时自然地以 `@发送者` 开头回应；这是模型行为，不是系统自动加的前缀。

## 7. 定时任务触发

群聊可配置 Cron 定时任务。到时间后，系统会生成一条“看起来像用户消息”的系统消息发入群聊，再按普通用户消息规则处理。

如果任务选择了多个 `agentIds` 或 `["*"]`，调度器会循环发送多条消息：每条消息只自动添加一个 `@助手名`，逐个触发目标助手。这样定时任务可以批量通知多个助手，同时保持“单条消息最多触发一个助手”的规则。

如果定时任务没有显式指定 `agentIds`，则后续行为与用户普通消息一致：
- 快速对话群：触发快速对话助手
- `coordinator` 模式普通群：触发 `群调度助手`
- `auto/manual` 模式普通群：触发默认助手（若已设置）

支持两种类型：
- `cron`：标准 cron 表达式（`0 9 * * *` 每天9点）
- `once`：指定某一时间点触发一次

## 8. 提示词设计建议

### 8.1 流程接力模式（辩论/比赛/多角色）

**群聊配置要求**：优先使用 `agentTriggerMode = auto`。如果要让系统自动判断下一步找谁，改用 `coordinator` 并把业务助手职责写清楚。

**核心设计原则**：
> 每个助手的回复中必须包含 `@下一个助手`，否则流程在该节点中断。

**可靠触发的两种策略（二选一）**：

**策略 A：显式规则（最可靠）**

在每个中间节点助手的提示词末尾加入：
```
发言结束后，在消息最末尾单独一行写 @主控助手名，通知其继续安排流程。
发言开头不要 @主控助手，称呼直接用文字，@ 只在末尾出现一次。
```

**策略 B：依赖 LLM 自然行为（不推荐单独使用）**

历史上下文中有触发方的名字，LLM 通常会自然以 `@触发方` 开头回应。但此行为不稳定（LLM 可能直接开始正文），不应作为唯一触发机制。

**推荐**：使用策略 A（显式规则），确保 100% 可用。

### 8.2 主控助手（主持人/裁判/导演）设计

```markdown
## 流程控制规则
- 每次只安排当前阶段的一个助手，不要提前安排下一轮
- 安排时直接写 "@助手名称 请[动作]"，不要附加多余内容
- 等待被点名的助手发言后（系统会再次触发你），再安排下一位
```

### 8.3 执行助手（辩手/选手/演员）设计

```markdown
## 流程通知规则
- 每次发言结束后，在消息最末尾单独一行写 @主控助手名
- 发言开头不要 @主控助手，称呼直接用文字
- @ 在整条消息中只出现一次（末尾）
```

### 8.4 群规则建议

群规定义的是**角色行为约束**，对 AI 触发机制无效。不要在群规中写"等待点名才能发言"来控制触发——触发由系统决定，不由助手自控。

| 适合写进群规 | 不适合写进群规 |
|------------|--------------|
| 发言内容的格式要求 | 触发条件（系统负责） |
| 角色定位说明 | 等待机制（系统负责） |
| 比赛/活动的业务规则 | 谁先谁后的顺序（提示词负责） |
| 语言风格要求 | @ 谁的规则（提示词负责） |

## 9. 常见问题

### Q：流程中断，助手不再响应？

先确认当前群模式：

- `auto`：检查链路中某个节点的助手回复是否包含可触发 `@下一个助手`。没有的话流程会中断
- `manual`：助手之间本来就不会自动接力
- `coordinator`：检查 `群调度助手` 是否把消息继续派发给了某个业务助手；普通助手自己写 `@` 不保证直接生效

### Q：助手被触发两次？

同一条消息中多次出现同一可触发 `@助手名`，系统只触发一次。但若开头和结尾各写一次 @ 导致**展示重复**（视觉上两次 @），需在提示词中明确"可触发 @ 只出现一次，开头用文字称呼"。

### Q：manual 模式下助手能否自动接力？

不能。manual 模式完全阻断助手间触发，只能靠用户手动用可触发 `@助手名` 触发每个助手。

### Q：coordinator 模式下为什么普通助手写了 `@另一个助手` 也没直接触发？

这是设计行为。协调模式下，普通助手消息会先交给内置 `群调度助手` 判断，只有 `群调度助手` 发出的有效 `@助手` 才会直接派发。

### Q：多个助手被同时 @ 会怎样？

不再并发触发。用户消息会被前端拦截；助手或外部平台消息如果包含多个不同助手 @，后端只处理第一个有效 @。
