# 助手触发系统规格说明

本文档说明 TeamAgentX 中助手的所有触发规则、群聊模式及提示词设计建议，供创建助手和群规则时参考。

---

## 1. 核心概念

### 1.1 触发的本质

每条消息进入群聊后，系统会扫描消息内容中的可触发 `@助手名称`，并将被点名的助手加入任务队列执行。没有被 @ 的助手不会收到任何消息，也不会有任何输出。

### 1.2 消息类型

| 类型 | 说明 |
|------|------|
| 人类消息 | 用户在聊天框发送的消息 |
| 助手消息 | 助手生成后由系统写入群聊的消息，`isHuman = false` |

---

## 2. 群聊触发模式（agentTriggerMode）

群聊有三种触发模式，决定了消息如何路由到助手。未配置时默认为 `coordinator`。

### 2.1 auto 模式

```
agentTriggerMode = 'auto'
```

助手消息中的 `@xxx` 或 ` @xxx` 可以触发其他助手。多助手协作、辩论、接力等场景必须使用此模式。

### 2.2 manual 模式

```
agentTriggerMode = 'manual'
```

助手消息中的可触发 `@xxx` **不触发**任何助手，仅作为展示文字。只有用户发消息才能触发助手。适合单助手对话、不需要助手互相协作的场景。

### 2.3 coordinator 模式（系统默认）

```
agentTriggerMode = 'coordinator'
```

所有消息先经过**内置群调度助手**路由，由它决定派发给哪个业务助手，再由被派发的助手执行任务。

路由规则：
- 用户消息**无 @**：交给调度助手判断，由调度助手派发
- 用户消息**有可触发 @**：跳过调度助手，直接触发被 @ 的助手
- 普通助手消息（无论有无 @）：交给调度助手判断是否需要接力
- 调度助手本身输出的 @ 消息：直接触发目标助手

适合希望系统自动协调多助手、无需用户手动指定助手的场景。局限：调度本身消耗一次 LLM 调用，存在一定延迟。

---

## 3. 完整触发规则表

以下规则按优先级从高到低执行：

| 消息来源 | 消息内容 | 群类型 / 模式 | 触发结果 |
|---------|---------|--------|---------|
| 用户 | 无 @ | 快速对话群 | 触发快速对话助手（跳过历史注入） |
| 用户 | 有一个可触发 `@xxx` | 快速对话群 | 触发被 @ 的助手（快速对话助手不触发） |
| 用户 | 无 @ | coordinator 模式 | 交给调度助手路由 |
| 用户 | 有一个可触发 `@xxx` | coordinator 模式 | 跳过调度，直接触发被 @ 的助手 |
| 用户 | 无 @ | 普通群，有默认助手 | 触发默认助手 |
| 用户 | 无 @ | 普通群，无默认助手 | 无响应 |
| 用户 | 有一个可触发 `@xxx` | 普通群（auto） | 触发被 @ 的助手 |
| 助手（非调度） | 任意内容 | coordinator 模式 | 交给调度助手判断是否接力 |
| 调度助手 | 有一个可触发 `@xxx` | coordinator 模式 | 触发被 @ 的业务助手 |
| 助手 | 有一个可触发 `@xxx` | auto 模式 | 触发被 @ 的助手（自己除外） |
| 助手 | 无 @ | auto 模式 | **无响应**，流程到此中断 |
| 助手 | 任意内容 | manual 模式 | **无响应**，消息仅展示 |

## 4. 助手触发助手

### 4.1 消息内容中直接 @（推荐）

助手在回复内容中写 `@目标助手名称`，或在正文中写空格加 `@目标助手名称`，系统自动解析并触发。

```
# 主持人提示词示例
宣布辩题后，写 "@正方一辩 请开篇立论"
```

**@ 解析规则**：
- `@` 位于行首，或 `@` 前一个字符是空格时触发
- 标点后直接 `@`、文字后直接 `@` 不触发（例如 `好的，@张鹏` 中的 @ 不会触发）
- 助手名后跟空格、行尾或标点符号（`!?.,:;！？。，；：`）
- 同一条消息中同一助手被多次 @，只触发一次
- 单条消息最多触发一个助手；出现多个不同助手 @ 时，前端会拦截用户消息，后端只处理第一个有效 @
- 长名称优先匹配（避免"一辩"误匹配"正方一辩"）
- 助手不能 @ 自身触发自身（自引用被静默跳过）
- 被 @ 的助手必须是当前群的成员且处于启用状态，否则静默跳过，不报错

---

## 5. 默认助手

每个群可设置一个默认助手（`defaultAgentId`）。用户发消息**且不包含任何 @** 时，默认助手自动触发。

- 默认助手必须是该群的成员
- 快速对话群不走此逻辑，改用 `quickChatAgentId`

---

## 6. 历史上下文注入

助手被触发时，系统自动将群历史注入上下文（除非 `injectGroupHistory = false`）：

| 配置项 | 默认值 | 说明 |
|-------|--------|------|
| `AGENT_HISTORY_THRESHOLD` | 20 | 超过此条数触发摘要压缩 |
| `AGENT_MEMORY_RECENT_MESSAGES` | 10 | 保留最近 N 条原文注入 |
| `AGENT_MEMORY_COMPACT_MESSAGES` | 40 | 触发摘要的消息数上限 |

**重要影响**：助手能看到触发它的消息是谁发的、内容是什么。这让 LLM 会自然地以 `@发送者` 开头回应（礼貌性称呼），这是模型行为，不是系统注入的前缀。

---

## 7. 定时任务触发

群聊可配置 Cron 定时任务，到时间后系统生成触发消息发入群聊，走普通消息流程触发默认助手或指定助手。

如果任务选择了多个 `agentIds` 或 `["*"]`，调度器会循环发送多条消息：每条消息只自动添加一个 `@助手名`，逐个触发目标助手。这样定时任务可以批量通知多个助手，同时保持“单条消息最多触发一个助手”的规则。

支持两种类型：
- `cron`：标准 cron 表达式（`0 9 * * *` 每天9点）
- `once`：指定某一时间点触发一次

---

## 8. 任务队列与服务重启

助手被触发后，任务会写入 `TaskQueue` 数据库表，状态为 `pending`，随后异步执行。

**服务重启时的行为**：服务启动时会将所有 `pending` 和 `executing` 状态的任务标记为 `interrupted` 并静默丢弃，**不会通知用户**。这意味着：

- 如果桌面应用在助手任务入队但尚未执行期间关闭，该任务会在下次启动后消失
- 受影响场景：auto 模式下助手间接力触发的任务（用户 @ 直接触发的任务通常已开始执行，风险较低）
- 现象：用户看到某助手"没有回应"，但实际上任务曾入队、从未执行

**临时解法**：重新发送消息手动触发对应助手。

## 9. 提示词设计建议

### 9.1 流程接力模式（辩论/比赛/多角色）

**群聊配置要求**：`agentTriggerMode = auto`，设置默认助手作为主控

**核心设计原则**：
> 每个助手的回复中必须包含 `@下一个助手`，否则流程在该节点中断。

**可靠触发的两种策略（二选一）**：

**策略 A：显式规则（最可靠）**

在每个中间节点助手的提示词末尾加入：
```
发言结束后，在消息最末尾单独一行写 @主控助手名，通知其继续安排流程。
发言开头不要 @主控助手，称呼直接用文字，@ 只在末尾出现一次。
```

**策略 B：依赖 LLM 自然行为（不推荐单独使用）**

历史上下文中有触发方的名字，LLM 通常会自然以 `@触发方` 开头回应。但此行为不稳定（LLM 可能直接开始正文），不应作为唯一触发机制。

**推荐**：使用策略 A（显式规则），确保 100% 可用。

### 9.2 主控助手（主持人/裁判/导演）设计

```markdown
## 流程控制规则
- 每次只安排当前阶段的一个助手，不要提前安排下一轮
- 安排时直接写 "@助手名称 请[动作]"，不要附加多余内容
- 等待被点名的助手发言后（系统会再次触发你），再安排下一位
```

### 9.3 执行助手（辩手/选手/演员）设计

```markdown
## 流程通知规则
- 每次发言结束后，在消息最末尾单独一行写 @主控助手名
- 发言开头不要 @主控助手，称呼直接用文字
- @ 在整条消息中只出现一次（末尾）
```

### 9.4 群规则建议

群规定义的是**角色行为约束**，对 AI 触发机制无效。不要在群规中写"等待点名才能发言"来控制触发——触发由系统决定，不由助手自控。

| 适合写进群规 | 不适合写进群规 |
|------------|--------------|
| 发言内容的格式要求 | 触发条件（系统负责） |
| 角色定位说明 | 等待机制（系统负责） |
| 比赛/活动的业务规则 | 谁先谁后的顺序（提示词负责） |
| 语言风格要求 | @ 谁的规则（提示词负责） |

## 10. 常见问题

### Q：流程中断，助手不再响应？

检查链路中某个节点的助手回复是否包含可触发 `@下一个助手`。auto 模式下，助手消息无可触发 @ 时流程必然中断。

### Q：助手被触发两次？

同一条消息中多次出现同一可触发 `@助手名`，系统只触发一次。但若开头和结尾各写一次 @ 导致**展示重复**（视觉上两次 @），需在提示词中明确"可触发 @ 只出现一次，开头用文字称呼"。

### Q：manual 模式下助手能否自动接力？

不能。manual 模式完全阻断助手间触发，只能靠用户手动用可触发 `@助手名` 触发每个助手。

### Q：多个助手被同时 @ 会怎样？

不再并发触发。用户消息会被前端拦截；助手或外部平台消息如果包含多个不同助手 @，后端只处理第一个有效 @。

### Q：coordinator 模式和 auto 模式如何选择？

- **coordinator 模式**：用户不需要知道该找哪个助手，由调度助手自动路由。适合对外服务、工作流不固定的场景。代价是每条消息多一次 LLM 调用。
- **auto 模式**：助手间直接接力，延迟低，流程由提示词明确控制。适合辩论、流水线、角色扮演等结构固定的多轮场景。

### Q：@ 写在中文标点后面不触发？

是已知限制。`好的，@张鹏` 中逗号直接跟 `@`，不满足解析规则，不会触发张鹏。  
解决方法：在 `@` 前加一个空格，写成 `好的， @张鹏`；或另起一行写 `@张鹏`。

### Q：助手 @ 了另一个助手，但对方没有响应？

可能原因：
1. 被 @ 的助手不是当前群成员（系统静默跳过）
2. 服务在任务入队后、执行前重启（任务被标记为 `interrupted` 丢弃），重新发消息手动触发即可
3. `@` 前面是非空格字符，解析未命中（见上一条）
4. coordinator 模式下助手的 @ 先被调度助手拦截，调度助手判断无需接力后不再转发
