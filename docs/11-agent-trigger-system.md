# 助手触发系统规格说明

本文档说明 TeamAgentX 中助手的所有触发规则、群聊模式及提示词设计建议，供创建助手和群规则时参考。

---

## 1. 核心概念

### 1.1 触发的本质

每条消息进入群聊后，系统会扫描消息内容中的 `@助手名称`，并将被点名的助手加入任务队列执行。没有被 @ 的助手不会收到任何消息，也不会有任何输出。

### 1.2 消息类型

| 类型 | 说明 |
|------|------|
| 人类消息 | 用户在聊天框发送的消息 |
| 助手消息 | 助手生成后由系统写入群聊的消息，`isHuman = false` |

---

## 2. 群聊触发模式（agentTriggerMode）

群聊有两种触发模式，决定了**助手消息能否触发其他助手**。

### 2.1 auto 模式（默认）

```
agentTriggerMode = 'auto'
```

助手消息中的 `@xxx` 可以触发其他助手。多助手协作、辩论、接力等场景必须使用此模式。

### 2.2 manual 模式

```
agentTriggerMode = 'manual'
```

助手消息中的 `@xxx` **不触发**任何助手，仅作为展示文字。只有用户发消息才能触发助手。适合单助手对话、不需要助手互相协作的场景。

> **注意**：`manual` 模式下，助手调用 `send_message_to_agent` 工具也会报错，彻底阻断助手间通信。

---

## 3. 完整触发规则表

以下规则按优先级从高到低执行：

| 消息来源 | 消息内容 | 群类型 | 触发结果 |
|---------|---------|--------|---------|
| 用户 | 无 @ | 快速对话群 | 触发快速对话助手（跳过历史注入） |
| 用户 | 有 `@xxx` | 快速对话群 | 触发被 @ 的助手（快速对话助手不触发） |
| 用户 | 无 @ | 普通群，有默认助手 | 触发默认助手 |
| 用户 | 无 @ | 普通群，无默认助手 | 无响应 |
| 用户 | 有 `@xxx` | 普通群 | 触发所有被 @ 的助手 |
| 助手 | 有 `@xxx` | auto 模式 | 触发所有被 @ 的助手（自己除外） |
| 助手 | 无 @ | auto 模式 | **无响应**，流程到此中断 |
| 助手 | 任意内容 | manual 模式 | **无响应**，消息仅展示 |

---

## 4. 助手触发助手的两种路径

### 4.1 消息内容中直接 @（推荐）

助手在回复内容中写 `@目标助手名称`，系统自动解析并触发。

```
# 主持人提示词示例
宣布辩题后，写 "@正方一辩 请开篇立论"
```

**@ 解析规则**：
- 必须以空格、行首或 Markdown 特殊符号（`*_>#\`-`）开头
- 助手名后跟空格、行尾或标点符号（`!?.,:;！？。，；：`）
- 同一条消息中同一助手被多次 @，只触发一次
- 长名称优先匹配（避免"一辩"误匹配"正方一辩"）

### 4.2 工具调用 send_message_to_agent

助手通过调用内置工具显式发消息给指定助手，系统在群聊中生成一条格式为 `@目标助手 内容` 的消息并触发对方。`manual` 模式下此工具调用会报错。

---

## 5. 默认助手

每个群可设置一个默认助手（`defaultAgentId`）。用户发消息**且不包含任何 @** 时，默认助手自动触发。

- 默认助手必须是该群的成员
- 快速对话群不走此逻辑，改用 `quickChatAgentId`

---

## 6. 历史上下文注入

助手被触发时，系统自动将群历史注入上下文（除非 `injectGroupHistory = false`）：

| 配置项 | 默认值 | 说明 |
|-------|--------|------|
| `AGENT_HISTORY_THRESHOLD` | 20 | 超过此条数触发摘要压缩 |
| `AGENT_MEMORY_RECENT_MESSAGES` | 10 | 保留最近 N 条原文注入 |
| `AGENT_MEMORY_COMPACT_MESSAGES` | 40 | 触发摘要的消息数上限 |

**重要影响**：助手能看到触发它的消息是谁发的、内容是什么。这让 LLM 会自然地以 `@发送者` 开头回应（礼貌性称呼），这是模型行为，不是系统注入的前缀。

---

## 7. 定时任务触发

群聊可配置 Cron 定时任务，到时间后系统生成一条触发消息发入群聊（格式：`[定时任务] 任务名: payload`），走普通消息流程触发默认助手或指定助手。

支持两种类型：
- `cron`：标准 cron 表达式（`0 9 * * *` 每天9点）
- `once`：指定某一时间点触发一次

---

## 8. 提示词设计建议

### 8.1 流程接力模式（辩论/比赛/多角色）

**群聊配置要求**：`agentTriggerMode = auto`，设置默认助手作为主控

**核心设计原则**：
> 每个助手的回复中必须包含 `@下一个助手`，否则流程在该节点中断。

**可靠触发的两种策略（二选一）**：

**策略 A：显式规则（最可靠）**

在每个中间节点助手的提示词末尾加入：
```
发言结束后，在消息最末尾单独一行写 @主控助手名，通知其继续安排流程。
发言开头不要 @主控助手，称呼直接用文字，@ 只在末尾出现一次。
```

**策略 B：依赖 LLM 自然行为（不推荐单独使用）**

历史上下文中有触发方的名字，LLM 通常会自然以 `@触发方` 开头回应。但此行为不稳定（LLM 可能直接开始正文），不应作为唯一触发机制。

**推荐**：使用策略 A（显式规则），确保 100% 可用。

### 8.2 主控助手（主持人/裁判/导演）设计

```markdown
## 流程控制规则
- 每次只安排当前阶段的一个助手，不要提前安排下一轮
- 安排时直接写 "@助手名称 请[动作]"，不要附加多余内容
- 等待被点名的助手发言后（系统会再次触发你），再安排下一位
```

### 8.3 执行助手（辩手/选手/演员）设计

```markdown
## 流程通知规则
- 每次发言结束后，在消息最末尾单独一行写 @主控助手名
- 发言开头不要 @主控助手，称呼直接用文字
- @ 在整条消息中只出现一次（末尾）
```

### 8.4 群规则建议

群规定义的是**角色行为约束**，对 AI 触发机制无效。不要在群规中写"等待点名才能发言"来控制触发——触发由系统决定，不由助手自控。

| 适合写进群规 | 不适合写进群规 |
|------------|--------------|
| 发言内容的格式要求 | 触发条件（系统负责） |
| 角色定位说明 | 等待机制（系统负责） |
| 比赛/活动的业务规则 | 谁先谁后的顺序（提示词负责） |
| 语言风格要求 | @ 谁的规则（提示词负责） |

## 9. 常见问题

### Q：流程中断，助手不再响应？

检查链路中某个节点的助手回复是否包含 `@下一个助手`。auto 模式下，助手消息无 @ 时流程必然中断。

### Q：助手被触发两次？

同一条消息中多次出现同一 `@助手名`，系统只触发一次。但若开头和结尾各写一次 @ 导致**展示重复**（视觉上两次 @），需在提示词中明确"@ 只在末尾出现一次，开头用文字称呼"。

### Q：manual 模式下助手能否自动接力？

不能。manual 模式完全阻断助手间触发，只能靠用户手动 @ 每个助手。

### Q：多个助手被同时 @ 会怎样？

全部并发触发，各自独立执行。如需严格按序，主控助手每次只 @ 一个助手。
