# 助手触发系统规格说明

[English](11-agent-trigger-system_EN.md) | 中文

本文档说明 TeamAgentX 中助手的当前触发规则、群聊模式及提示词设计建议，供创建助手和群规则时参考。

> 重要变更（2026-06）：原「自由协作（auto）」与「协调模式（coordinator）」已合并为单一的 **「智能协作」模式**，对外只暴露 **智能协作 + 手动** 两种模式。存储层为减少迁移面仍沿用 `coordinator` 作为智能协作的取值，`auto` 作为历史兼容别名（等同智能协作）；产品文案统一称「智能协作」。合并设计见 [13-unified-collaboration-mode-design.md](13-unified-collaboration-mode-design.md)。

---

## 1. 核心概念

### 1.1 触发的本质

每条消息进入群聊后，系统会先解析消息内容中的可触发 `@助手名称`，再结合群聊模式与协作预算决定谁进入任务队列执行。

- 用户显式 `@` 是强意图，直接触发（智能协作模式下多 `@` 可并行触发）
- 快速对话群在用户消息未 `@` 其他助手时，直接触发 `quickChatAgentId`
- 智能协作模式走「快路径优先 + 协调器兜底」：常规接力由助手自己在回复里 `@下一个助手` 完成，**只有异常/汇合/兜底时**才唤起内置 `群调度助手` 裁决
- 手动模式下助手消息中的 `@` 只作展示，不触发其他助手

### 1.2 消息类型

| 类型 | 说明 |
|------|------|
| 人类消息 | 用户在聊天框发送的消息，`isHuman = true` |
| 助手消息 | 助手生成后由系统写入群聊的消息，`isHuman = false` |

## 2. 群聊触发模式（agentTriggerMode）

群聊现在只有两种对外模式，决定**用户消息和助手消息如何流转**。

### 2.1 智能协作模式（默认）

```
agentTriggerMode = 'coordinator'   # 存储值；'auto' 为历史兼容别名，行为相同
```

新建普通群聊的默认模式。设计目标是 **「auto 的快路径 + coordinator 的完整能力」**：

- **结构化快路径（零协调成本）**：助手调用 `mention_agents` 明确登记目标和任务，TaskQueue 结算后直接派发，不经协调器重猜
- **协调器兜底**：只处理用户路由、用户/协调器计划、旧并行/串行计划汇合和 stall watchdog；结构化助手交接不经过协调器
- **多助手派发**：协调器可把一次决策拆成多个独立任务，按 `dispatchMode` **并行**（同时入队、汇合裁决）或**串行**（按顺序逐个推进）执行
- **收敛保证**：单目标按血缘接力；多目标降级为叶子并由发起者汇合；扇出/depth/总预算/有限重访四重护栏防止扇出风暴和死循环（见 §5）

适合绝大多数多角色协作群——既有固定接力的低延迟，又有自动判断下一步该找谁的能力。

### 2.2 手动模式

```
agentTriggerMode = 'manual'
```

助手消息中的可触发 `@xxx` **不触发**任何助手，仅作为展示文字。只有用户发消息才能触发助手。适合单助手对话、用户全程手动调度、不需要助手互相接力的场景。手动模式不注入交接协议提示词。

## 3. 完整触发规则表

以下规则按优先级从高到低执行。

### 3.1 用户消息

| # | 消息内容 | 群类型 / 模式 | 触发结果 |
|---|---------|-------------|---------|
| 1 | 无 `@` | 快速对话群 | 触发快速对话助手（跳过历史注入） |
| 2 | 一个可触发 `@xxx` | 快速对话群 | 触发被 `@` 的助手（快速对话助手不触发） |
| 3 | 一个可触发 `@xxx` | 普通群任意模式 | 触发被 `@` 的助手 |
| 4 | 多个可触发 `@` | 智能协作 | 视为歧义信号，**交协调器裁决**（拆为单任务 / 并行批次 / 串行链），不静默截断 |
| 5 | 多个可触发 `@` | 手动 / 快速对话 | 只触发第一个有效 `@` |
| 6 | 无 `@`，手动引用某条助手消息 | 普通群 | 触发被引用消息的助手 |
| 7 | 无 `@`，上一条恰是助手 `@` 本用户的提问 | 智能协作 | 直达回复：触发该助手 |
| 8 | 无 `@`，设置了默认助手 | 普通群 | 触发默认助手（智能协作模式**保留**默认助手） |
| 9 | 无 `@`，以上均不命中 | 智能协作 | 交协调器裁决（介入点 ①） |
| 10 | 无 `@`，无默认助手 | 手动 | 无响应 |
| 11 | 用户任意时刻发言 | 智能协作 | 熔断进行中的 watchdog；结构化叶子批次改为用户接管 |

> 用户无 `@` 时的路由优先级：**手动引用回复 > 直达回复 > 默认助手 > 协调器兜底**。

### 3.2 助手消息（非并行批次中）

| # | 消息内容 | 模式 | 触发结果 |
|---|---------|------|---------|
| 12 | 调用 `mention_agents`，一个合法目标 | 智能协作 | TaskQueue 结算后结构化直连接力 |
| 13 | 调用 `mention_agents`，目标拼错/不在群/停用 | 智能协作 | 工具即时拒绝或轮末确定性 ask_owner |
| 14 | 调用 `mention_agents`，多个合法目标 | 智能协作 | 并行叶子批次；全部完成后重新唤醒发起助手收口 |
| 15 | 未调用 `mention_agents` | 智能协作 | 不接力，仅启动 stall watchdog 防抖兜底 |
| 16 | `@` 用户（提问/等确认） | 智能协作 | 不触发助手，等用户答复走规则 7；watchdog 跳过 |
| 17 | 工具目标为自己 | 智能协作 | 工具即时拒绝 `self` |
| 18 | 正文/代码块/引用块内的 `@助手` | 任意 | 只展示，不参与业务助手派发 |
| 19 | 任意内容 | 手动 | 无响应，`@` 仅展示 |

### 3.3 并行批次进行中

| # | 场景 | 处理方式 |
|---|------|---------|
| 20 | 成员完成且带单个合法 `@`（含 `@` 批次内其他成员） | 不触发，标记完成，交接意向**挂起**到汇合点 |
| 21 | 成员消息 `@` 多个助手 | 同样挂起，汇合时一并裁决 |
| 22 | 成员 `@` 用户 | 消息立即可见，不算交接；批次照常等其余成员 |
| 23 | 用户在批次期间发言（任何形式） | **用户介入即接管**：批次只做完成计数收口，汇合时不再自动派发；后续由用户消息路由驱动 |
| 24 | 最后一个成员完成（join） | 协调器汇合裁决（介入点 ③）：独立→开新批次；有依赖→串行/合并；不明→ask_owner |
| 25 | 批次外的独立链路（用户另 `@` 无关助手） | 不受影响，独立走快路径（批次按成员 agentId 追踪，不锁整房间） |

## 4. `@` 解析规则

助手在回复内容中写 `@目标助手名称` 触发接力时，系统先解析有效 mention，再按群模式决定是否触发。

```
# 主持人提示词示例
宣布辩题后，写 "@正方一辩 请开篇立论"
```

**解析规则**：
- `@` 位于行首，或 `@` 前一个字符是空格时才触发
- 标点后直接 `@`、文字后直接 `@` 不触发
- 助手名后须跟空格、行尾或标点符号（`!?.,:;！？。，；：`）
- 同一条消息中同一助手被多次 `@`，只触发一次
- 代码块 / 行内代码 / 引用块中的 `@` 不算可触发提及
- 长名称优先匹配（避免「一辩」误匹配「正方一辩」）

**模式差异**：
- **智能协作**：业务助手必须调用 `mention_agents` 产生结构化交接；正文里的任何 `@助手` 都只展示、不触发
- **手动**：业务助手不做自动交接；正文里的 `@助手` 同样只展示
- **协调器来源**：内置 `群调度助手` 自己发出的 `@`（来源为 `GROUP_COORDINATOR_ID`）直接入队，不再回到协调器（防递归）

## 5. 协作预算与收敛（防扇出 / 防死循环）

智能协作模式下，助手交接从自由文本解析改为 `mention_agents` 结构化意图，并在 TaskQueue 结算后统一派发。

**结构层**：工具只登记目标，不即时派发；轮末读取同一 TaskQueue 的 buffer。单目标直接接力；多目标创建叶子批次，不再让协调器二次猜测目标或任务。

**发布时序与遗漏兜底**：业务助手的最终正文和流式正文先暂存，不立即对外发布。buffer 已有交接时直接合并“正文 + 交接块”一次性发布；首轮正常结束但 buffer 为空时，在发布前复用同一会话做一次静默交接复核。复核输出不发群，只允许补登记 `mention_agents`；最多一次且有独立超时，最终正文只发布一次，失败后才回落到 watchdog。

**批次/链层**：
- **结构化交接批次**：发起者 A 一次交接 B/C 时，B/C 以 `isLeaf=true` 并行执行；叶子调用 `mention_agents` 只形成公开建议，不继续扇出；全部结束后系统重新唤醒 A 综合收口（`structured-handoff-runtime`）；
- **协调器并行批次（fork-join）**：用户/协调器产生的 N 个任务仍由 `parallel-batch-tracker` 汇合；
- **串行链（顺序推进）**：每次只入队队首一个任务，由**队列任务结算事件**（completed）推进下一步，而非助手输出里的 `@`，后续助手通过增量群历史看到前序产出（`serial-chain-tracker` + `task-lifecycle`）。

**预算层**：结构化交接使用四个正交护栏，按根用户消息与血缘确定性计数：

| 熔断器 | 默认值 | 环境变量 | 触发后行为 |
|--------|--------|---------|-----------|
| 单次扇出 | 20 | `AGENT_HANDOFF_FANOUT_MAX` | 整次不派发，ask_owner |
| 血缘 depth | 100 | `AGENT_HANDOFF_DEPTH_MAX` | 不触发下一跳，ask_owner |
| 整棵级联总派发数 | 20 | `AGENT_HANDOFF_BUDGET_MAX` | 不触发下一跳，ask_owner |
| 同一助手单链重访次数 | 1 | `AGENT_HANDOFF_REVISIT_MAX` | 不触发下一跳，ask_owner |

> 熔断升级后只允许 `ask_owner`（以群调度助手名义 `@群主` 说明卡点），采用确定性通知、不经 LLM、不允许协调器续跑。

另有「卡住检测」兜底（stall watchdog）：助手发完消息、房间内无在跑/排队任务且超过 `AGENT_STALL_WATCHDOG_DELAY_MS`（默认 180s）无新活动时，唤醒群调度助手裁决任务是否真结束（介入点 ④），受连续救援上限 `AGENT_STALL_WATCHDOG_MAX_CONSECUTIVE`（默认 5）约束。

## 6. 协调器介入时机（4 个点）

智能协作模式下，内置 `群调度助手`（协调器）从「每跳必经的调度总线」收窄为「路由兜底 + 汇合裁决 + 异常仲裁」，正常链路上它是沉默的。

| # | 介入点 | 触发条件 | 协调器职责 |
|---|--------|---------|-----------|
| ① | 用户消息路由落空 | 用户无 `@`，且引用回复/直达回复/默认助手均不命中 | 裁决派给谁（dispatch / ask_owner / no_dispatch） |
| ② | 用户多 `@` | 用户一次指定多个助手 | 判断并行 / 串行并拆分具体任务 |
| ③ | 协调器批次 / 串行链汇合（join） | 协调器计划的并行批次全部成员完成，或串行链队尾完成 | 拿全部产出统一裁决下一步 |
| ④ | stall watchdog 救援 | 房间闲置超时、无进行中任务、最后一条是业务助手消息且未 `@` 用户 | 真完成→no_dispatch；断链→派发续跑 |

结构化助手交接的目标校验、叶子汇合和护栏升级全部是确定性代码，不调用协调器 LLM；命中护栏时仅借用群调度助手身份 `@群主` 通知。

协调器每次决策写入 `CoordinatorLog`（决策类型、目标助手、是否原文转发、原因、来源等），可在前端「调度日志」查看。介入频率从原 coordinator 模式的 O(每跳) 降到 O(异常 + 汇合点)。

## 7. 默认助手

每个普通群可设置一个默认助手（`defaultAgentId`）。用户发消息**且不包含任何可触发 `@`**、且未命中引用回复/直达回复时，默认助手自动触发。

- 默认助手必须是该群的成员
- 快速对话群不走此逻辑，改用 `quickChatAgentId`
- **智能协作模式保留默认助手**（合并后不再像旧 coordinator 模式那样清空 `defaultAgentId`）：默认助手优先，落空时才由协调器兜底

## 8. 历史上下文注入

助手被触发时，系统通常会将群历史注入上下文（除非 `injectGroupHistory = false`）：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `AGENT_HISTORY_THRESHOLD` | 20 | 超过此条数触发摘要压缩 |
| `AGENT_MEMORY_RECENT_MESSAGES` | 10 | 保留最近 N 条原文注入 |
| `AGENT_MEMORY_COMPACT_MESSAGES` | 40 | 触发摘要的消息数上限 |

补充说明：
- 快速对话群触发 `quickChatAgentId` 时，当前实现会跳过历史注入
- 普通群中的业务助手默认仍按 `injectGroupHistory` 配置注入历史

**重要影响**：助手能看到触发它的消息是谁发的、内容是什么。这会让 LLM 有时自然地以 `@发送者` 开头回应；这是模型行为，不是系统自动加的前缀。

## 9. 定时任务触发

群聊可配置 Cron 定时任务。到时间后，系统会生成一条「看起来像用户消息」的系统消息发入群聊，再按普通用户消息规则处理。

如果任务选择了多个 `agentIds` 或 `["*"]`，调度器会循环发送多条消息：每条消息只自动添加一个 `@助手名`，逐个触发目标助手。这样定时任务可以批量通知多个助手。

如果定时任务没有显式指定 `agentIds`，则后续行为与用户普通消息一致：
- 快速对话群：触发快速对话助手
- 智能协作普通群：按用户消息路由（默认助手 / 协调器兜底）
- 手动普通群：触发默认助手（若已设置）

支持的调度类型：
- `cron`：标准 cron 表达式（`0 9 * * *` 每天 9 点）
- `interval`：固定分钟间隔
- `once`：指定某一时间点触发一次

## 10. 群调度规则（dispatchRules，工作流）

智能协作模式下，群可配置 **群调度规则**（`ChatRoom.dispatchRules`，YAML）来约束协调器的多助手协作编排——相当于把「这个群按什么流程跑」写成结构化工作流。

- 由群助手的 `generate_dispatch_rules` 工具生成：有指令时按指令优化，无指令时按群内助手名册自动生成
- 保存时做结构化（zod）校验，非法格式拒绝落库
- 规则以 YAML 注入群调度助手系统提示，用于裁决「下一棒交给谁」；业务助手按棒在 query 中注入「任务流转/交接（下一棒）」提示
- 无规则时不注入，回退到协调器自由裁决
- 前端「群调度规则」弹框：只读流程图 + YAML 源码编辑 + 保存校验 + 多工作流 tab

## 11. 提示词设计建议

### 11.1 流程接力模式（辩论/比赛/多角色）

**群聊配置要求**：使用智能协作模式（默认）。固定接力链路由助手调用 `mention_agents` 推进；群调度规则可提示每个角色应该把哪部分工作交给谁。

**核心设计原则**：
> 智能协作模式下，中间节点必须调用 `mention_agents`，正文 `@助手` 只展示、绝不触发。

**可靠触发的两种策略（二选一）**：

**策略 A：显式规则（最可靠）**

在每个中间节点助手的提示词末尾加入：
```
发言结束后，调用 mention_agents，将 agent 设为主控助手名、task 写明下一步要处理的事项。
不要依赖正文中的 @助手名触发交接。
```

**策略 B：依赖 LLM 自然行为（不推荐单独使用）**

历史上下文中有触发方的名字，LLM 通常会自然以 `@触发方` 开头回应。但此行为不稳定，不应作为唯一触发机制。

**推荐**：使用策略 A（显式规则），确保稳定接力。

### 11.2 主控助手（主持人/裁判/导演）设计

```markdown
## 流程控制规则
- 每次只安排当前阶段的一个助手，不要提前安排下一轮
- 安排时直接写 "@助手名称 请[动作]"，不要附加多余内容
- 等待被点名的助手发言后（系统会再次触发你），再安排下一位
```

### 11.3 执行助手（辩手/选手/演员）设计

```markdown
## 流程通知规则
- 每次发言结束后，在消息最末尾单独一行写 @主控助手名
- 发言开头不要 @主控助手，称呼直接用文字
- @ 在整条消息中只出现一次（末尾）
```

### 11.4 群规则建议

群规则定义的是**角色行为约束**，对触发机制无效。不要在群规中写「等待点名才能发言」来控制触发——触发由系统决定，不由助手自控。多助手协作的编排顺序应写进**群调度规则**（§10）或各助手提示词，而非群规则。

| 适合写进群规则 | 不适合写进群规则 |
|---------------|----------------|
| 发言内容的格式要求 | 触发条件（系统负责） |
| 角色定位说明 | 等待机制（系统负责） |
| 比赛/活动的业务规则 | 谁先谁后的顺序（群调度规则/提示词负责） |
| 语言风格要求 | `@` 谁的规则（群调度规则/提示词负责） |

## 12. 常见问题

### Q：流程中断，助手不再响应？

先确认当前群模式：

- **智能协作**：检查执行详情中是否调用了 `mention_agents`、目标是否仍在群内，以及是否命中扇出/depth/总预算/重访护栏。正文 `@助手` 不会触发
- **手动**：助手之间本来就不会自动接力，需用户手动 `@` 每个助手

### Q：助手被触发两次？

`mention_agents` 在同一 TaskQueue 执行中按 agentId 并集去重，同一目标后一次 task 覆盖前一次。正文里重复出现 `@助手名` 不会触发任务。

### Q：智能协作模式下，为什么助手写了 `@另一个助手` 却没直接触发？

这是预期行为：业务助手正文里的 `@另一个助手` 已经永久关闭触发能力。助手必须调用 `mention_agents`；未知目标和 @自己会在工具调用时立即返回错误。

### Q：用户一次 `@` 多个助手会怎样？

智能协作模式下会**交协调器裁决**：由协调器把多个 `@` 拆成多个独立任务，按 `dispatchMode` 并行（同时执行、汇合裁决）或串行（按顺序逐个执行）下发，而不是无脑全触发。手动模式与快速对话群只触发第一个有效 `@`。

### Q：怎么避免助手互相 `@` 死循环？

结构化交接携带完整 lineage，并同时限制单次扇出、depth、根消息总派发预算和同一助手重访次数。命中任一护栏都会停止自动派发并 `@` 群主；用户在并行叶子批次期间发言会接管后续推进。
