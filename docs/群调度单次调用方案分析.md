# 群调度助手：单次结构化 LLM 调用方案分析

> 状态：方案评估（未实施）
> 范围：把"群调度助手 / 内置协调器"从 ACP 执行栈改为一次性结构化 LLM 调用
> 关联代码：`internal-coordinator-agent.ts`、`coordinator-context.ts`、`parallel-batch-tracker.ts`、`agent-dispatch.service.ts`、`room-message-index.service.ts`、`agent-handler/handler.ts`、`executor-manager.ts`

---

## 1. 背景与现状

群聊"协调模式"下，所有消息（人类发言、助手产出）都先交给一个隐藏的系统助手 **群调度助手**（`GROUP_COORDINATOR_ID`）裁决：该不该调度、调给谁、转发什么。它只路由，不回答、不执行。

### 1.1 当前执行栈

- 群调度助手是 **ACP 助手**（`type: 'acp'`, `acpTool: 'claude'`），通过 `createExecutor` 走完整的 Claude Code / Codex agent 运行时（`executor-manager.ts:140`）。
- `stateless: agent.id === GROUP_COORDINATOR_ID`（`executor-manager.ts:154`）——每次裁决都是独立调用，不依赖 session 记忆。
- `injectGroupHistory=false`——它不是群成员，看不到群历史段。

### 1.2 输出怎么落地

- 它的输出是**一条自由文本群消息**，经 `emit → buildAIMessage` 广播进群（`processor.ts`）。
- 下游靠这条消息里的 `@助手名` 文本触发目标助手；多助手并行 = 一条消息 `@A @B 原文`。
- 三个"终态哨兵"：
  - `无需调度`（中文）
  - `Cannot dispatch: no suitable assistant in this chatroom`（英文）
  - `Cannot dispatch: system-management request`（英文）

### 1.3 并行任务追踪（已有后端状态机）

并行完成判断**不在 LLM**，而在后端内存状态机 `parallel-batch-tracker.ts`：

1. 协调器派发多个助手 → `handler.ts:447` 收集 `dispatchedAgentIds`，>1 时 `startParallelBatch(chatRoomId, ids)`。
2. 每个并行助手完成 → `handler.ts:268` 先 `markBatchAgentComplete`：`pending`（还有人没完成→不唤醒协调器）/ `last`（全完成→唤醒一次）/ `none`（不在批次）。
3. "全完成"= `pendingAgentIds` Set 清空，确定性判断。

提示词里"等所有并行助手完成才进下一阶段"只是第二道保险。

## 2. 现状问题清单

| # | 问题 | 依据 | 严重度 |
|---|------|------|--------|
| P1 | 两个英文哨兵会泄露成群消息 | 代码只识别 `'无需调度'`（`internal-coordinator-agent.ts:12`、`processor.ts:146`），`Cannot dispatch:` 无任何抑制逻辑，会原样广播进群 | 高（用户可见噪音） |
| P2 | 哨兵字符串匹配脆弱 | 精确 `trim===`，模型补标点/翻译即漏判 | 中 |
| P3 | 助手名靠文本 @ 解析，可能编造/写错 | 协调器输出文本名 → mention 解析；写错则派发缺漏，连带 `startParallelBatch` 批次集合不准 | 高 |
| P4 | 用户原文靠提示词约束"不得改写"，执行力弱 | 大段提示词约束，仍可能被模型扩写 | 中 |
| P5 | 无群历史时缺 `[待裁决消息]` 标记 | `withCoordinatorContext` 空上下文原样返回（`coordinator-context.ts:56`），与提示词"开头有标记"不一致 | 低 |
| P6 | 参考历史是 100 字截断 + 去换行 | `normalizePreview`（`room-message-index.service.ts:35`），并行/阶段判断信息不足 | 中 |
| P7 | 跑完整 ACP 运行时做纯路由，开销大 | 每次带内置工具表/框架 preamble，对路由无用 | 中（成本） |

## 3. 提议方案：单次结构化 LLM 调用

### 3.1 核心思想

协调决策本质是**"分类 + 抽取"**（该不该调度、调给谁、转发什么），不需要 agentic loop、文件/bash 工具、session 记忆（`stateless=true` 已证明）。把它从 ACP executor 换成**一次性结构化 LLM 调用**（function calling / JSON schema 强制输出），复用现有 `LlmProvider`（项目已有直接调 LLM 的先例：`prompt-optimize.service.ts`）。

### 3.2 结构化输出 schema

```ts
// tool: dispatch_decision（强制调用）
{
  decision: "dispatch" | "no_dispatch" | "ask_owner" | "cannot_dispatch",

  // decision === "dispatch" 时必填，取自注入的群成员 ID 清单（真实 agentId，不是名字）
  targetAgentIds?: string[],

  // dispatch / ask_owner 时要转发的内容
  content?: string,

  // true = 后端用原始消息库内容派发，忽略模型对 content 的任何改写
  forwardVerbatim?: boolean,

  // cannot_dispatch 细分
  reason?: "no_suitable_assistant" | "system_management",
}
```

后端按 `decision` 分发：

- `dispatch` → 对每个 `targetAgentIds` 调用现成的 `dispatchToAgent(targetAgentId)`（`agent-dispatch.service.ts`，支持 ID 派发 + 全套校验）。
- `ask_owner` → `@群主`，`content` 原样保留 Markdown 格式。
- `no_dispatch` / `cannot_dispatch` → 静默（仅记审计，不进群）。

### 3.3 解决了哪些问题

| 问题 | 结构化方案如何解决 |
|------|---------------------|
| P1/P2 哨兵 | `decision` 是枚举字段，不再有字符串匹配；三个终态变成结构化分支，提示词里"精确输出 xxx"全删 |
| P3 助手名 | `targetAgentIds` 用真实 ID，直接喂 `dispatchToAgent`，根除编造/写错；批次集合一定准确 |
| P4 原文 | `forwardVerbatim: true` + 原始消息 ID，后端用库里原文派发，模型连改写机会都没有 |
| P7 开销 | 不起 ACP 运行时，无内置工具表/框架 preamble |

### 3.4 协议适配（强制结构化输出）

协调器的 `LlmProvider` 协议可能是 `anthropic` 或 `openai`，两边都要适配：

- **Anthropic 协议**：`tool_choice: {type: "tool", name: "dispatch_decision"}` 强制调用单工具。
- **OpenAI 协议**：`tool_choice` 指定函数 + `response_format: {type: "json_schema", ...}`。

> 落地时需对两套协议各验证一次"强制输出"行为，不能纸面假设。

## 4. 消息布局与 Prompt Caching 策略

### 4.1 渲染顺序与缓存原理

Prompt caching 是**前缀匹配**，渲染顺序 `tools → system → messages`，前缀任一字节变化 → 其后全部失效。把**稳定内容放前面**、**易变内容放最后**。

```
[tools]    dispatch_decision schema        ← 不变，前缀
[system]   协调器系统提示词                 ← 不变，前缀
[system]   群成员清单（含 agentId）         ← 半稳定（成员变动才变）
─────────── cache_control 断点 ───────────
[messages] [群最近消息 · 仅供裁决参考]       ← 每次变，不缓存
[messages] [待裁决消息] <原文>              ← 每次变，不缓存
```

### 4.2 关键认知：协调器的"历史"不是稳定前缀

普通多轮聊天能吃到缓存，是因为历史 **append-only**（旧的固定、新的追加）。但协调器是 **stateless**，"最近 N 条预览"是**滑动窗口**——每次重新取、内容每次变、还是截断预览，**不是 append-only，放前面也缓存不了**。真正可缓存的只有：系统提示词、工具 schema、（半稳定的）成员清单。

### 4.3 缓存门槛（决定缓存是否真的生效）

| 模型 | 最小可缓存前缀 |
|------|----------------|
| Opus 4.8 / 4.7 / 4.6 / 4.5、Haiku 4.5 | **4096 token** |
| Fable 5、Sonnet 4.6、Haiku 3.5 | 2048 token |
| 老 Sonnet | 1024 token |

协调器系统提示词约 1941 字符 ≈ 1.0k–1.4k token，加工具 schema + 成员清单，**在 Opus 上大概率不到 4096 → 静默不缓存**（不报错，`cache_creation_input_tokens=0`）。

**结论**：prompt caching 不是本方案的主要省钱理由。确定的省钱来自砍掉 ACP 框架开销（§3.3 的 P7）；缓存只在协调器跑 Sonnet 4.6（2048 门槛）或固定前缀自然超 4096 时才兑现红利。

## 5. 并行任务管理

### 5.1 单次调用不影响并行追踪

并行完成判断本来就在后端 `parallel-batch-tracker.ts`，**不在 LLM**。状态机在进程内存，与协调器是 ACP 还是单次调用完全正交。无状态调用照样工作：后端决定"何时唤醒"，LLM 只在被唤醒那次回答"下一步给谁"。

### 5.2 结构化方案让批次更准

现状批次集合依赖"协调器文本 @ 被正确解析成 `dispatchedAgentIds`"，写错就漏。结构化输出 `targetAgentIds` 是真实 ID，`startParallelBatch` 直接用，文本解析误差消失。

### 5.3 现有 tracker 的独立局限（重构可顺手加固）

这些与"单次调用"无关，是现状就有的：

1. **纯内存 + key 只到 chatRoomId**：进程重启丢批次；一个房间同时只能有一个批次，嵌套/连续并行会互相覆盖。
2. **失败/超时不标完成**：`markBatchAgentComplete` 只在助手"产出进协调流程的消息"时调用。某并行助手报错/卡死/没产消息 → 永不从 Set 删除 → 批次永不 `last` → 协调器永不唤醒，靠 `stall-watchdog` 兜底。
3. "完成"判定点是"产了条消息"，非语义完成。

**建议升级**：用 `batchId` + 绑定 `ExecutionRecord` 的 `completed/failed` 确定性信号来标完成，而非"产了消息"；必要时持久化。这样失败的并行助手也能正确计入批次终结。

## 6. 参考历史注入策略

### 6.1 现状

`buildMessageIndex` 在源头把每条 `toIndexMessage` 成 100 字预览（`DEFAULT_PREVIEW_CHARS=100`），且 `content: preview` 覆盖原文（`room-message-index.service.ts:61`）——协调器拿不到原文、换行被压平。

### 6.2 分层注入设计

- **最近 5 条**：保留较完整内容，长消息取**首尾各 300 字、中间省略**，建议**保留换行**（列表/选项/代码块结构不丢）。
- **更早 K 条**：仍用 100 字头部截断。

```ts
// 近 5 条
function headTailPreview(content: string, head = 300, tail = 300): string {
  const t = content.trim();
  if (t.length <= head + tail) return t;
  return `${t.slice(0, head)}\n…（中间省略 ${t.length - head - tail} 字）…\n${t.slice(-tail)}`;
}
```

### 6.3 落地障碍与改法

`buildMessageIndex` 有 4 个消费者（`executor-manager`、`processor`、`agent-dispatch`、`coordinator-context`），**不能改全局截断**。推荐：给 `buildMessageIndex` 加 `options?: { limit?; includeRawContent? }`，`RoomMessageIndexHistoryMessage` 加可选 `rawContent`；其他 3 个消费者不传 options，行为不变；协调器传更大 limit + `includeRawContent`，在 `coordinator-context.ts` 自己做分层格式化。

### 6.4 待裁决消息保持完整

它是核心裁决对象，不截断；附件前缀 `[附件:...]` 格式不变。

## 7. 成本分析

### 7.1 单次调用 ≠ 更贵

协调器现状只注入"系统提示词 + 最近 5 条预览 + 当前消息"（非全历史）。单次调用注入量**与现状持平**。

反而更省：ACP 每次裁决携带十几个内置工具 schema + 框架 preamble（约 1–3k+ token 固定开销），对路由无用但每次付费。单次调用只带系统提示词 + 成员清单 + 历史 + 待裁决 + 一个 dispatch schema。

### 7.2 分层历史的 token 量级

- 近 5 条 × 最多 600 字 ≈ 3000 字
- 更早 K=10 × 100 字 ≈ 1000 字
- 合计 ≈ 4000 字 ≈ **4k–6k token**（中文），比现状（5×100=500 字）约高 8 倍

这部分历史每次变、缓存不了，是每次实付的增量。是否值得取决于并行/阶段误判率，建议上线后对着误判率调 K 与首尾字数。

### 7.3 缓存经济性（够 4096 线时）

- cache read ≈ 基础输入价 0.1×；cache write 1.25×（5 分钟 TTL）/ 2×（1 小时）
- 5 分钟 TTL 下两次请求回本；协调器高频，够线就稳赚
- 但 Opus 门槛 4096，固定前缀大概率不够（§4.3）

## 8. 收益总结

1. **正确性**：消灭哨兵字符串匹配脆弱（P1/P2）、助手名编造（P3）、原文改写（P4）三类问题。
2. **可靠性**：`targetAgentIds` 真实 ID 让并行批次集合一定准确。
3. **可维护性**：终态变枚举，提示词大幅缩短，删掉"精确输出 xxx"的脆弱约束。
4. **成本**：砍掉 ACP 框架固定开销（确定收益）；prompt caching 为额外红利（条件性）。

## 9. 风险与不确定性

| 风险 | 说明 | 缓解 |
|------|------|------|
| 两套协议强制输出 | anthropic / openai 协议的 `tool_choice` / `response_format` 行为需各自验证 | 落地前各跑一次验证，不纸面假设 |
| 成员清单需带 agentId | 现状注入给模型的成员信息可能只有名字 | 注入时补 `agentId`，模型才能填 `targetAgentIds` |
| 中等重构 | 需为协调器单开"结构化 LLM 调用"路径，绕开 `executor.factory` | 与小修解耦，独立 PR |
| 历史 token 上升 | 分层后 4k–6k token/次 | K 与首尾字数可调；先保守 |
| 缓存可能不生效 | Opus 4096 门槛 | 不把缓存当主要省钱理由 |

## 10. 工作量与落地分步

**第一步（止血，低风险，可先做）**
- 哨兵静默：识别并抑制三个终态哨兵（含尾随标点容错）。
- `[待裁决消息]` 一致性：空上下文也包裹标记（P5）。
- 这两项是纯函数 + 提示词字符串改动，无 DB/接口变更。

**第二步（结构化协调器，独立 PR）**
1. 新增 `dispatch_decision` schema + 两套协议的强制输出适配。
2. 给协调器单开"结构化 LLM 调用"路径，复用 `LlmProvider`。
3. 成员清单注入补 `agentId`。
4. 后端按 `decision` 分发到 `dispatchToAgent` / `@群主` / 静默。
5. 参考历史分层注入（§6 改法）。
6. 消息布局 + cache_control 断点（§4）。

**第三步（可选加固）**
- `parallel-batch-tracker` 升级到 `batchId` + `ExecutionRecord` 信号（§5.3）。

## 11. 待决策项

1. **协调器实际 provider/模型**：决定缓存门槛是否够线（Sonnet 4.6=2048 / Opus=4096）。
2. **更早历史取几条（K）**：建议 10 左右或用总 token 预算卡。
3. **近 5 条是否保留换行**：建议保留（否则首尾 300 仍是平文本）。
4. **第 1 条英文哨兵走静默还是友好提示**：建议静默（避免内部裁决泄露成群消息）。
