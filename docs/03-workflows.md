# 03 · 工作流程

> 这一章把"用户从下需求到拿到交付"的全过程拆开，看每一步在平台层走了什么，对应哪些 v0.1.0 已实现的能力。

## 1. 完整生命周期

```
① 建群        用户：新建"我的博客重构"群，分配工作目录
② 配团队      用户：拉总控+工程师A+工程师B+质检入群，套默认开发模板
③ 设群规则    用户：选/改群规则（艾特规范、轮次上限、收尾权）
④ 下需求      用户：@总控 "把首页改成 SSR + 加深色模式"
⑤ 拆任务卡    总控：理解 → 产出 N 张任务卡（含 owner、验收标准、边界）
⑥ 派单        总控：@工程师A 接 TC-001、@工程师B 接 TC-002（可并行）
⑦ 执行        工程师：领卡 → 改文件 → 跑测试 → 更新任务卡状态
⑧ 验收        工程师交付 → 总控 @质检 → 质检按"预期输出 + 不需要做"审查
⑨ 闭环        通过 → 任务卡 done；不通过 → 退回工程师 + 备注
⑩ 交付收尾    所有任务卡 done → 总控 [讨论结束] + 汇总变更清单
```

每一步对应的平台能力：

| 步骤 | 平台能力 | v0.1.0 状态 |
|------|---------|------------|
| ① 建群 | 工作目录隔离、群创建 | ✅ |
| ② 配团队 | 助手模板复用、agentLevel 区分 system/normal、批量加成员 | ✅ |
| ③ 设群规则 | 群规则注入到所有助手上下文 | ✅（仅自律层）|
| ④ 下需求 | @ 触发、默认接收助手 | ✅ |
| ⑤ 拆任务卡 | 任务看板 | ✅（schema 待统一）|
| ⑥ 派单 | @ 路由、串行响应 | ✅ |
| ⑦ 执行 | 流式输出、工具调用、文件系统访问 | ✅ |
| ⑧ 验收 | 质检助手 + LLM 复核 | 🟡（无客观验收钩子）|
| ⑨ 闭环 | 任务状态切换 | 🟡（无强约束）|
| ⑩ 收尾 | 防循环、讨论结束标记 | 🔵（待硬化）|

## 2. 群内消息流转协议

```
用户/助手发言
   ↓
[消息总线] 解析 @ 列表 + 群规则校验
   ↓
   ├─ 自律层规则：注入到所有助手上下文（提示词层）
   └─ 他律层规则（计划中）：消息总线 hook 拦截违规
   ↓
被 @ 的助手 → 进入"待响应"队列
未被 @ 的助手 → 默默吃上下文不发言（如订阅了相关任务卡）
   ↓
按队列顺序串行响应（不抢答）
   ↓
助手响应 = 系统提示 + 群规则（自律层）+ 群上下文窗口 + 自己的记忆切片 + 装载 skill
   ↓
   ├─ 流式输出展示：streamingContent / streamingThinking / toolCalls
   └─ 持久化：messages 表 + executionRecords
   ↓
响应里如含新的 @ → 入队下一轮（自动模式）
响应里如含新的 @ → 不入队，仅显示提及（手动模式）
响应里如含 [讨论结束] 且发言者是总控 → 群闭环
   ↓
轮次 +1，到达上限（默认 3 轮）→ 强制总控收尾
```

### 2.1 三个底层不变量

1. **只有总控能宣布结束**（独占结束权 → 防循环）
2. **助手不抢答**（被 @ 才发言 → 防 @ 触发歧义）
3. **每轮先解析 @ 再发言**（排除歧义后再消耗 token → 防上下文爆炸）

### 2.2 触发模式（v0.1.0 已实现）

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **自动模式（推荐）** | 助手消息中的 @ 会触发其他助手执行任务 | 协作流水（总控派单 → 工程师交付 → 质检审查）|
| **手动模式** | 助手消息中的 @ 不会触发其他助手，仅作提及 | 用户掌控调度，助手不串台 |

### 2.3 默认接收助手

群主（用户）发送的消息没 @ 任何人时，自动 @ 群配置的"默认接收助手"。
通常默认设为**总控**——用户随口一句"这个谁来看一下"，总控接住后再具体派单。

## 3. 任务卡状态机

```
        ┌──────────┐  总控创建
        │   todo   │ ◄──────────────
        └────┬─────┘
             │ 工程师接单
             ▼
        ┌──────────┐  遇阻 ┌──────────┐
        │in_progress├─────►│ blocked  │
        └────┬──────┘      └────┬─────┘
             │ 工程师交付       │ 总控/用户解阻
             ▼                  │
        ┌──────────┐ ◄──────────┘
        │ in_review│
        └────┬─────┘
       通过 │   │ 不通过（带备注退回）
            ▼   └─────► in_progress
        ┌──────────┐
        │   done   │
        └──────────┘
```

### 3.1 状态切换的合法触发方

| 状态切换 | 唯一合法触发方 | 推荐附带证据 |
|---------|-------------|------------|
| `todo` → `in_progress` | owner 助手（领单） | 接单时间戳 |
| `in_progress` → `in_review` | owner 助手（交付） | diff hash + 测试 log（计划） |
| `in_review` → `done` | reviewer 助手（验收通过）| verifications 全过 + 截图（计划） |
| `in_review` → `in_progress` | reviewer 助手（退回）| 退回备注 |
| `*` → `blocked` | 任意助手 | 必须填 `blockers` 字段 |
| `blocked` → `in_progress` | 总控或用户 | 解阻说明 |

非法切换被消息总线拒绝并广播给总控（保证状态一致性）。

### 3.2 任务看板（v0.1.0 已实现）

任务在 6 个状态列展示：

```
┌────────┬────────┬────────┬────────┬────────────┬────────┐
│ 待处理  │等待执行 │ 执行中  │ 已完成  │失败/已取消  │ 待恢复  │
├────────┼────────┼────────┼────────┼────────────┼────────┤
│ TC-003 │ TC-004 │ TC-001 │ TC-000 │   TC-XX    │ TC-005 │
│ TC-006 │        │ TC-002 │        │            │        │
└────────┴────────┴────────┴────────┴────────────┴────────┘
```

- "任务队列按时间顺序执行，当前正在执行第一个待处理任务"
- 点击任务卡跳转对应消息（任务 ↔ 消息双向关联）
- 列可显示/隐藏

## 4. 阻塞上报与人类介入

### 4.1 助手主动上报阻塞的三种情形

| 情形 | 例子 | 上报后流向 |
|------|------|-----------|
| **能力越界** | 工程师被分到"决定要不要做这个功能" | 总控接管，必要时升级到用户 |
| **信息缺失** | 任务卡说接 SSO，但没说用哪个 provider | 总控 @ 资料员 或 @ 用户 |
| **资源/工具失败** | API 失败、依赖装不上 | 总控判断重试 / 换路 / 升级 |

### 4.2 人类介入的三个明确触发点

平台**主动喊人**，不靠用户主动盯：

1. **总控判定阻塞需要人决策** → 推送通知给用户
2. **轮次上限到达且共识未达成** → 强制总控收尾并附"待用户裁决"标签
3. **质检验收三次未通过** → 升级人类介入，避免工程师 ↔ 质检死循环

**核心原则**：用户是"董事会"不是"客服"——平台尽量自闭环，**只在真的需要拍板时才打扰**。

### 4.3 计划中的扩展（详见 04 章 K 节）

引入**风险等级 + 权限模式**——把"是否需要用户拍板"从语言判断升级到平台规则：

```
high 风险动作（删文件 / git push / 调收费 API）→ 平台层前置确认弹窗
medium 风险动作 → 看群权限模式决定
low 风险动作（读文件 / 查文档）→ 自动放行
```

## 5. 群级 Cron 工作流（独有特性）

### 5.1 典型用例

| 场景 | Cron 配置 | 执行内容 |
|------|----------|---------|
| 每日竞品扫描 | `每天 9:00` | "@资料员 爬一下竞品本周更新" |
| 每周代码总结 | `每周五 18:00` | "@总控 总结本周变更并写 changelog" |
| 监控服务状态 | `每小时` | "@监控员 检查服务并报警" |
| 一次性提醒 | `2026-05-15 15:00 一次性` | "@文档员 写 v1 发布说明" |

### 5.2 Cron 触发流转

```
[Cron Worker] 时间到达
   ↓
读取 cron-task 配置（chatRoomId, executionContent, autoMentionAgent）
   ↓
往群里发一条系统消息：
  "@资料员 [Cron 触发] 爬一下竞品本周更新"
   ↓
被 @ 的助手照常进入待响应队列，按正常流转执行
   ↓
执行结果回写 executions 表（耗时、是否成功、错误信息）
```

### 5.3 Cron 执行管理

- 历史执行记录：`GET /cron-tasks/:id/executions`
- 立即测试：`POST /cron-tasks/:id/test`（不等到点也跑一次）
- 启用 / 停用：`PATCH /cron-tasks/:id/enable`
- 最大重试次数：失败自动重试

## 6. 快速对话工作流（轻量分支）

### 6.1 何时用快速对话

| 场景 | 群聊 | 快速对话 |
|------|------|---------|
| 复杂多步任务 | ✅ | ❌ |
| 1v1 速问速答 | 重 | ✅ |
| 多角色协作 | ✅ | ❌ |
| 一次性 prompt | 重 | ✅ |

### 6.2 快速对话特点（v0.1.0 已实现）

- 跳过建群直接 1v1 与某助手聊
- "这是一个快速对话群聊，消息会直接发送给助手，无需 @ 提及"
- 可指定独立 workDir，或留空使用默认（每次新会话独立目录）
- 创建 endpoint：`POST /agents/quick-chat`
- 列表 endpoint：`GET /agents/:id/quick-chat-rooms?userId=...`
- 内部仍是一个 chatRoom，标 `isQuickChat: true`

## 7. 跨工作流的横切关注点

### 7.1 上下文管理（与第 04 章 B 节联动）

每条消息流经助手时，助手实际看到的上下文 = 系统提示 + 群规则 + **可见的群消息** + **自己的记忆切片** + skill 提示。

v0.1.0 控制点：
- 加助手到群时 `injectGroupHistory: true|false` 决定是否注入历史
- "清空上下文"按钮重置某助手在某群的对话记忆
- "查看上下文"可检视当前助手能看到什么

计划中：消息分级订阅、自动滚动摘要（详见 04 章 B 节）。

### 7.2 文件协作（与第 04 章 D/E 节联动）

v0.1.0 现状：
- 群级共享 workDir
- 助手通过 Edit/Write/Bash 工具操作文件
- 暂无并发控制、暂无变更广播

计划中：
- 文件级悲观锁（任务卡 `related_files` 升级为锁定表）
- 文件变更事件总线（自动 diff 摘要广播给订阅者）
- 任务卡 ↔ git 分支强绑定（每张卡独立分支，物理隔离）

### 7.3 验收链路（与第 04 章 G/M 节联动）

v0.1.0 现状：
- 质检助手 LLM 复核
- 任务卡 `expected_output` 自然语言

计划中：
- 任务卡 `verifications` 字段（机器可校验断言）
- 三级验收：客观 > 质检 LLM > 用户拍板
- 回归套件：每张完成的任务卡产出契约测试

## 8. 工作流状态可视化

```
[群] ──→ [任务卡看板]
         │
         ├──→ [待处理]    ← 总控刚拆出来
         ├──→ [等待执行]  ← 已派给某助手未接
         ├──→ [执行中]    ← owner 接了正在做
         │      │
         │      └──→ [流式输出可视化]
         │              streamingContent
         │              streamingThinking
         │              toolCalls
         │
         ├──→ [已完成]    ← 通过验收
         ├──→ [失败/已取消]
         └──→ [待恢复]    ← 中断恢复机制（计划增强）
```

每张任务卡都可点击穿透到对应的消息，看到完整的执行链路（`/messages/:id/execution`）。
