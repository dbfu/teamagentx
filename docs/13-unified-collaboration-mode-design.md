## 1. 背景与动机

群聊目前有三种助手触发模式（`ChatRoom.agentTriggerMode`）：

- `auto`（自由协作）：助手之间靠回复里的 @ 自动接力，卡住时由 stall watchdog 延迟唤醒协调器兜底救援；
- `coordinator`（协调模式）：每条助手消息（以及用户无 @ 消息）都交给内置群调度助手裁决派发；
- `manual`（手动模式）：仅用户 @ 触发，助手消息中的 @ 只作展示。

`auto` 和 `coordinator` 在实现上已经高度趋同——auto 的兜底就是协调器本身（`stall-watchdog.ts` 超时后调用 `runCoordinatorDispatch`），而 coordinator 相当于"把兜底前置到每一跳"。两者的本质区别只剩一个旋钮：**助手交接由谁裁决、何时裁决**。

| 行为 | 自由协作 auto | 协调模式 coordinator | 手动 manual |
|---|---|---|---|
| 用户显式 @ | 直接触发（截断到第 1 个 @） | 直接触发（截断到第 1 个 @） | 直接触发 |
| 用户无 @ 消息 | 默认助手 / 回复目标 / 直达回复 | 一律协调器裁决（建群清空 defaultAgentId） | 默认助手 |
| 助手消息带 @ | 直接触发被 @ 助手（接力） | 交协调器裁决后转发 | 仅展示，不触发 |
| 助手消息无 @ | watchdog 闲置超时兜底 | 每条都过协调器（通常 no_dispatch） | 不处理 |
| 并行派发多助手 | 不支持 | 支持，含并行批次追踪 | 不支持 |
| ask_owner 转发 @用户问题 | 仅 watchdog 兜底时有 | 有 | 无 |
| 系统提示词 | 注入「收尾交接协议」+ 每轮 handoff reminder | 无 | 无 |

两种模式各有代价：

- coordinator 模式：一个 10 跳的协作链要付 10+ 次协调 LLM 调用，每跳增加延迟，大量调用只是输出"无需调度"；
- auto 模式：缺失并行派发、ask_owner、非法 @ 纠错；多 @ 静默截断（`getTriggerMentionNames` 的 `slice(0, 1)`）可能触发错误的助手。

本方案将两者合并为一种**「智能协作」模式**，最终模式收敛为两个：**智能协作 + 手动**。

## 2. 核心设计

一句话概括：**auto 的快路径 + coordinator 的完整能力。协调器从"每跳必经的调度总线"收窄为"路由兜底 + 汇合裁决 + 异常仲裁"，正常链路上它是沉默的。**

### 2.1 基本规则

1. **用户显式 @** → 直接触发。用户多 @ 是强意图，不经协调器二次裁决；@ 多个助手时直接并行触发并开并行批次。并行数受并发上限约束，**超限时必须以可见消息明确提示用户哪些助手已触发、哪些被截断（不允许静默截断）**，被截断的助手由用户在当前任务完成后单独 @。
2. **助手消息带恰好一个合法 @**（目标存在、活跃、是群成员，且当前不在并行批次中）→ 快路径直接触发接力，零协调成本。
3. **助手消息 @ 数量 ≠ 1 或目标不合法** → 交协调器裁决（详见 §3 介入点 ②）。
4. **助手消息无 @** → 按交接协议视为"任务完成或交还用户"，不立即处理，仅启动 stall watchdog 防抖兜底。
5. **用户无 @ 消息** → 路由优先级：手动引用回复 > 直达回复 > 默认助手 > 协调器裁决。与现状 coordinator 模式不同：**保留默认助手**（不再建群时强制清空 defaultAgentId），默认助手优先、协调器兜底。
6. **系统提示词**：智能协作模式统一注入「收尾交接协议」（`buildAgentBaseSystemPrompt` 中的 `collaborationTriggerCheckSection`）和每轮 handoff reminder（`buildHandoffTurnReminder`）。**手动模式不注入交接协议**——手动模式的语义是"助手 @ 不触发"，注入"必须用 @ 交接"会造成提示词与系统行为冲突；改为注入手动模式专用说明："助手消息中的 @ 仅作展示，不要依赖 @ 触发其他助手，交接建议交由用户决定"。

### 2.2 收敛性设计（防发散 / 防死循环）

执行图的形状由三层机制锁死，保证任何一轮协作的任务总量有确定上界（≈ 并发上限 × 跳数预算）：

**结构层：助手出度 ≤ 1，分叉只能发生在裁决点。**

- 业务助手消息最多直接触发一个 @（硬限制，沿用现有 `slice(0, 1)` 思路，但多 @ 不再静默截断而是升级协调器）；
- 并行派发是协调器和用户的特权；助手的分叉意图（多 @）必须经协调器背书后以批次形式合法化。

**批次层：fork-join 语义，分叉必须汇合。**

- 协调器或用户并行派发 N 个助手时开启并行批次（复用现有 `parallel-batch-tracker`）；
- **批次进行中，成员消息里的任何 @ 一律不触发、只记录**，挂起到汇合点；
- 最后一个成员完成（join）时，协调器拿全部产出 + 挂起的交接意向统一裁决下一步；
- 由此保证每次分叉后宽度在裁决点归一，树不会在批次内部生长。

> 设计取舍说明：批次内挂起 @ 是**收敛性优先于协作效率**的有意取舍，不是性能最优解。
> B 已经明确 @D 时仍要等 C 完成才派发 D，会带来一次额外延迟；换来的是"任何分叉必经
> 汇合点归一"的硬保证，杜绝批次内扇出失控。该延迟仅出现在低频的并行批次场景。

**预算层：三重熔断，计数窗口均为"两次人类发言之间"。**

| 熔断器 | 默认值 | 触发后行为 |
|---|---|---|
| 跳数预算：自动触发任务带深度计数，派发时继承 +1 | 20 跳（`AGENT_MAX_HANDOFF_HOPS` 可调）。病态环路主要由环路检测兜底，跳数只做绝对保险；定低了会误杀游戏/长流水线等轮辐式长链路 | 不触发下一跳，直接 ask_owner 说明卡点 |
| 环路检测：同一对助手（A↔B）**连续**往返、无第三方/用户介入 | 3 个来回 = 6 连跳（`AGENT_HANDOFF_CYCLE_REPEAT_LIMIT`） | 截断，直接 ask_owner |
| 并发上限：单次派发同时触发的助手数 | 3（`AGENT_MAX_PARALLEL_DISPATCH`） | 协调器派发硬截断；用户多 @ 截断并可见提示 |

> 熔断后的决策空间被限制为 ask_owner：既然已命中安全熔断，就不再允许协调器 dispatch
> 续跑。实现上采用确定性通知（不依赖 LLM）：以群调度助手名义 @群主 说明最后一跳/循环
> 模式与卡点，由群主决定是否继续。

人类发言清零全部计数（沿用 `resetStallWatchdog` 模式）；用户随时介入即随时熔断进行中的自动调度（沿用 `abortWatchdogDispatch` 链路）。watchdog 自身的连续救援上限（`stallWatchdogMaxConsecutive`）保持不变。

## 3. 协调器介入时机（仅 5 个点）

| # | 介入点 | 触发条件 | 协调器职责 |
|---|---|---|---|
| ① | 用户消息路由落空 | 用户无 @，且引用回复 / 直达回复 / 默认助手均不命中 | 裁决派给谁（dispatch / ask_owner / no_dispatch） |
| ② | 助手 @ 异常 | @ 目标不合法（拼错、不在群、停用），或一条消息含 ≥2 个可触发 @ | 判断真实意图：纠错派发 / 开批次并行派发 / ask_owner |
| ③ | 并行批次汇合（join） | 批次全部成员完成 | 拿全部产出 + 挂起的交接意向统一裁决下一步 |
| ④ | stall watchdog 救援 | 房间闲置超时、无进行中/排队任务、最后一条是业务助手消息且未 @ 用户 | 真完成 → no_dispatch 静默；断链 → @ 助手续跑 |
| ⑤ | 熔断升级 | 跳数预算超限或环路检测命中 | 仅 ask_owner（确定性通知，不经 LLM、不允许 dispatch 续跑） |

明确**不介入**的快路径：用户显式 @（单/多）、助手单个合法 @ 接力、助手无 @ 结束（仅 watchdog 兜底，避免退化回"每次任务收尾白付一次 LLM 输出无需调度"）、助手 @ 用户提问（走直达回复等用户答复，watchdog 跳过）。

介入频率从现状 coordinator 模式的 O(每跳) 降到 O(异常 + 汇合点)。

## 4. 全场景处理对照表

### 4.1 用户消息

| # | 场景 | 处理方式 |
|---|---|---|
| 1 | 用户 @ 一个助手 | 直接触发 |
| 2 | 用户 @ 多个助手 | 直接并行触发，开并行批次 |
| 3 | 用户无 @，手动引用某条助手消息 | 触发被引用消息的助手 |
| 4 | 用户无 @，上一条恰是助手 @ 本用户的提问 | 直达回复：触发该助手 |
| 5 | 用户无 @，设置了默认助手 | 触发默认助手 |
| 6 | 用户无 @，以上均不命中 | 协调器裁决（介入点 ①） |
| 7 | 用户任意时刻发言 | 熔断进行中的 watchdog 自动调度，清零跳数/救援计数 |

### 4.2 助手消息（非批次中）

| # | 场景 | 处理方式 |
|---|---|---|
| 8 | 恰好一个合法 @ | 快路径直接触发，跳数 +1 |
| 9 | 恰好一个 @ 但目标不合法 | 协调器裁决（介入点 ②，纠错或 ask_owner） |
| 10 | ≥2 个可触发 @ | 协调器裁决（介入点 ②）：真并行 → 开批次；单一真交接 → 派一个；不明 → ask_owner |
| 11 | 无 @ 结束 | 不立即处理，仅启动 watchdog 防抖定时器 |
| 12 | @ 用户（提问/等确认） | 不触发助手，等用户答复走场景 4；watchdog 跳过 |
| 13 | @ 自己 | 忽略（沿用 selfMention 跳过） |
| 14 | 代码块/引用块内的 @ | 不算可触发提及（沿用 `parseKnownMentions` 排除规则） |

### 4.3 并行批次进行中

| # | 场景 | 处理方式 |
|---|---|---|
| 15 | 成员完成且消息带单个合法 @（含 @ 批次内其他成员） | 不触发，标记完成，交接意向挂起到 join |
| 16 | 多个成员各自 @ 不同助手 | 全部挂起；join 时协调器统一裁决：独立 → 开新批次；有依赖 → 串行/合并；不明 → ask_owner |
| 17 | 成员消息 @ 多个助手 | 同样挂起到 join，汇合时一并裁决 |
| 18 | 成员 @ 用户 | 消息立即可见，不算助手交接；批次照常等其余成员 |
| 18a | **用户在批次期间发言（任何形式）** | **用户介入即接管**（明确原则）：批次只继续做完成计数收口，join 触发时**不再自动派发**（静默化，含挂起的交接意向），杜绝 join 裁决与用户介入后的新链路（直达回复 / 显式 @ / 兜底裁决）重复派发或上下文竞争。用户消息照常路由（场景 1-6），后续推进由用户驱动；若因此停滞，watchdog 兜底 |
| 19 | 最后一个成员完成（join） | 协调器汇合裁决（介入点 ③） |
| 20 | 批次外的独立链路（如用户另 @ 无关助手 E） | 不受影响，独立走快路径（批次按成员 agentId 追踪，不锁整房间） |

### 4.4 兜底与熔断

| # | 场景 | 处理方式 |
|---|---|---|
| 21 | 房间闲置超时且空闲，最后一条是业务助手消息且未 @ 用户 | watchdog 唤醒协调器（介入点 ④），受连续救援上限约束 |
| 22 | 跳数超预算或环路检测命中 | 直接 ask_owner（介入点 ⑤）：以群调度助手名义确定性通知群主，不经 LLM、不允许续跑 |
| 23 | 协调器派发数量超并发上限 | 硬截断或分批 |
| 24 | 协调器自己发出的多 @ 派发消息 | 不再进协调器（防递归），目标助手直接入队（沿用现有 `GROUP_COORDINATOR_ID` 来源判断） |

## 5. 实现要点

### 5.1 关键改造

- **并行批次计数搬出协调器分支**：现状 `markBatchAgentComplete` 调用位于 `handler.ts` 的协调器触发块内（`handler.ts:46-49` 注释明确警告过批次计数丢失会导致任务永久卡死）。合并后助手可能走快路径直接接力，批次拦截必须移到快路径**之前**的公共路径。
- **挂起意向无需新增存储**：批次中成员的消息照常入库和广播（用户可见），"挂起"只是不调用 `enqueueAgentTask`；join 时协调器走现有 `buildCoordinatorLayeredContext` 读取近期消息，交接意向自然在上下文中。
- **多 @ 不再静默截断**：`getTriggerMentionNames` 的 `slice(0, 1)` 语义改为"≠1 个可触发 @ 时升级协调器"（协调器来源除外）。
- **工作台状态流转统一**：现状 `task-failure-notification.ts`、`agent-dispatch.service.ts` 按 `manual || coordinator` 分支；coordinator 模式靠协调器 no_dispatch 推进 waiting_review，auto 模式靠 processor 的 finally。合并后统一到"群内空闲时流转"判定（两条路径现已趋同，`syncWorkbenchOnRoomIdle` 可作为统一入口）。
- **系统提示词统一**：`collaborationTriggerCheckSection` 与 `buildHandoffTurnReminder` 不再以 `agentTriggerMode === 'auto'` 为条件，智能协作模式统一注入。
- **默认助手保留**：去掉 `chatroom.service.ts` 中 coordinator 模式清空 defaultAgentId 的逻辑（`shouldClearDefaultAgent`）。
- **用户介入接管批次**：`parallel-batch-tracker` 记录批次期间的用户发言（`markBatchUserIntervention`），join 返回 `last_user_intervened` 时静默收口、不触发协调器（场景 18a）。
- **并发截断可见提示**：用户多 @ 超上限时以群调度助手名义发可见提示（仅 UI 同步广播、被截断助手名不带 @，防止提示消息再次触发分派）。

### 5.2 数据迁移与兼容

- `agentTriggerMode` 存量值迁移：保留 `coordinator` 作为智能协作模式的存储值、将存量 `auto` 通过一次 Prisma 数据迁移映射过去（读取层另做别名兜底），减少迁移面；`manual` 不变。**注意：`coordinator` 只是存储层兼容策略，不是产品语义**——UI/API 文案、用户文档统一使用「智能协作」，不再对外暴露「协调模式」的叫法，避免新旧概念混淆。
- 兼容外溢点：
  - `template-snapshot.ts` 的模板导出类型为 `'auto' | 'manual' | 'coordinator'`，老模板导入需兼容映射；
  - 移动端 Flutter 的群设置项；
  - Web 端建群弹窗（`create-group-modal.tsx`）、群设置面板（`room-settings-panel.tsx`）的选项与 i18n 文案（`triggerModeAuto/Coordinator/Manual` 系列）。
- 遵循项目迁移规则：schema 变更（如新增跳数计数相关字段，若需要持久化）必须配套 Prisma migration，禁止 ad-hoc ALTER。

### 5.3 风险

| 风险 | 说明 | 缓解 |
|---|---|---|
| 放弃每跳裁决 | 现状 coordinator 模式每条助手消息都经审核（防 @ 错人、重复触发）；合并后单 @ 快路径只做目标合法性校验，不做语义审核 | 实施前先统计 `CoordinatorLog`：协调器实际改写助手交接目标的比例。若 forwardVerbatim 原样转发占绝对多数，则每跳裁决是纯开销，合并收益确定；若纠正率高，需重新评估 |
| 批次内挂起增加延迟 | B 想交接 D，须等 C 完成、join 裁决后才派发 | 并行批次本身低频，且"汇总各方结果"环节本就该存在；接受此代价换收敛性 |
| 存量 coordinator 群行为变化 | 助手单 @ 从"协调器转发"变为"直接接力"，链路提速但少一层审核 | 同风险 1，依赖 CoordinatorLog 数据决策；发布说明中明示行为变化 |
| 批次期间用户介入的时序 | 用户答复批次成员的 @ 提问后，join 裁决可能与直达回复重复派发 | 已按「用户介入即接管」原则闭合（场景 18a）：用户发言后 join 静默化，不存在双派发路径 |
