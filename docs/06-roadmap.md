# 06 · 差异化定位与路线图

## 1. 我们到底在卖什么

**不是**"另一个 Cursor"。
**不是**"另一个 Devin"。
**不是**"GUI 版的 CrewAI"。

**是**：**让一个开发者拥有"能调度的 AI 团队"——配齐角色、各司其职、按任务卡干活、有边界有验收的虚拟办公室**。

## 2. 三大产品护城河

1. **群聊隐喻**：复用人类的"群协作肌肉记忆"，零学习成本上手
2. **任务卡 + 群即项目**：把"乱聊"变成"有产出的协作"，文件、任务、记忆三层隔离防互相污染
3. **可观测的多 Agent 协作**：流式思考链、执行记录、调试信息、上下文检视——**看得见的协作**比"黑盒 Agent"更有信任感

## 3. 路线图总览

```
阶段一 · 核心闭环硬化（1-2 个月）           ← P0 5 项
阶段二 · 智能化与体验（3-4 个月）           ← P1 7 项
阶段三 · 度量与生态（5-6 个月）             ← P2 / 长期
```

---

## 4. 阶段一 · 核心闭环硬化（1-2 个月，P0 共 5 项）

> 目标：把"任务卡协作"从"看着像"变成"实打实硬"。
> Milestone 名：**"硬化任务卡协作"**

### 4.1 任务列表

| # | 任务 | 对应问题 | 工作量 | 联动 |
|---|------|---------|-------|------|
| **T1** | **任务卡 schema 定稿 + 状态机强约束** | G、所有问题的基础 | S | - |
| **T2** | **A2 扇出风暴防御**（完成型禁艾特 + 聚合窗口 + debounce）| A2、A1 | M | - |
| **T3** | **任务卡 ↔ git 分支强绑定** | D、I、G（联动 1）| M | T1 |
| **T4** | **文件变更事件总线 + 反向订阅** | E | M | T1、T3 |
| **T5** | **客观验收钩子（verifications 字段 + verifier runner）**| G、M（部分）| M | T1 |

### 4.2 周排期建议（8 周）

```
Week 1-2 · T1 任务卡 schema 定稿
  - 5 大字段定稿（owner / steps / expected_output / out_of_scope / verifications）
  - 状态机转移规则代码强约束（非法切换拒绝）
  - 任务看板 UI 适配新 schema

Week 2-4 · T2 扇出风暴防御
  - 群规则增加默认 he 律层条款（完成型回复禁艾特）
  - 消息总线 hook 拦截违规
  - 任务事件通道（task_event）独立于对话通道
  - 总控聚合窗口（按 expected_returns 收齐或超时）
  - 消息 debounce 合并

Week 3-5 · T3 任务卡 ↔ git 分支
  - 任务卡进 in_progress 自动开分支 task-card/<id>
  - 助手所有写文件操作落该分支
  - 每完成一个 step 自动 commit
  - in_review 时自动产出 diff 摘要
  - 完成后由总控/质检合并到 main

Week 5-6 · T4 文件变更事件总线
  - Edit/Write/Bash 操作产生 file-changed 事件
  - 任务卡 related_files 字段反向索引
  - 订阅了相关路径的助手收到 diff 摘要

Week 6-8 · T5 客观验收钩子
  - verifications 字段定义（file_contains/command_passes/screenshot_matches）
  - in_progress → in_review 切换时自动跑校验
  - evidence 落卡（diff_hash / test_log_url / screenshot_url）
  - 不附 evidence 的状态切换被拒绝
```

### 4.3 阶段一交付标准

- [ ] 任务卡 schema 文档定稿，所有字段有规范
- [ ] 状态机非法切换 100% 被拒绝
- [ ] 完成型回复的扇出风暴在测试场景下不再出现
- [ ] 两个工程师并行改同一文件不再丢改动（git 分支隔离）
- [ ] A 改了文件，B 在下一次发言前能看到 diff 摘要
- [ ] 任务卡 done 状态必须有 evidence

## 5. 阶段二 · 智能化与体验（3-4 个月，P1 共 7 项）

> 目标：从"硬"到"聪明"——让团队自己会调度、会避坑、会省钱。

### 5.1 任务列表

| # | 任务 | 对应问题 | 工作量 | 联动组 |
|---|------|---------|-------|--------|
| **T6** | **角色边界从提示词升级为工具白名单** | F | M | 联动 3 |
| **T7** | **风险等级 + 权限模式（plan/normal/acceptEdits/bypass）**| K | M | 联动 3 |
| **T8** | **群规则分层（自律层 + he 律层 hooks）**| L | M | 联动 2 |
| **T9** | **角色组 @ + 智能兜底路由** | C | M | - |
| **T10** | **任务卡心跳产出 + 静默告警** | H | S | - |
| **T11** | **任务卡 step_progress + 决策日志** | I | M | T3 已奠基 |
| **T12** | **助手 → 模型策略路由（complexity 字段）**| J | M | T1 已奠基 |

### 5.2 联动组实施

**联动组 2 · "对话状态分离 + he 律层"**（解决 A2 残留 + L）

T8 与阶段一 T2 联动：T2 的"完成型禁艾特"是默认 he 律层规则之一。T8 把整个 he 律层 schema 化、可配置化、可优先级化。

**联动组 3 · "能力即权限"**（解决 F + K）

T6 + T7 一起做：
- T6 给所有 agent 配 `tools` 白名单
- T7 给所有 tool 打 `risk_level`
- 用户在群级别选 permission_mode
- high 风险动作前置确认弹窗

### 5.3 阶段二交付标准

- [ ] 工程师助手没装 `code-review` 工具就无法 review
- [ ] 删文件、git push 等 high 风险动作有用户前置确认
- [ ] 群规则可分自律层/he 律层独立配置
- [ ] 用户随口"这个谁来看一下"自动路由到合适助手
- [ ] 任务静默 N 分钟后总控收到告警并自救
- [ ] 中断恢复后助手能从 step_progress 继续，不重复或漏步
- [ ] 助手按任务卡 complexity 自动用合适的模型

## 8. 资源与风险

### 8.1 资源估算（按一人开发计）

| 阶段 | 工期 | 关键依赖 |
|------|------|---------|
| 阶段一 | 8 周 | 单人全栈即可 |
| 阶段二 | 16 周 | 可能需要前端 + 后端协同 |
| 阶段三 | 持续 | 涉及生态运营，非纯研发 |

### 8.2 风险与缓解

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|---------|
| 群聊隐喻不够好用，用户还是想要"一键交付" | 中 | 高 | 提供"自动模式"——用户给需求，平台自动建群配团队拆任务自跑，把群聊作为可观测层而非交互层 |
| 多 Agent 成本太高 | 高 | 中 | T12 模型路由策略 + 默认编队精简（3 人最小）+ T15 成本看板让用户看见 |
| 跟单 Agent 工具的竞争 | 中 | 中 | 把 Claude Code / Cursor 等做成可嵌入的 Engineer Agent，不正面竞争而是吸纳（v0.1.0 "本地 Agent" 已开始） |
| 用户不会写群规则 / 任务卡 | 高 | 中 | prompt 优化（已有）+ 默认模板 + AI 协助拆任务（总控的核心能力）|
| 任务卡机制太重，对小任务过设计 | 中 | 中 | 区分"快捷模式"（quick-chat 已有）和"项目模式"（任务卡只在项目模式启用）|
| git 分支策略对非代码项目不适用 | 中 | 低 | 仅对有 `.git` 的工作目录启用，纯文本 / 文档项目用 file snapshot 替代 |
| he 律层规则配置复杂吓退用户 | 中 | 中 | 默认套用合理预设，高级用户可自定义 |
| 客观验收依赖用户写好 verifications | 高 | 中 | 总控自动从 expected_output 推导 verifications，用户可改不必从零写 |

### 8.3 阶段一是真正的"生死线"

阶段一的 5 项 P0 是**必须做**的——不做的话：
- A2 扇出风暴会持续烧用户的钱包，留存崩
- D 文件冲突会让多 Agent 协作变成"反向加速器"
- E/G 让"任务卡 done"成为谎言，用户被愚弄一次就不再信
- T1 schema 是后续所有改造的地基

阶段二阶段三可以根据用户反馈灵活调整顺序，但**阶段一必须按计划走完**。

## 10. 决策记录（DR）

> 路线图制定过程中的关键取舍记下来，将来回顾时知道为什么这么决定。

### DR-001 · 为什么阶段一选这 5 项而不是其他？

候选 P0：A1 / A2 / D / E / G / T1 / 其他。

**最终选 T1 + T2 + T3 + T4 + T5**：
- T1 是地基，所有后续都要它
- T2 解决用户重点提到的扇出风暴
- T3 + T4 + T5 是**联动 1**，一鱼多吃同时解决 D / E / G / I 四个问题
- A1 的紧迫度低（v0.1.0 已有手动模式 + 群规则缓解），归到 T2 顺手做
- F / K 推到阶段二（角色越界严重程度低于扇出风暴和文件冲突）

### DR-002 · 为什么 T6 和 T7 一起做？

T6（工具白名单）和 T7（风险等级）共用同一套"动作打标 + 平台拦截层"基础设施。分开做会做两次类似的事。

### DR-003 · 为什么不一开始就做 M（评估盲区）？

M 是**长期投入**——回归套件、质量看板、trace 平台都需要数据积累。阶段一二做完积攒了执行数据后做 M 才有效，否则看板里全是空数据。

### DR-004 · 为什么 B（上下文爆炸）放阶段三？

B 的当前体感不是最痛——v0.1.0 有 `injectGroupHistory` 和"清空上下文"两个工具用户可手动控制。彻底解决需要消息分级 + 助手订阅，工作量大且依赖前面阶段的任务卡 schema 稳定。

### DR-005 · 为什么不做 IDE 插件？

走出 IDE 是产品定位的核心——我们要做"虚拟办公室"覆盖编辑器之外（资料调研、文档、任务管理）。如果做插件就被锁死在编辑器里，跟 Cursor 同质化。
