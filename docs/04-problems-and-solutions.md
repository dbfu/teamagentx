# 04 · 重大问题点与解决方案

[English](04-problems-and-solutions_EN.md) | 中文

> 这是整套文档的**核心章**。把群聊调度多 AI 助手的 13 个真实坑分到 4 个主题，每个问题用同一套 **7 段式** 展开：现象 / 根因 / 现有应对 / **推荐方案（含落地步骤）** / **备选 / 补充方案**（多路径并列）/ 优先级 / 工作量评估。

> **2026-06 进展（部分问题已硬化）**：智能协作模式（合并 auto/coordinator）落地后，以下问题已由「协作预算 + 三重熔断 + 并行批次 fork-join + 协调器 5 点兜底」机制覆盖：
> - **A1 防循环 / A2 扇出风暴**：跳数（20）/ 连续环路（3 来回）/ 并发（3）三重熔断，命中即停并 `@` 群主；用户多 `@` 超限可见截断。
> - **C @ 触发歧义**：助手 `@` 异常（拼错/多 `@`）升级协调器纠错；保留默认助手兜底。
> - **H 死锁**：环路检测 + stall watchdog 兜底，人类发言随时接管。
> 实现见 [11-agent-trigger-system.md](11-agent-trigger-system.md) 与 [13-unified-collaboration-mode-design.md](13-unified-collaboration-mode-design.md)。下文方案描述仍保留作设计依据与未覆盖部分（客观验收、文件并发、状态机强约束等）的参考。

## 主题一 · 流程控制

### A1 · 助手互相 @ 收不了尾

#### 现象

总控艾特 → 大家回复 → 总控汇总 → 大家补充 → 总控再汇总 → ……
token 烧光、用户体感"群一直在吵"。

#### 根因

1. 每个助手把"礼貌回应"当默认，没人有"宣布结束"的权力。
2. 没人有"我不该发言"的克制——只要被 @ 就回。
3. 缺少明确的"结束条件"。

#### 现有应对（v0.1.x）

- ✅ **智能协作 / 手动两种触发模式**——手动模式下助手消息中的 @ 不触发其他助手；智能协作下单 @ 快路径接力，异常交给协调器。
- ✅ **协作预算三重熔断**——跳数、连续环路、并发上限命中后停止自动派发并 `@` 群主。
- ✅ **stall watchdog**——协作卡住时由协调器兜底裁决。
- ✅ 群规则机制存在（虽是软约束）。
- 🔵 v1 设计稿提出"总控独占结束权"概念，未硬编码到平台层。

#### 推荐方案

**短期（1 周）· 群规则默认条款硬化**

往群规则里写入三条强制条款，初始化新群时默认勾选：

```
1. 只有总控（群主指定）能宣布讨论结束。其他助手不得使用 [讨论结束] 标记。
2. 助手发言后只准艾特总控，不得艾特其他助手。
3. 单个话题最多讨论 3 轮，第 3 轮总控必须强制收尾。
```

**中期（2-4 周）· 平台级"结束权"代码强制**

在消息总线层加判定：

```typescript
// pseudo
function shouldStopGroupChat(group, message): boolean {
  if (message.contains('[讨论结束]') && message.sender === group.coordinator) {
    return true;
  }
  if (group.currentRound >= 3) {
    triggerForceSummary(group.coordinator);
    return true;
  }
  if (detectConsensus(group.recentMessages)) {  // LLM 判共识
    return true;
  }
  return false;
}
```

配套 UI："已收尾" 状态显示，禁止再发言（输入框置灰）。

**长期** · 共识检测从总控提示词升级为**独立的 judge 模型**调用，避免总控自己既当裁判又当队长。

#### 备选 / 补充方案

> 推荐方案核心是"总控独占结束权"。以下方案**单独用**或**叠加**都行，按成本和场景挑。

| 编号 | 思路 | 优点 | 缺点 | 与推荐的关系 |
|------|------|------|------|------------|
| **A1-α · token 预算硬上限** | 任务卡设 token 预算，烧光自动收尾并报告 | 实现简单（1 天）、绝对兜底 | 可能截断有用讨论 | 叠加：作为最后兜底 |
| **A1-β · 时长预算硬上限** | 群讨论持续超过 X 分钟自动收尾 | 用户体感好（避免"挂群烧钱"）| 长任务可能被误伤 | 叠加 |
| **A1-γ · 独立 judge 模型仲裁** | 用不参与对话的小模型每 N 轮判"是否该结束" | 不依赖总控自我判断 | 多一份模型成本 | 替代"共识检测"那一档 |
| **A1-δ · 消息相似度检测** | 最近 3 条与之前 3 条 cosine 相似度 >0.85 → 判定循环自动收尾 | 客观、不靠 LLM 判断 | 复述式讨论可能误判 | 叠加 |
| **A1-ε · 用户硬截断按钮** | UI 上"立即结束讨论"按钮，群主一键截断 + 让总控立即产汇总 | 给用户兜底掌控感 | 需要 UI 工作量 | 叠加，作为最后逃生口 |
| **A1-ζ · 阶段制讨论** | 把讨论分阶段（"提出 / 反驳 / 总结"），每阶段最多 1 轮 | 强结构、防发散 | 灵活度差 | 替代"轮次上限" |

**推荐组合**：推荐方案 + α（token 兜底）+ ε（用户逃生口）。三层防御足以覆盖 95% 的循环。

#### 优先级 · 🔴 高（P0）
#### 工作量 · S（小：1-2 周）

### A2 · 扇出-扇入风暴 ⭐

> 用户重点提的痛点。原文档未单列，本节单列。

#### 现象

```
总控 @工A @工B @质检（一条派单消息）
  ↓
工A 回："已完成，等待 @总控 汇总"
工B 回："已完成，等待 @总控 汇总"
质检 回："已审完，等待 @总控 汇总"
  ↓
消息总线收到 3 条 @总控 → 总控被触发 3 次（每次都跑一次 LLM）
  ↓
总控每次响应又分别 @ 工A / 工B / 质检
  ↓
3 个助手各被 @ 一次，又都回 @ 总控 …… 雪球越滚越大
```

一次正常的 1→3 派单，几轮下来变成 **3ⁿ 级别**的消息和 token 爆炸。

#### 根因

三层叠加：

1. **逐条响应**：消息总线把每条 @ 总控当独立事件触发，总控每次都重新跑一次 LLM 推理。
2. **回复必带 @**：助手把"状态变更"误用了"对话"通道——本来只该说"我交付了"，结果变成 @ 总控来一次对话。
3. **无聚合窗口**：总控没有"等齐再汇总"概念，每收到一条就独立处理一次。

#### 现有应对（v0.1.x）

- ✅ **并发上限**：用户多 @ 或协调器并行派发最多触发 3 个目标，超限目标会被可见截断。
- ✅ **并行批次 fork-join**：批次内成员完成后汇合，统一交由协调器裁决下一步。
- ✅ **协作预算熔断**：多轮扇出/回环命中预算后停止自动派发并 `@` 群主。
- 🔵 **任务卡事件聚合**仍未完整落地；下方“状态变更通道与聚合窗口”仍是后续建议。

#### 推荐方案

**核心原则**：**对话通道只跑讨论，状态变更走任务卡。回复不主动 @，让事件聚合而不是消息回声。**

**第 1 步（1 周）· 群规则增加"完成型回复禁止艾特"**

```
群规则 - 完成型回复禁艾特：
助手在「任务完成 / 已交付 / 审查完毕」类型的回复中，
不得 @ 任何人。状态变更应通过任务卡状态机推进，
而非通过对话通道艾特总控。
```

平台层做 hook 校验：消息发送前正则匹配"完成|已交付|搞定|完毕"等关键词，命中且消息含 @ 则拒绝并提示。

**第 2 步（2-3 周）· 状态变更通道与对话通道分离**

引入**任务卡事件流**作为独立通道：

```
对话消息（chat message）：原有 @ 触发逻辑
   ↓ 完全独立
任务事件（task event）：状态变更走 task_card 状态机
   ↓
   总控订阅"task_event:in_review"事件，事件入聚合窗口
```

任务卡状态机切换 `in_progress → in_review` 时**不发任何对话消息**，只产生一个 `task_event`。

**第 3 步（2-3 周）· 总控聚合窗口**

派单时记录预期回收：

```typescript
const dispatch = {
  id: 'dispatch-001',
  expected_returns: ['工A', '工B', '质检'],
  received: [],
  timeout_at: now + 30 * 60 * 1000,  // 30 分钟超时
  on_complete: () => coordinator.summarize(this.received),
};
```

聚合窗口的两个收口条件：
- **满员**：收到所有 expected_returns 的交付事件 → 触发一次总控汇总
- **超时**：超时未满员 → 触发一次总控汇总（带"以下助手未交付"标记）

**第 4 步（1 周）· 消息总线 debounce**

短时间内（默认 3 秒窗口）对同一接收方的多次 @ 合并成一条触发：

```typescript
const debouncedMentions = new Map();
function onMention(target, msg) {
  const key = target.id;
  const queue = debouncedMentions.get(key) ?? [];
  queue.push(msg);
  debouncedMentions.set(key, queue);
  scheduleDebounce(key, 3000, () => {
    target.handle(queue);  // 一次性处理
    debouncedMentions.delete(key);
  });
}
```

#### 备选 / 补充方案

> 扇出风暴是用户重点提的痛点，方案越多越好——这里给 6 种思路，可组合。

| 编号 | 思路 | 优点 | 缺点 | 与推荐的关系 |
|------|------|------|------|------------|
| **A2-α · 完全事件驱动模式** | 彻底改造：@ 退化为"提及"不触发响应；所有触发走 task_event 事件流 | 根上消除扇出 | 改动量大、改变交互习惯 | 推荐方案的"激进版" |
| **A2-β · 内容相似度去重** | 同一接收方短时间内收到内容相似度 >0.9 的多条 @ → 合并成一次响应 | 透明无感、与现有 UI 兼容 | 相似度阈值不好调 | 叠加：和 debounce 一起 |
| **A2-γ · 限频（rate limit）** | 每个助手对同一接收方每分钟最多发 1 次 @ | 实现极简（1 天）| 偏粗暴 | 叠加：和 debounce 一起 |
| **A2-δ · 后台 worker 批处理** | 总控不实时响应每条 @，由后台 worker 收集 N 秒消息后让总控一次性处理 | 完全消除"逐条触发" | 引入延迟 | 替代"聚合窗口"或与之等价 |
| **A2-ε · 助手交付 = 只更新任务卡，不发对话** | 工程师/质检完成时只切状态机，不在群里发"已完成"消息，由系统自动产出一条系统消息 | 干净彻底 | 用户看不到"工作过程" | 叠加：与"完成型禁艾特"等价但更彻底 |
| **A2-ζ · 群级"消息节奏"配置** | 用户可选"实时模式"（旧行为）/ "聚合模式"（推荐）/ "汇报模式"（只看总控汇总） | 给用户掌控权 | 选项多增加学习成本 | 包装层：把 α/β/γ/δ 做成预设 |
| **A2-η · @ 总控时显示队列预览** | 用户/助手 @ 总控时 UI 显示"还有 2 条 @ 总控待处理"，让发送方主动避免重复 | 教育用户 | 助手不会看 UI | 仅适合用户场景 |
| **A2-θ · 总控合并响应** | 总控被触发后先扫描自己 inbox 的所有未处理 @，一次性回应（不只回当前那条）| 用最少改动改善体验 | 仍可能扇出，只是收口在总控侧 | 替代"聚合窗口"的轻量版 |

**推荐组合**（按改造成本递增）：
- **轻**：γ（限频）+ θ（总控合并）—— 1 周内就能上
- **中**（强烈推荐）：推荐方案（完成型禁艾特 + 聚合窗口 + debounce）+ β（相似度去重）
- **重**：α（事件驱动）—— 一次性根治，但要重构消息总线

#### 优先级 · 🔴 高（P0，用户重点提）
#### 工作量 · M（中：4-6 周）

### C · @ 触发歧义

#### 现象

- 用户随口说"这个谁来看一下"，没 @ → 全场沉默
- @ 多人 → 抢答或互相推让
- @ 角色名（"工程师"）而非具体助手 → 路由不到

#### 根因

- 群消息缺少**强制路由**机制。
- 助手发言权来自"被 @"这一弱信号。
- 没有"角色组 @"概念。

#### 现有应对（v0.1.0）

- ✅ **默认接收助手**：群主发言无 @ 时自动 @ 该助手（已实现）。
- ✅ 串行响应：被 @ 的助手按队列顺序响应，不抢答。
- 🔵 暂无"角色组 @"。

#### 推荐方案

**第 1 步 · 角色组 @**

引入"角色"作为第二种 @ 实体：

```
@工程师 → 平台展开为该群所有 role=工程师 的助手中按规则选一个
@all-工程师 → 全部并行
```

选人规则（按优先级）：
1. 该角色当前空闲（agentStatus=idle）
2. 装载相关 skill 最齐全
3. 历史上对该任务卡领域负载最少
4. 随机

**第 2 步 · 智能兜底（"这个谁来看一下"场景）**

用户的消息没 @ 任何人时：
- 当前已支持 → 路由到群"默认接收助手"
- 增强 → 由总控 LLM 判断该消息适合哪个角色，再二次路由

**第 3 步 · 抢答防御**

被 @ 多人时已经串行（v0.1.0 已实现），增强为：
- 同一时刻只有一个助手"持麦"
- 其他助手即使被 @ 也排队，不能抢说话

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **C-α · 输入框 @ 推荐** | 用户敲消息时基于上下文推荐"想 @ 谁" | UX 顺滑 | 推荐不准会反感 | 叠加 |
| **C-β · 二级路由器** | 所有无 @ 的消息默认到总控，总控决定下发给谁 | 兜底强 | 总控负载重 | 与"默认接收助手"等价的推荐做法 |
| **C-γ · 负载均衡轮询** | 同角色多助手时按"当前任务数"轮询派单 | 防过载 | 助手能力差异未考虑 | 叠加到"角色组 @"|
| **C-δ · 助手能力打分选人** | 每个助手在每类任务上有历史成功率，路由时按分排序 | 智能、长期看效果好 | 冷启动数据不足 | 替代"固定规则"|
| **C-ε · 群主显式指定路由表** | 群规则里写"所有 'bug' 类消息默认 @工程师A" | 100% 可控 | 用户要懂规则语法 | 叠加 |
| **C-ζ · 多人收件并广播但只接一人** | @ 多人时，所有人都收到通知，但只有第一个 ack 的接活（先到先得）| 模拟人类群协作 | 可能竞争混乱 | 替代串行响应 |

**推荐组合**：推荐方案 + α（输入框推荐）+ ε（用户可写路由表）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · M（4-6 周，含 LLM 路由判断）

### H · 死锁与僵局

#### 现象

- 工程师等质检反馈
- 质检等工程师补全
- 双方都"等对方"
- 群冷场，用户睡了觉醒来发现整夜无进展

#### 根因

- 异步消息系统没有超时
- 任务卡状态机没有"卡死"探测器
- 没有"心跳"概念

#### 现有应对（v0.1.0）

- 🔵 任务看板有"待恢复的任务"列，暗示有状态恢复机制，但没有自动僵局检测。

#### 推荐方案

**第 1 步 · 任务卡心跳**

任务卡进入 `in_progress` 后，每 N 步必须更新一次 `last_active_at`：

```yaml
task_card:
  status: in_progress
  last_active_at: 2026-05-09T12:00:00Z
  heartbeat_interval: 5min     # 期望心跳间隔
```

**第 2 步 · 静默告警**

平台后台 worker 每分钟扫描所有 in_progress 状态的任务卡：

```typescript
function checkSilentTasks() {
  const cards = await taskCards.findInProgress();
  for (const card of cards) {
    const silentDuration = now - card.last_active_at;
    if (silentDuration > card.heartbeat_interval * 3) {
      notifyCoordinator(card, 'TASK_SILENT');
    }
  }
}
```

**第 3 步 · 自救与升级**

总控收到 TASK_SILENT 通知后：

1. 第一次：@ owner 助手询问进度
2. 第二次（仍静默）：尝试换路——重新派给另一个同角色助手
3. 第三次：升级到人类用户

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **H-α · 任务依赖图分析** | 把任务卡间依赖建图，A 等 B / B 等 A 立即检测出环 | 数学上严谨 | 只解决"显式等待"型死锁 | 替代"超时检测"的精确版 |
| **H-β · 总控周期巡检** | 总控每 N 分钟主动 @ 所有 in_progress 助手询问"还在干吗"| 主动而非被动 | 引入额外通信成本 | 替代"心跳上报" |
| **H-γ · 用户层"催促"按钮** | UI 一键 @ 所有相关助手，附"用户在等" | 直接给用户掌控权 | 仍依赖用户盯 | 叠加，作为人工兜底 |
| **H-δ · 超时自动换人** | 任务卡静默超时 → 自动换给同角色的另一个助手 | 自愈 | 同角色无替代时无效 | 叠加到推荐方案的"自救" |
| **H-ε · 强制状态变更频率** | 任务卡 in_progress 期间每 X 步必须 commit / 写日志，否则系统判僵局 | 跟 git 分支策略联动 | 简单任务可能误判 | 叠加：和 git 分支结合 |
| **H-ζ · 助手"卡住自报"模板** | 系统提示词里强制：每次响应末尾自评"我接下来会做 / 如果做不到我会..." | 让助手主动暴露卡点 | 模型可能照本宣科 | 叠加 |
| **H-η · 死信队列** | 静默/超时任务进 dead-letter 列，统一让用户审批是丢弃/重试/人工接 | 让用户每天定期处理 | 不实时 | 叠加，作为长期清理 |

**推荐组合**：推荐方案（心跳 + 静默告警）+ α（依赖图）+ γ（用户催促按钮）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · S（2-3 周）

### K · 人类介入时机

#### 现象

要么平台太"乖"——什么都问用户，打断频繁。
要么太"野"——擅自决策走偏，用户事后才发现。

#### 根因

缺少**介入策略**，把"问不问人"完全压在总控提示词里——模型每次回答不一致。

#### 现有应对（v0.1.0）

- 🟡 暂无统一介入策略，依赖助手提示词。

#### 推荐方案

**引入"风险等级 + 权限模式"二元体系**

**第 1 步 · 给所有 Agent 动作打风险等级**

```
risk_level:
  low:
    - 读文件
    - 查文档（资料员）
    - 写新文件（不覆盖）
    - 跑只读脚本
  medium:
    - 修改已有文件
    - 跑构建/测试
    - 创建任务卡
  high:
    - 删除文件
    - git push
    - 执行外部 API（带费用）
    - 改用户配置
    - 装包
```

**第 2 步 · 用户在群级别设权限模式**（参考 Claude Code）

```
permission_mode:
  - plan        # 全部动作只规划不执行，每条都要用户确认
  - normal      # low 自动执行，medium/high 需要用户确认
  - acceptEdits # low+medium 自动执行，high 需要用户确认
  - bypass      # 全部自动（仅适合熟手）
```

**第 3 步 · 高危动作前置确认 UI**

high 风险动作弹出确认弹窗：

```
工程师A 即将执行：
  动作：删除文件 src/legacy/
  原因：清理过时代码
  影响：3 个文件，约 200 行
  撤销：可通过 git checkout 恢复

[确认] [取消] [永久允许此类]
```

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **K-α · 成本阈值触发** | 单任务卡花费超过阈值（如 $5）自动暂停等用户 | 防"暗中烧钱"| 任务复杂度无关 | 叠加 |
| **K-β · 失败次数累计** | 某助手在该任务卡上连续失败 N 次 → 自动暂停升级 | 防无限重试 | N 阈值不好选 | 叠加 |
| **K-γ · 分歧检测** | 多助手对同一问题给出冲突结论 → 暂停等用户裁决 | 利用群协作天然信号 | 需要语义对比 | 替代部分场景 |
| **K-δ · 固定签到点** | 任务卡可设"必须用户确认"的检查点（如"测试通过后等用户")| 用户掌控关键节点 | 加重用户负担 | 叠加 |
| **K-ε · 智能通知策略** | 用户在线时实时推；离线时 buffer 进通知中心，用户上线后批量审 | 不打扰、可异步 | 离线期可能误事 | 叠加 |
| **K-ζ · 信任度机制** | 每个助手有"信任级别"，新装的助手默认低信任 → 任何动作都问；信任高了再放权 | 渐进式放权符合直觉 | 信任度怎么算需要设计 | 替代"风险等级"的助手维度 |
| **K-η · "重要决定"语义识别** | LLM 判断当前是否是"关键决定"，是则停下来问用户 | 智能、覆盖面广 | 易误判 | 叠加，作为补充信号 |

**推荐组合**：推荐方案（风险等级 + 权限模式）+ α（成本兜底）+ β（失败兜底）+ ε（智能通知）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · M（4-6 周，含 UI）

## 主题二 · 上下文与状态

### B · 上下文爆炸

#### 现象

群跑 30 轮后，每个助手 prompt 上下文塞满了无关消息，响应变慢、token 飙升、关键信息被淹没。

#### 根因

默认所有群消息都进所有助手的上下文。但实际上：
- 质检不需要看工程师写的每行代码
- 工程师 B 不需要看工程师 A 的内部调试 log

#### 现有应对（v0.1.0）

- ✅ **`injectGroupHistory` 开关**：加助手到群时可选择是否注入历史。
- ✅ 助手记忆按群隔离（跨群不污染）。
- ✅ "清空上下文" 按钮（v0.1.0 实现）。
- 🔵 群内消息仍是一锅炖。

#### 推荐方案

**第 1 步 · 消息分级**

每条消息打 type 标签：

```typescript
type MessageType =
  | 'chat'           // 自由讨论
  | 'task-update'    // 任务卡状态变更通知
  | 'file-changed'   // 文件变更广播
  | 'system'         // 平台系统消息
  | 'directive';     // 总控指令
```

**第 2 步 · 助手按级订阅**

助手配置中加订阅过滤：

```yaml
agent: 工程师A
subscriptions:
  - chat: own_mentions_only      # 只看 @ 自己的对话
  - task-update: own_tasks_only  # 只看跟自己相关的任务
  - file-changed: related_only   # 只看自己 related_files 里的变更
  - directive: from_coordinator  # 只看总控指令
```

**第 3 步 · 任务卡作为天然上下文边界**

助手读取上下文时，默认只看：
- 当前在做的任务卡（owner 或 reviewer）
- 该任务卡的 `related_files` 历史变更
- 总控指派该任务卡时的派单上下文

**第 4 步 · 自动滚动摘要**

群消息超过阈值（如 200 条）触发 compactor：
- 老消息按任务卡聚合摘要
- 摘要写入"群档案"，可检索但不进活跃上下文

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **B-α · 滚动 compact** | 上下文超阈值时自动让 LLM 把老消息压缩成摘要（参考 Claude Code）| 透明无感、广泛验证可行 | 摘要可能丢细节 | 替代"消息分级"的轻量版 |
| **B-β · 相关性打分** | 每条消息对当前助手做相关性打分，低于阈值不进上下文 | 精准 | 实时打分有成本 | 叠加 |
| **B-γ · RAG 检索注入** | 历史消息存向量库，按需检索相关片段进上下文 | 完全无遗忘 | 引入 RAG 基础设施 | 替代"全量注入" |
| **B-δ · 分阶段会话** | 长任务自动切多个 session，每段独立上下文，session 间通过任务卡传递关键信息 | 干净彻底 | 切换边界要设计好 | 替代"持续注入" |
| **B-ε · 助手自报感兴趣的标签** | 每个助手声明"我只关心 #frontend / #login 等标签的消息" | 助手自驱过滤 | 助手可能漏看 | 叠加到"按级订阅" |
| **B-ζ · 任务卡视图** | 助手只看与自己 owner/reviewer 任务卡相关的消息 | 边界清晰 | 跨任务上下文丢失 | 推荐方案的具体实现 |
| **B-η · 上下文窗口大小动态调** | 任务复杂度 high 给大窗口，low 给小窗口 | 与模型路由联动 | 调参麻烦 | 叠加 |
| **B-θ · "群档案"按需检索** | 长群保留完整原始记录但默认不注入；助手 @ 调用查档工具时才取 | 保留原始性 | 助手要主动想到查 | 叠加：长期归档机制 |

**推荐组合**：推荐方案（消息分级 + 订阅）+ α（滚动 compact 兜底）+ ζ（任务卡视图作主过滤）+ θ（长期可检索）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · L（大：6-10 周）

### E · 信息同步盲区（A 改了文件 B 不知道）

#### 现象

工程师 A 把 `utils.ts` 的某个函数签名改了，工程师 B 仍按旧签名写调用方代码 → 合并冲突或运行时报错。

#### 根因

群里"对话同步"和"工作目录同步"是两条独立通道。文件层的真实变更没有自动广播到对话层。

#### 现有应对（v0.1.0）

- 🔵 依赖工程师 A 在群里**手动**说"我改了 utils.ts"，全靠自觉。

#### 推荐方案

**第 1 步 · 文件变更事件总线**

任何助手的写文件操作（Edit/Write/Bash 涉及文件）都自动产生事件：

```typescript
interface FileChangedEvent {
  type: 'file-changed';
  path: string;
  changed_by: agentId;
  diff_summary: string;       // 自动生成的 diff 摘要
  full_diff_url: string;      // 完整 diff 可点查看
  timestamp: Date;
  task_card_id: string;
  message_id: string;
}
```

**第 2 步 · 反向订阅**

任务卡 `related_files` 字段做反向索引：
- 谁的任务卡含该文件，谁的助手就订阅该文件的变更事件
- 工程师 B 下次发言前自动看到 diff 摘要插入上下文头部

**第 3 步 · 提交前"知会一下"摘要**

工程师 A 完成任务卡进入 `in_review` 时，自动产生一条系统消息：

```
[file-changed] 工程师A 在 TC-001 中修改了：
  - src/utils.ts (+12 -8)：updateUser 签名变更
  - src/types.ts (+3 -0)：新增 UserOptions 类型
[查看完整 diff]
```

不 @ 任何人（避免触发扇出风暴），但订阅该路径的助手会在下一轮自动看到。

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **E-α · 助手任务前自动 git status** | 每个助手开始任务前必须先跑 `git status` + `git log -10`，了解最新变更 | 简单、习惯型 | 增加每次启动的 token 成本 | 叠加：作为助手"上岗仪式" |
| **E-β · 任务卡间显式依赖** | 任务卡可声明 `depends_on: [TC-001]`，依赖任务变更时自动推送通知 | 显式清晰、像 Make/Bazel | 用户得手动写依赖 | 叠加到"反向订阅" |
| **E-γ · 文件 watcher 广播** | 监听 work_dir 文件变更，立即作为系统消息广播到群里 | 实时、无遗漏 | 大量变更会刷屏 | 替代"事件总线" |
| **E-δ · 状态切换附 diff 摘要** | 任务卡 in_progress→in_review 时自动产出 diff，所有相关助手强制阅读 | 与状态机绑定 | 仅在状态切换时同步 | 叠加 |
| **E-ε · repo map 自动维护**（参考 Aider） | 维护一份项目结构 + 关键符号摘要，每次注入助手上下文 | 解决"不知道项目长啥样" | 大项目摘要太长 | 叠加 |
| **E-ζ · 助手发言前"同步检查"** | 每次发言前看一眼最近 N 个 file-changed 事件，把摘要拼到上下文头部 | 自动、无感 | 每次发言成本+1 | 推荐方案的实现细节 |
| **E-η · 共享变更日志（CHANGELOG.md）** | 平台维护一份机器写的变更日志，所有助手共读 | 像人类团队的 standup 笔记 | 需要规范写法 | 叠加 |
| **E-θ · 跨助手"广播"协议** | 工程师交付时调用 `broadcast(diff_summary)`，订阅者下次看 | 与推荐方案等价 | 实现已经包含 | 推荐方案的别名 |

**推荐组合**：推荐方案 + α（git status 上岗）+ β（显式依赖）+ ε（repo map）。这四件叠在一起 ≈ 完整的"看见队友在干什么"。

#### 优先级 · 🔴 高（P0）
#### 工作量 · M（4-6 周）

### G · 任务卡状态漂移

#### 现象

- 任务卡显示 `done`，但代码根本没跑测试 / 文件没改
- 反过来：实际做完了，状态卡在 `in_progress` 没人切

#### 根因

状态字段是助手**自己声称**的，没有客观核验。LLM 倾向于"乐观汇报"。

#### 现有应对（v0.1.0）

- 🟡 任务看板有 6 状态分类，但状态切换无强约束。
- 🔵 验收依赖质检助手 LLM 判断，可被表象骗过。

#### 推荐方案

**核心思路**：让 `expected_output` 字段从自然语言升级为**可机器校验**的断言。

**第 1 步 · 任务卡 schema 增加 verifications 字段**

```yaml
task_card:
  expected_output:               # 自然语言描述（给人看）
    - 文件 src/Login.tsx 包含 SSOButton
    - 测试 login.test.ts 全部通过
  verifications:                 # 机器校验（给平台跑）
    - type: file_contains
      path: src/Login.tsx
      pattern: "SSOButton"
    - type: command_passes
      command: npm test login.test.ts
    - type: screenshot_matches
      url: http://localhost:3000/login
      reference: docs/refs/login-with-sso.png
      threshold: 0.95
```

**第 2 步 · 客观验收钩子**

`in_progress → in_review` 切换时，平台层先跑所有 verifications：

```typescript
async function tryTransitionToReview(card: TaskCard) {
  const results = await runVerifications(card.verifications);
  if (results.every(r => r.passed)) {
    card.status = 'in_review';
    card.evidence = results;        // 证据落卡
    notifyReviewer(card);
  } else {
    notifyOwner(card, 'VERIFICATION_FAILED', results);
    // 拒绝状态切换，owner 必须修复
  }
}
```

**第 3 步 · 状态切换附带证据**

每次状态切换必须附 evidence：

```yaml
history:
  - time: 2026-05-09T14:00:00Z
    actor: 工程师A
    event: in_progress→in_review
    evidence:
      verifications_passed: 3/3
      diff_hash: abc123
      test_log_url: ...
      screenshot_url: ...
```

不附 evidence 的切换被消息总线拒绝。

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **G-α · 双盲质检** | 质检不读工程师"我做了什么"的陈述，只读最终产物（diff/截图/测试 log）| 防被陈述误导 | 增加质检成本 | 叠加 |
| **G-β · 多质检对赌** | 同一任务卡跑 2 个独立质检，意见一致才算通过 | 高质量场景必须 | 成本翻倍 | 仅适合 high 复杂度任务 |
| **G-γ · 用户最终验收** | 高风险任务卡必须用户拍板才能 done | 100% 兜底 | 用户负担重 | 叠加，仅高风险 |
| **G-δ · 历史可信度调整** | 助手历史误报率高 → 后续要求附更详细证据 | 自适应、长期更准 | 冷启动数据不足 | 叠加 |
| **G-ε · 测试覆盖率门槛** | expected_output 要求"新代码必须有 X% 测试覆盖" | 强制工程标准 | 仅适合代码任务 | 叠加，作为 verification 之一 |
| **G-ζ · 提交前自检 checklist** | 助手交付前必须自答一组检查问题（"测试跑了吗""覆盖率达标吗"） | 让模型 reasoning 先过一遍 | 助手可能照本宣科 | 叠加 |
| **G-η · 客观验收脚本由总控起草** | 总控派单时自动生成 verifications，工程师只能加不能减 | 减轻用户写 verifications 负担 | 总控自动起草准确度有限 | 推荐方案的"启动加速器" |
| **G-θ · 二次回归** | 任务 done 后过 N 小时再跑一次同样的 verifications | 防"刚交付时碰巧过"| 慢 | 叠加 |
| **G-ι · 行为录像作证据**（参考 Devin）| 关键步骤截屏/录终端 session，存进 evidence | 可视化强、回放方便 | 存储成本 | 叠加，long-term |

**推荐组合**：推荐方案（verifications + evidence）+ α（双盲）+ η（自动起草 verifications）。η 解决"用户不会写 verifications"这个真实痛点。

#### 优先级 · 🔴 高（P0）
#### 工作量 · M（4-6 周）

---

### I · 长任务中断恢复

#### 现象

任务跑到一半（机器睡眠 / 网络中断 / token 用尽），第二天恢复，助手对"我们刚才进行到哪了"完全没概念。

#### 根因

助手的"工作记忆"主要在对话上下文里，对话上下文是**易失**的——群消息可以持久化，但当时的 chain-of-thought、临时变量、未保存的判断不一定能复原。

#### 现有应对（v0.1.0）

- ✅ 群消息 + 任务看板持久化。
- ✅ 任务看板有 "待恢复的任务" 列。
- ✅ 助手有 `streamingThinking` 流式思考链可视化。
- 🔵 中断恢复机制未明确实现。

#### 推荐方案

**第 1 步 · 任务卡内部加 `step_progress`**

```yaml
task_card:
  steps:
    - 在 /login 加 OAuth 按钮组件          # 已完成
    - 接通 /api/oauth/callback             # 进行中
    - 加单元测试                            # 未开始
  step_progress:
    current_step: 2
    sub_progress: "已实现 callback 路由，正在写参数校验"
    last_updated: 2026-05-09T14:30:00Z
```

**第 2 步 · 决策日志**

重要分叉点助手必须记录"为什么走 A 不走 B"：

```yaml
decisions:
  - time: 2026-05-09T14:00:00Z
    question: 用 OAuth 库还是手写
    chosen: 用 next-auth
    reason: 项目已经有 next-auth 依赖，避免引入新库
    alternatives_rejected:
      - {option: 手写, reason: 增加维护成本}
```

恢复时直接读决策日志，避免换个模型实例就重选路径。

**第 3 步 · 任务卡 ↔ git 分支绑定**

任务卡进 `in_progress` 自动开分支：

```
task-card/TC-001
```

助手所有写文件操作落在该分支，每完成一个 step 自动 commit：

```
[TC-001 step 1] 在 /login 加 OAuth 按钮组件
[TC-001 step 2] 接通 callback 路由（进行中）
```

中断恢复 = 切回分支看最后一次 commit。同时解决问题 D（文件并发冲突）和问题 G（状态漂移，commit hash 即证据）。

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **I-α · 定期 snapshot** | 每 N 分钟把任务卡完整状态（含上下文窗口）序列化落盘 | 任意点恢复 | 存储成本、敏感信息持久化 | 叠加 |
| **I-β · 助手"自检"机制** | 中断后恢复时，助手先读完 history + decisions 后说一句"我准备从 step 3 继续"，等用户确认 | 防瞎接 | 用户每次都要 ack | 叠加，作为安全门 |
| **I-γ · 行为录像**（参考 Devin）| session 全程录终端 / 截屏，恢复时回放给用户看 | 用户体验好 | 存储巨大、隐私风险 | 替代"决策日志"的可视化版 |
| **I-δ · LangGraph checkpointer** | 引入显式 state machine + checkpointer 库 | 工业级方案 | 引入框架依赖 | 替代 git 分支策略的工程化方案 |
| **I-ε · 增量持久化** | 每条工具调用结果立即落库，不等 N 分钟 | 颗粒度细、丢失最少 | I/O 频繁 | 叠加 α |
| **I-ζ · 任务卡断点 = 当前 in_progress 步骤** | 不存"recoverable point"，直接以 `step_progress` 为断点 | 简单 | 步骤内中断恢复粗糙 | 推荐方案的细化 |
| **I-η · 让助手在恢复时主动 git diff** | 恢复后助手先 `git diff` 看自己改了啥，决定继续还是回滚 | 利用 git 自然能力 | 助手要主动想到 | 叠加，与 D/G 共用 git 基础设施 |
| **I-θ · 决策日志强制结构化** | 重要分叉点助手必须输出固定格式的 JSON 决策记录 | 机器可解析、恢复时直接读取 | 助手可能输出不规范 | 推荐方案的 schema 化 |

**推荐组合**：推荐方案（step_progress + 决策日志 + git 分支）+ β（自检 ack）+ η（git diff 唤醒）。

#### 优先级 · 🟡 中（P1，但与 D/G 联动建议合并实施）
#### 工作量 · M（4-6 周）

## 主题三 · 边界与冲突

### D · 文件并发冲突

#### 现象

工程师 A 和工程师 B 同时改 `Login.tsx`，A 改样式 B 改逻辑，后写的覆盖先写的。Git 上看到的是一次"看似干净"的提交，实际丢了 A 的改动。

#### 根因

群即项目共享一个 `work_dir`，但平台没有对**写文件操作**做并发控制。LLM 助手不会主动加锁，也不会主动 git pull/rebase。

#### 现有应对（v0.1.0）

- ✅ 三层 workDir 策略已就位（群级共享、助手默认、快速对话独立）。
- 🔵 共享群目录内无并发控制。

#### 推荐方案

**第 1 步 · 文件级悲观锁**

任务卡 `related_files` 字段升级为锁定表：

```yaml
task_card:
  status: in_progress
  related_files:
    - path: src/Login.tsx
      mode: write           # write 锁，独占
    - path: src/types.ts
      mode: read            # read 锁，可多读
```

派单时平台检测：
- 已有 `write` 锁 → 拒绝派单（提示总控冲突）或排队
- 已有 `read` 锁 + 新 `write` 锁 → 等读释放
- 多个 `read` 锁可共存

**第 2 步 · 任务卡绑 git 分支（与问题 I 共用方案）**

每张任务卡一个分支，物理隔离写操作：

```
main
├── task-card/TC-001 （工程师A 改 Login.tsx）
└── task-card/TC-002 （工程师B 改 Login.tsx）
```

分支合并由总控/质检审 conflict，避免静默覆盖。

**第 3 步 · 派单时冲突预检**

总控派单算法：
- 检测两张任务卡 `related_files` 重叠
- 重叠则强制串行（先做 TC-001 再做 TC-002）
- 不重叠才并行

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **D-α · 乐观锁 + 自动 rebase** | 每次 write 前比对版本，冲突时自动让助手重新生成基于新版本的修改 | 不阻塞、并发度高 | "重新生成"消耗 token | 替代悲观锁 |
| **D-β · 派单时强制串行** | 总控派单算法检测同一文件 → 强制串行执行 | 实现简单（1 周）、零冲突 | 失去并行能力 | 与悲观锁等价的简化版 |
| **D-γ · 分文件夹分工** | 按目录树切分，每个工程师独占一个子目录 | 物理隔离、简单 | 跨模块改动尴尬 | 替代锁机制 |
| **D-δ · CRDT/OT 实时合并**（参考 VS Code Live Share）| 工程师改动实时合并、字符级冲突自动解决 | 真正的并发协作 | 工程量极大 | 长期方案 |
| **D-ε · Pre-write hook 自动 git pull/rebase** | 每次 write 前先 rebase main，冲突由助手解决 | 借力 git 工具链 | rebase 失败需要人介入 | 叠加 |
| **D-ζ · 影子目录写入** | 助手写到独立的 sandbox 目录，定期由总控合并到主目录 | 主目录绝对干净 | 多一层目录映射 | 与 git 分支等价的 fs 版 |
| **D-η · 文件级"占用 token"** | 助手写文件前必须 acquire token，同 file 的 token 只有一份 | 通用机制、不限 git 项目 | 实现简单但缺少 git 历史 | 替代 git 分支（适合非代码工作目录）|
| **D-θ · 派单时按依赖图拓扑排序** | 任务卡间依赖建图，自动决定先后顺序 | 与 H-α 联动 | 用户要写依赖 | 叠加 β |

**推荐组合**：推荐方案（悲观锁 + git 分支）+ β（强制串行兜底）+ ε（pre-write rebase）。代码项目用 git 分支，文档/资料类项目用 η（文件 token）。

#### 优先级 · 🔴 高（P0）
#### 工作量 · M（4-6 周）

### F · 角色越界

#### 现象

工程师跑去做"批判式 review" / 质检直接动手改代码 / 总控被某细节带着走亲自实现。

#### 根因

角色边界写在系统提示词里——**软约束**。LLM 看到"这事我能做"就倾向于做。

#### 现有应对（v0.1.0）

- 🟡 助手系统提示词约束角色定位。
- 🔵 没有工具白名单层。

#### 推荐方案

**核心原则：能力 ≠ 权限，能力即权限。**

**第 1 步 · 工具白名单（参考 CrewAI）**

每个助手配置加 `tools` 白名单：

```yaml
agent:
  name: 总控
  tools: [
    create_task_card,
    assign_task,
    read_task_status,
    summarize,
    mention,
  ]
  # 没有 write_file，物理上无法改代码

agent:
  name: 质检
  tools: [
    read_file,
    run_test,
    compare_screenshot,
    review_diff,
    reject_task,
    approve_task,
  ]
  # 没有 write_file，无法改代码

agent:
  name: 工程师
  tools: [
    read_file, write_file, edit_file,
    run_command,
    update_task_status,
  ]
  # 没有 create_task_card，无法擅自加任务
```

**第 2 步 · 输出 schema 锁死（参考 MetaGPT）**

任务卡的 `expected_output` 用结构化 schema：

```yaml
expected_output:
  schema:
    type: code_diff
    files: [src/Login.tsx]
  # 工程师必须输出 diff，输出"评审意见"会被拒
```

**第 3 步 · 越界检测**

助手输出包含其角色 schema 之外的内容时，平台层自动拒收并提示：

```
[REJECTED] 工程师A 试图输出"评审意见"，
但当前任务卡 expected_output.schema=code_diff。
请先完成代码改动，再让质检 review。
```

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **F-α · 输出 schema 校验**（参考 CrewAI）| 任务卡 expected_output 用 pydantic-like schema，输出格式不对就拒收 | 强约束、机器可校验 | 自然语言任务难写 schema | 叠加到推荐方案 |
| **F-β · 后置审查 LLM** | 助手发言后再过一层"角色合规"小模型检查 | 兜底强 | 多一份成本、慢 | 叠加 |
| **F-γ · 动态权限** | 助手只在被 @ 时启用对应工具，平时工具不可用 | 减少越界机会 | 实现复杂 | 替代静态白名单 |
| **F-δ · 简化助手原则** | 一个助手只装最少必需工具，避免"能做的事太多"诱发越界 | 设计原则、零成本 | 用户要懂 | 叠加，作为最佳实践引导 |
| **F-ε · 工具调用前自检 prompt** | 每次工具调用前模型自答"这个动作在我的角色范围内吗" | 让 reasoning 帮把关 | 模型可能强词夺理 | 叠加 |
| **F-ζ · 角色违规惩罚机制** | 助手违规直接禁用 N 分钟，让用户感知 | 强反馈 | 影响体验 | 仅适合开发期 |
| **F-η · 总控集中下发"任务卡式权限"** | 工具白名单不只看角色，还看当前任务卡——任务卡才是权限载体 | 最精细、最安全 | 总控复杂 | 推荐方案的进阶版 |
| **F-θ · 输出类型分类器** | 训一个小分类器判断"这是 review 输出 / code 输出 / 决策输出"，与角色不符就拒 | 客观 | 训练数据需积累 | 叠加 β 的工程化版本 |

**推荐组合**：推荐方案（工具白名单）+ α（schema 校验）+ δ（简化助手原则）+ η（任务卡式权限）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · M（4-6 周）

### L · 群规则失效

#### 现象

群规则写得很全，但跑几轮就开始有人不遵守；规则之间互相冲突时，助手随机选一边。

#### 根因

群规则注入到每个助手系统提示词里——**自律**而非**他律**。LLM 在长上下文下注意力衰减；规则冲突时模型自己仲裁，不一致。

#### 现有应对（v0.1.0）

- ✅ 群规则注入到所有助手上下文。
- 🔵 全是软约束，无强制层。

#### 推荐方案

**核心思路**：群规则分两层——**自律层**（提示词）和**他律层**（消息总线钩子）。

**第 1 步 · 区分两层**

| 层 | 类型 | 例子 | 实现 |
|----|------|------|------|
| **自律层** | 鼓励性、风格类 | "回复简洁，不要闲聊"、"用中文交流" | 注入提示词 |
| **他律层** | 强制性、结构类 | "完成型回复禁止艾特"、"消息必须带任务卡 id" | 消息总线 hook |

**第 2 步 · he 律层 schema**

```yaml
chat_room:
  rules:
    self_discipline:        # 自律层
      - 回复简洁
      - 用中文
    he_discipline:          # 他律层
      - id: must_have_task_id
        when: pre_message_send
        match:
          speaker_role: [MEMBER]
          speaker_level: normal     # system 助手豁免
        check: "msg.contains_task_id || msg.is_meta"
        on_fail:
          action: reject
          hint: "回复必须带任务卡 id（除元讨论外）"
      - id: no_at_on_done
        when: pre_message_send
        match:
          content_pattern: "(完成|已交付|搞定)"
        check: "msg.mentions.length === 0"
        on_fail:
          action: reject
          hint: "完成型回复请走任务卡状态变更"
```

**第 3 步 · 优先级与冲突解决**

每条 he 律层规则带 `priority` 字段，冲突时高优先胜出。

**第 4 步 · 违规计数器**

连续违规的助手自动降级：

```yaml
agent_violations:
  agent_id: 工程师A
  count_30d: 5
  status: probation         # 暂时不被派单
```

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **L-α · 违规直接拒收 + 重生成** | 首次违规直接 block 输出，要求助手重写 | 立即纠偏 | 增加成本 | 推荐方案的强化 |
| **L-β · 规则前置 + 后置双校验** | 既在 pre_message_send 拦截，也在 post_message_send 抽样审查 | 双层保险 | 慢 | 叠加 |
| **L-γ · 把规则做成 skill** | 关键规则封装成 skill 注入到所有助手，比系统提示词更前置 | 复用性好 | 仍是软约束 | 替代部分自律层规则 |
| **L-δ · 用户随时"提醒注入"** | UI 一键把"提醒：本群规则 X" 插入下条消息 | 简单兜底 | 用户要主动 | 叠加 |
| **L-ε · 微调小模型固化规则** | 对核心规则做小规模 SFT，让模型从根上记住 | 规则深入模型 | 训练成本高、不通用 | 长期 |
| **L-ζ · 规则可观测性** | UI 显示"本轮触发了哪些规则、谁违规了多少次" | 让用户看到规则在生效 | 需要 UI 工作量 | 叠加 |
| **L-η · 规则优先级 + 冲突解决树** | 规则间有显式 priority，冲突时高优先胜出 | 推荐方案的核心 | 用户要理解 | 推荐方案的子模块 |
| **L-θ · 默认规则模板库** | 内置一组验证过的规则模板（防扇出 / 防越界 / 防多语等），用户勾选启用 | 用户零负担 | 模板覆盖有限 | 推荐方案的入门级 |

**推荐组合**：推荐方案（自律层 + 他律层）+ α（违规拒收）+ θ（默认模板）+ ζ（可观测性）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · M（4-6 周）

## 主题四 · 质量与成本

### J · 模型能力错配

#### 现象

- 给 Haiku 派"评审复杂架构"——输出浅显，漏掉关键问题
- 给 Opus 派"改一行 typo"——钱白花，还慢
- 同一助手既写复杂业务逻辑又改 typo，固定模型左右不是

#### 根因

助手层目前是 **`assistant ↔ model` 一对一**绑定，模型选择**人为静态决定**。但任务难度是**动态**的。

#### 现有应对（v0.1.0）

- ✅ 助手可手动选模型。
- ✅ 默认模板按角色给出推荐模型。
- 🔵 无任务级动态路由。

#### 推荐方案

**第 1 步 · 助手 → 模型策略而非一对一**

```yaml
agent:
  name: 工程师A
  model_strategy:
    primary: claude-sonnet-4-6
    fallback_simple: claude-haiku-4-5     # 简单任务降级
    fallback_complex: claude-opus-4-7     # 复杂任务升级
    decision_rule: by_task_complexity
```

**第 2 步 · 任务卡 complexity 字段**

```yaml
task_card:
  complexity: medium       # low / medium / high
  # 由总控估，或由 LLM 根据 steps + expected_output 自动估
```

**第 3 步 · 路由规则**

```typescript
function chooseModel(agent, taskCard) {
  switch (taskCard.complexity) {
    case 'low':  return agent.model_strategy.fallback_simple;
    case 'high': return agent.model_strategy.fallback_complex;
    default:     return agent.model_strategy.primary;
  }
}
```

**第 4 步 · 成本看板**

```
群: 我的博客重构
本月 token 消耗: 1.2M  ($14.50)
按助手:
  - 工程师A: 850K ($9.20)
  - 总控:    200K ($3.50)
  - 质检:    150K ($1.80)
按任务卡:
  - TC-001: 400K ($5.00) ← 最贵
  - TC-002: 200K ($2.00)
按模型:
  - opus-4-7:   30% ($10.50) ← 占比最高
  - sonnet-4-6: 60% ($4.00)
  - haiku-4-5:  10% ($0.10)
```

让用户能看到"钱花在哪"，决定是否调整模型策略。

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **J-α · 历史成功率自适应** | 记录每个模型在各类任务上的成功率，自动选最优 | 长期看效果好 | 冷启动慢、需要数据 | 替代静态 complexity 路由 |
| **J-β · 成本预算路由** | 任务卡设"成本上限 $X"，超了自动降级到便宜模型 | 防烧钱 | 可能交付质量打折 | 叠加 |
| **J-γ · cascading（先小后大）** | 先用便宜模型试，质检不过则升级到贵的重做 | 自动平衡成本与质量 | 重做翻倍成本 | 替代复杂度路由 |
| **J-δ · A/B 测试模式** | 同一任务并行跑两个模型，选好的输出 | 输出质量最优 | 成本翻倍 | 仅适合关键任务 |
| **J-ε · 用户在任务卡级手动指定** | 任务卡可显式指定模型，覆盖助手默认策略 | 最精细控制 | 用户负担 | 叠加 |
| **J-ζ · 思考 / 执行双模式** | 复杂规划用大模型，机械执行用小模型（参考 Claude Code plan/execute、Aider architect/editor）| 与产品形态契合 | 助手要分两种调用 | 推荐方案的具体实现 |
| **J-η · 上下文长度路由** | 上下文超 N tokens 自动换长上下文模型 | 解决"上下文不够" | 中间换模型可能产生不一致 | 叠加 |
| **J-θ · 用户预设"成本档位"** | 群级别选"经济 / 平衡 / 性能" 三档，自动设全套模型策略 | 零配置 | 自由度低 | 包装层 |
| **J-ι · 模型熔断** | 某 provider 连续失败 N 次自动切到 fallback provider | 防 provider 抽风 | 需要监控 | 叠加 |

**推荐组合**：推荐方案（complexity 路由 + 成本看板）+ ζ（思考/执行双模）+ β（预算上限）+ ι（熔断）+ θ（一键档位预设给小白用户）。

#### 优先级 · 🟡 中（P1）
#### 工作量 · M（4-6 周，路由 + 看板分别 2-3 周）

### M · 评估盲区

#### 现象

任务卡走完 `done`，群里看似一切顺利，但用户实际打开应用——按钮歪了、测试只覆盖了顺利路径、文档跟代码不一致。**质检"通过"不等于"真的通过"**。

#### 根因

1. 验收标准是**自然语言**，由质检助手用 LLM 判断"是否满足"——主观，易被表象骗。
2. 没有跨任务卡的**回归视角**——这次改 A 模块会不会把之前 B 模块跑通的测试搞挂。
3. 没有"质量历史"——这个工程师助手过去交付的 bug 率多少、这个质检过去漏判率多少，全没记录。

#### 现有应对（v0.1.0）

- 🟡 质检 LLM 复核。
- 🟡 任务卡 `expected_output` 写得越具体越好。
- 🔵 无客观度量层。

#### 推荐方案

**第 1 步 · 三级验收**

| 级别 | 实施 | 优先级 |
|------|------|--------|
| **级别 1 · 客观验收** | 跑脚本/测试/校验 hash/截图比对 | 能客观就不主观 |
| **级别 2 · 质检 LLM 复核** | 当前已有的方式 | 兜底 |
| **级别 3 · 用户最终拍板** | 高风险任务由用户最后确认 | 仅高风险 |

实现：第 G 节的 `verifications` 字段就是级别 1 的载体。

**第 2 步 · 回归套件**

每张完成的任务卡产出一组"该任务卡作为后续不能破坏的契约"测试：

```yaml
task_card:
  id: TC-001
  status: done
  regression_tests:
    - npm test login.test.ts
    - npm test e2e/login-sso.spec.ts
```

新任务卡进 `in_review` 时**自动跑全套已 done 任务卡的回归测试**，挂了就退回。

**第 3 步 · 质量看板**

每个助手的长期表现：

```
工程师A 最近 30 天：
  交付任务卡数: 45
  一次通过率: 67%
  平均重做轮次: 1.4
  平均 token / 任务: 32K
  平均耗时: 18min

质检 最近 30 天：
  审查任务卡数: 60
  通过率: 70%
  事后被发现漏判数: 3
  平均审查耗时: 4min
```

用户基于真实数据调整团队配置。

**第 4 步 · 与 LangSmith / Langfuse 类似的 trace 平台对接（可选）**

如果用户想要更专业的可观测性，开放 OTLP 导出，对接外部 trace 平台。

#### 备选 / 补充方案

| 编号 | 思路 | 优点 | 缺点 | 关系 |
|------|------|------|------|------|
| **M-α · 用户星级标注** | 每次交付用户打 1-5 星，长期数据驱动模型/助手优化 | 简单、用户友好 | 用户可能不打 | 叠加，数据基础 |
| **M-β · 第三方 LLM 复评** | 用不同家的模型独立 review（Claude 写、GPT 复审）| 减少同模型偏见 | 多花钱 | 替代单一质检 |
| **M-γ · 自动测试生成** | AI 帮用户基于 expected_output 生成测试用例 | 解决"用户不会写测试" | 生成的测试可能不严 | 叠加，作为冷启动加速 |
| **M-δ · 基于 trace 的事后剖析** | 任务完成后自动跑诊断（哪一步浪费 token / 哪个工具失败多）| 持续优化 | 需要 trace 基础设施 | 叠加，与 OTLP 同方向 |
| **M-ε · 竞争式验收** | 多个独立质检打分，分歧大时自动升级用户 | 利用多视角 | 成本高 | G-β 在 M 维度的延伸 |
| **M-ζ · 错题本（mistake notebook）** | 每次失败都记一笔（什么任务、什么模型、什么错、怎么修），下次类似任务先查 | 持续学习 | 需要分类与检索 | 叠加 |
| **M-η · 黄金集回归** | 用户标注一组"标准答案"任务，每次平台升级时跑回归对比 | 防"升级反而退化" | 维护成本 | 平台开发期内部用 |
| **M-θ · 跨群质量对比** | 同一类任务在不同群不同助手的表现对比 | 发现配置最佳实践 | 需要数据聚合 | 叠加 |
| **M-ι · 实时输出打分** | 助手每条输出都过一个 reward model 实时打分，分太低拒发 | 防低质量泄露给用户 | reward model 训练成本 | 长期方案 |
| **M-κ · 用户审计模式** | 用户可按任务卡 / 助手 / 时间段过滤，导出"质量审计报告" | 适合外部交付场景 | 主要是 UI 工作 | 叠加 |

**推荐组合**：推荐方案（三级验收 + 回归套件 + 质量看板）+ α（用户标注）+ ζ（错题本）+ γ（自动生成测试）。第三方 LLM 复评（β）适合关键交付场景按需启用。

#### 优先级 · 🟢 低（P2，长期投入）
#### 工作量 · L（>10 周，看板 + 回归套件 + trace 各占一块）

## 联动建议（一鱼多吃）

实施时这几组方案应**捆绑**做：

### 联动 1 · "硬化任务卡协作"（解决 D / E / G / I）

核心改造：**任务卡 ↔ git 分支强绑定 + verifications 字段 + 文件变更事件总线**

一次改造同时解决：
- D 文件并发冲突（每张卡独立分支）
- E 信息同步盲区（文件变更事件流）
- G 状态漂移（commit hash 作为 evidence）
- I 中断恢复（git 分支天然可恢复）

### 联动 2 · "对话状态分离"（解决 A2 / L）

核心改造：**对话通道 vs 任务事件通道分离 + he 律层 hooks**

一次改造同时解决：
- A2 扇出风暴（状态变更不走对话）
- L 群规则失效（hooks 强制结构性规则）

### 联动 3 · "能力即权限"（解决 F / K）

核心改造：**工具白名单 + 风险等级**

一次改造同时解决：
- F 角色越界（没装就没法做）
- K 人类介入时机（high 风险动作前置确认）

### 联动 4 · "可观测协作"（解决 J / M）

核心改造：**任务卡 complexity 字段 + 成本看板 + 质量看板**

延展能力，长期积累用户决策的数据基础。
