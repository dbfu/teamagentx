# 语音功能需求文档

## 1. 背景

TeamAgentX 已支持多助手群聊、语音消息附件和浏览器侧基础语音播报配置，但当前实现仍然偏功能拼接：

- 语音消息、语音播报、语音识别分散在不同模块
- 浏览器本地 TTS 直接暴露给业务层
- 助手语音配置只适合本地朗读参数，难以承载远程 TTS 或语音大模型

这会带来两个直接问题：

- 本地浏览器 TTS 自然度有限，无法成为长期主方案
- 如果以后接远程 TTS、STT 或实时语音模型，现有业务层要大改

因此本阶段的核心目标不是继续堆功能，而是先建立统一语音抽象层。

## 2. 目标

第一阶段目标：

- 将语音能力抽象为独立 domain，统一承载 `tts`、`stt`、`realtime-chat`
- 保证业务层不直接依赖浏览器 TTS / STT API 或某家远程语音 SDK
- 保留并收编现有语音消息、浏览器播报、录音能力
- 为后续接入远程 TTS / STT 和语音大模型预留稳定扩展点
- 尽量减少对当前分支已完成功能的推倒重来

## 3. 非目标

以下能力不属于第一阶段交付范围：

- 完整实时语音通话产品
- 复杂 VAD / 打断 / 双工语音状态机
- provider 管理后台与动态注册系统
- 移动端完整语音运行时实现
- 一次性接入多家语音供应商

## 4. 用户故事

### 4.1 助手播报

作为群聊用户，我希望助手回复的语音播报未来可以切换不同引擎和音色，而不用每次改业务逻辑。

### 4.2 语音输入

作为群聊用户，我希望录音输入与转写能力未来既可以使用本地方案，也可以平滑切换到远程方案。

### 4.3 角色语音人格

作为群主或配置者，我希望每个助手的语音配置不只包含 `voiceId`、`speed`，还可以逐步扩展到情绪、风格、提示词和供应商特定参数。

### 4.4 语音大模型扩展

作为研发者，我希望以后接入语音大模型时，只新增 provider，而不是重写消息流、播放器、配置结构和任务调度。

## 5. 约束

- 必须兼容现有语音消息附件结构
- 必须允许前端本地 provider 与服务端远程 provider 并存
- 第一阶段 provider 注册机制使用代码内置，不引入额外动态配置平台
- 不能把领域接口写死成某一家厂商 SDK 透传
- 不能为了未来能力过度拆表或引入复杂控制面

## 6. 验收标准

### 6.1 抽象边界

1. While 业务需要调用语音能力，when 发起 TTS / STT / realtime-chat 请求，the TeamAgentX system shall 通过统一语音服务入口发起，而不是直接调用具体浏览器 API 或第三方 SDK。
2. While 新增一个语音 provider，when 实现 provider 接口并注册后，the TeamAgentX system shall 不需要改动上层消息流语义。

### 6.2 兼容现有能力

1. While 当前浏览器本地 TTS 仍然存在，when 抽象层落地后，the TeamAgentX system shall 继续支持其作为 `browser-local` provider 使用。
2. While 现有语音消息附件已存在，when 抽象层落地后，the TeamAgentX system shall 继续支持上传、展示、播放与回看。

### 6.3 配置扩展性

1. While 用户配置助手语音，when 保存语音配置，the TeamAgentX system shall 支持通用 `SpeechProfile` 字段，而不是只支持浏览器朗读参数。
2. While 某个 provider 需要额外参数，when 系统保存配置，the TeamAgentX system shall 允许通过 `vendorOptions` 承载扩展字段而不破坏通用结构。

## 7. 成功指标

- 浏览器本地 TTS / STT 被成功纳入统一抽象层
- 至少能明确接入一个远程 provider 的技术路径，且无需推翻领域模型
- 助手语音配置从 `voiceConfig` 平滑升级到 `speechProfile`
- 新增 provider 时，改动集中在 provider 实现和注册层

## 8. 风险与开放问题

- 本地 provider 运行在前端，远程 provider 运行在服务端，统一语义需要清晰分层
- `voiceConfig` 到 `speechProfile` 的兼容策略需要谨慎，避免影响已保存配置
- 未来 `realtime-chat` 与当前消息模型未必完全一致，需要先保留会话接口，不急着落完整产品
- 远程 TTS 的缓存、费用和失败重试策略应在第二阶段细化
