# 01 · 产品概览：定位与核心架构

## 1. 一句话定义

**TeamAgentX** 把"开发团队"做成一个能拉进群、能听懂 @、按任务卡干活的 **AI 群聊**——让一个开发者一个人就能调度一支多角色协作的虚拟团队。

## 2. 目标用户

**单兵作战的开发者 / 极客**。

典型画像：
- 一个人维护多个项目
- 已经在用 Claude Code、Cursor、Aider 等单 Agent 工具
- 已经感受到"单 Agent 干不完一件复杂事"的边界——任务里既要写代码、又要审代码、又要跑测试、又要查资料
- 单线程来回切换的代价：要么爆 token、要么丢线索、要么角色错位（让会写代码的去做"批判式 review"效果总打折）

## 3. 核心隐喻：群聊 ≠ 工作流 ≠ 单 Agent

| 形态 | 比喻 | 灵活度 | 上手成本 | 代表产品 |
|------|------|--------|----------|---------|
| 工作流型 | 流水线 | 低（DAG 固定） | 中 | n8n / Dify / Coze |
| 单 Agent | 超级员工 | 高 | 低 | Cursor / Aider / Cline |
| 多 Agent 框架 | 写程序定义流程 | 高 | 高（写代码） | CrewAI / AutoGen / MetaGPT |
| **群聊型（本平台）** | **微信群指挥同事** | **高** | **极低（人人会用群）** | **TeamAgentX** |

**关键判断**：把"叫工程师写一下"、"@ 质检看一下"、"@ 总控拆个任务"做成第一类交互，复用每个开发者已有的"群协作肌肉记忆"。

## 4. 与现有形态的本质区别

| 形态 | 调度方式 | 协作粒度 | 上下文管理 | 适合场景 |
|------|---------|---------|-----------|---------|
| IDE 插件型 | 编辑器内单 Agent | 文件/函数 | 单线程 | 写代码 |
| 工作流型 | 预编排 DAG | 节点 | 节点间显式传参 | 重复性流程 |
| 单 Agent 自主型 | Agent 自规划 | 任务 | Agent 自管 | 中等复杂度独立任务 |
| 多 Agent 框架 | 代码定义角色 | 角色 | 框架内传递 | 工程师可定制场景 |
| **群聊型（本平台）** | **群规则 + @ 触发** | **任务卡** | **群即项目，三层隔离** | **个人开发者的复杂多步任务** |

差异化三件套：**任务卡**（做什么）+ **群隐喻**（怎么调度）+ **上下文隔离**（在哪做）。

## 5. 为什么不直接用 ChatGPT 群聊？

通用 AI 聊天产品（ChatGPT、豆包、Kimi 群聊等）也能"拉几个机器人进群讨论"，但本质是**闲聊场**而不是**干活场**。差距不在"能不能多 Agent 对话"，而在以下七件事：

| 能力 | 通用 AI 群聊 | TeamAgentX |
|------|-------------|-----------|
| 持久工作目录 | ❌ 每次对话即焚 | ✅ 群即项目，文件落地 |
| 任务卡机制 | ❌ 全靠自然语言 | ✅ 结构化、可追踪、可验收 |
| 角色边界强制 | ❌ 角色靠提示词，越界无人管 | ✅ 群规则 + 任务卡边界 |
| 技能（Skill）装载 | ❌ 仅人格设定 | ✅ 助手可独立装 skill |
| 单助手单模型 | ❌ 整群一个模型 | ✅ 每个助手独立选模型 |
| 阻塞 / 人类介入 | ❌ 没有上报通道 | ✅ 阻塞上报机制 |
| 上下文隔离 | ❌ 一锅炖 | ✅ 文件 / 任务卡 / 助手记忆三层 |

一句话：**通用 AI 群聊是"会议室白板"，TeamAgentX 是"配齐工位、文件柜、流程卡的真办公室"**。

## 6. 标准开发团队配置（示例）

把"个人开发者"翻译成具体群配置：

| 助手 | 角色定位 | 推荐模型 | 装载 Skill | 触发方式 |
|------|---------|---------|-----------|---------|
| 🎯 **总控** | 需求拆解、任务卡分发、阻塞处理、收尾 | Opus 4.7 | `brainstorming`、`writing-plans` | @总控 / 群主默认 |
| 🛠 **工程师A**（主力实现） | 写代码、跑测试 | Sonnet 4.6 | `react` / `nodejs-backend` 等 | @工程师A 或总控分配 |
| 🛠 **工程师B**（辅助 / 并行） | 处理可并行的子任务 | Sonnet 4.6 | 同上但不同栈 | 同上 |
| 🔍 **质检** | 按"验收标准 + 不需要做"审查输出 | Opus 4.7（审查比写更需推理） | `code-review`、`verify` | 工程师交付后总控触发 |
| 📚 **资料员**（可选） | 查文档、查竞品 | Haiku 4.5（便宜快） | `WebSearch`、`WebFetch` | 总控按需 @ |
| 📝 **文档员**（可选） | 写 README、changelog | Haiku 4.5 | `writing-clearly` | 交付前 @ |

**最小可用编队 = 总控 + 工程师 + 质检（3 人）**；扩展按需加资料员、文档员、第二工程师并行。

每个助手的"模型 / 技能 / 提示词"三件套独立配置，群规则负责把他们粘合成"团队"——**这就是平台的底层产品形态**。

## 7. 核心架构

### 7.1 五大核心要素

| 层级 | 要素 | 解决的问题 |
|------|------|-----------|
| 🔧 流程 | **任务卡机制** | 做什么 |
| 👤 责任 | **角色边界** | 谁负责 |
| ✅ 质量 | **验收标准** | 怎么算完成 |
| 📦 空间 | **群即项目** | 在哪做 |
| 🛡 安全 | **阻塞上报** | 出问题怎么办 |

### 7.2 四层模型（产品的四个一等公民）

```
群（ChatRoom）              ← 上下文容器、任务卡载体
  └─ 助手（Agent）          ← 角色 + 配置
        ├─ 模型（LLM Provider）  ← 推理引擎
        └─ 技能（Skill）         ← 能力包
```

| 层 | 是什么 | 复用粒度 | 关键配置项 |
|----|-------|---------|-----------|
| **模型** | API key + provider + model id | 全平台共享 | `provider/model_id/api_key/api_url/api_protocol/is_default` |
| **技能（Skill）** | 提示片段 + 工具白名单 + 触发器（兼容 Claude Code skill 格式） | 跨助手复用 | `slug/version/description/prompt/tools/triggers` |
| **助手** | 模型 + 系统提示 + 一组 skill + 角色定位 | 跨群复用 | `name/role/model_ref/system_prompt/skills[]/agent_level` |
| **群** | 一组助手 + 群规则 + 任务卡 + 工作目录 | 一个群 = 一个项目 | `members[]/rules/work_dir/task_cards[]/cron_tasks[]` |

**关键设计：四层都是引用而非嵌套。** 改一个模型 key，所有引用它的助手立即生效；删一个 skill，所有装它的助手收到失效信号。

### 7.3 群即项目：上下文隔离的三层

| 隔离层 | 边界 | 实现 | 防止的事 |
|-------|------|------|---------|
| **文件层** | 每个群独占一个 `work_dir` | 创建群时分配独立目录，助手所有文件操作锁定该目录 | 助手改错项目的文件 |
| **任务卡层** | 任务卡只属于当前群 | 任务卡 id 带 group 前缀，跨群不可见 | A 项目的进度污染 B 项目 |
| **助手记忆层** | 助手在每个群有独立记忆切片 | 助手实例 = `assistant_template + group_id`，对话历史/记忆按群分库 | 跨项目串味、信息泄漏 |

**三层缺一不可**：缺文件层 → 改错文件；缺任务卡层 → 状态串；缺记忆层 → 上下文污染。

**助手模板 vs 助手实例**：模板是"工程师 A 的人设和能力"，可在多个群中实例化；实例才持有具体的工作记忆和任务历史。这是"群即项目"原则在 Agent 层的对应实现。

v0.1.0 已实现的相关能力：
- 群级 `workDir`（"群内所有助手共享这个运行目录"）
- 助手级默认 `workDir`（"助手默认目录"）
- 快速对话级独立 `workDir`（"留空时，每次对话将创建独立的会话目录"）
- 加助手到群时可选 `injectGroupHistory: true|false`（控制群内上下文注入）
- 清空助手在该群的上下文（`POST /chatrooms/:id/agents/:id/clear-context`）

### 7.4 通讯模型

```
[用户 / 助手] 发言
   ↓
[消息总线] 解析 @ 列表 + 群规则校验（he 律层 hooks）
   ↓
被 @ 的助手 → 入"待响应"队列
未被 @ 的助手 → 默默吃上下文不发言
   ↓
按队列顺序串行响应（不抢答）
   ↓
助手响应 = 系统提示 + 群规则（自律层）+ 群上下文窗口 + 自己的记忆切片 + 装载 skill
   ↓
响应里如含新的 @ → 入队下一轮
响应里如含 [讨论结束] 且发言者是总控 → 群闭环
   ↓
轮次 +1，到达上限 → 强制总控收尾
```

**三个底层不变量**：

1. **只有总控能宣布结束**（独占结束权 → 防循环）
2. **助手不抢答**（被 @ 才发言 → 防 @ 触发歧义）
3. **每轮先解析 @ 再发言**（排除歧义后再消耗 token → 防上下文爆炸）

详细工作流程见 [03-workflows.md](03-workflows.md)。

## 8. 三大产品护城河

1. **群聊隐喻**：复用人类的"群协作肌肉记忆"，零学习成本上手
2. **任务卡 + 群即项目**：把"乱聊"变成"有产出的协作"，文件、任务、记忆三层隔离防互相污染
3. **可观测的多 Agent 协作**：流式思考链、执行记录、调试信息、上下文检视——**看得见的协作**比"黑盒 Agent"更有信任感
